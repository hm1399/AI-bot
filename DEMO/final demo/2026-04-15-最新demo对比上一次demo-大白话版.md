# 这次最新 demo 比上一次 demo 多了什么

> 说明：这份对比是按 `2026-04-15` 当前仓库里的真实代码和配置来写的，尽量只写已经落地的东西，不把纯规划当成功能。

## 先说结论

上一次 demo 更像是在证明一件事：`这个硬件能录音、能识别、能让 AI 回答、还能顺手控一下电脑。`

这一次最新 demo 已经不是单纯的“语音链路打通”了，而是长出了一套比较完整的产品骨架：有桌面工作台、有机器人配对流程、有聊天会话管理、有任务/日历/提醒、有控制中心，也开始有人格、场景、物理交互这类更像产品的层。

简单讲就是：

- 上一次 demo：`像一个能说话的硬件样机`
- 这一次 demo：`像一个正在成型的桌面 AI 助手系统`

## 上一次 demo 的重点是什么

根据 `demo流程.md`，上一次 demo 的核心亮点主要是这几件事：

- 机器人开机，连上 WiFi 和后端。
- 你按住触摸板说话，松开后开始处理。
- AI 的回复会显示在屏幕上，也会发到 WhatsApp。
- 还能执行简单电脑控制，比如打开微信、打开 GitHub、回答天气。

所以那一版 demo 的重点，本质上是：`硬件语音主链路通了，而且能做几个直观动作。`

## 这次最新 demo 比上一次多出来的东西

### 1. 不再只有“硬件 + WhatsApp”，而是有完整桌面 App 工作台了

上一次 demo 的主要展示面还是设备屏幕和 WhatsApp。

现在 `app/` 已经是正式 Flutter 工程，而且不只是一个占位壳子，已经有这些页面：

- `Connect`
- `Home`
- `Chat`
- `Agenda`
- `Tasks`
- `Control Center`
- `Settings`
- `Demo Mode`

这意味着现在 demo 不只是“机器人回你一句”，而是“你有一个完整的桌面操作界面可以看状态、改设置、管任务、管聊天、管设备”。

### 2. 机器人首次配对流程已经产品化了

上一次 demo 更像默认开发者已经提前把设备、WiFi、后端地址都配好了。

现在 `Connect` 页已经做出了明确的三步式流程：

- 先连接 live backend
- 再通过 USB 连机器人
- 再下发 WiFi 和局域网地址的 pairing bundle

而且 `device pairing` 这条链路不是随便写写 UI，前后端都有对应实现。大白话说，就是以前更像“开发者手工准备好再演示”，现在已经在往“别人按流程也能配起来”走。

### 3. 聊天已经从“问一句答一句”变成“会话型产品”

上一次 demo 的聊天体验，本质上还是单轮问答演示。

现在 Chat 这边已经多了不少真正的产品能力：

- 多会话列表
- 新建会话
- 会话改名
- 置顶
- 归档
- 当前会话切换

也就是说，现在不再只是“这个机器人会聊天”，而是“这个系统已经开始管理多段聊天关系了”。

### 4. 聊天里已经有人格和场景模式，不只是一个固定 AI

上一次 demo 里的 AI 更像一个固定风格的助手。

现在 Chat 界面里已经有 `scene` 和 `persona` 这两层：

- Scene：`Focus / Off Work / Meeting`
- Persona：多种预设人格风格

而且从当前代码看，这些不只是 UI 标签，后端已经有 `ExperienceService` 和对应接口在管理这些状态。

大白话说：以前是“只有一个 AI”，现在开始变成“同一个 AI 可以切工作状态和说话风格”。

### 5. 新增了任务、日历、提醒这一整套 planning 能力

这块是这次和上一次差距非常大的地方。

上一次 demo 的重点还是语音交互和即时执行，几乎没有真正的“事情管理工作台”。

现在已经有：

- `Agenda` 页面
- `Tasks` 页面
- `Task / Event / Reminder` 的新增、编辑、删除
- `planning overview`
- `planning timeline`
- `planning conflicts`
- `planning bundle create`

换句话说，项目现在已经不只是“AI 能回答你”，而是开始变成“AI 说完之后，后续事情能不能被组织起来、被看见、被改动、被安排”的系统。

### 6. 提醒不只是存一条数据，而是真的开始有调度器了

旧 demo 里的逻辑更偏即时响应。

现在后端已经有 `ReminderScheduler`，会维护：

- 下一次触发时间
- snooze
- complete
- reschedule
- reconcile

大白话说，提醒已经不是“列表里放一条”，而是“后台真的开始管它什么时候该响、推迟后怎么算、完成后怎么收口”。

### 7. 电脑控制已经从“能开几个东西”升级成结构化控制中心

上一次 demo 里最直观的电脑控制，是像“打开微信”“打开 GitHub”这种口头指令。

现在的 `Control Center` 已经明显更产品化了，支持结构化 action，比如：

- `open_app`
- `open_path`
- `open_url`
- `run_shortcut`
- `run_script`
- `system_info`

同时还多了这些产品化能力：

- pending approvals
- recent actions
- permission hints
- confirm / cancel 流程

大白话讲就是：

- 上一次：`AI 能控电脑`
- 这一次：`AI 控电脑已经开始有面板、有权限感、有回看记录、有审批感`

### 8. 设备控制和运行状态可视化，比以前完整很多

以前很多状态主要靠设备屏幕上那几行字来体现。

现在 `Home` 和 `Control Center` 已经能看到很多运行态信息，比如：

- 设备在线/离线
- voice pipeline 状态
- 当前 runtime task
- task queue
- notifications / reminders 数量
- todo summary / calendar summary
- 设备音量
- LED 亮度
- LED 颜色
- speak / stop / sync / toggle LED

这说明 demo 已经从“做出一个效果”往“让操作者清楚知道现在系统在干嘛”进了一大步。

### 9. 语音链路已经扩展到桌面麦克风桥接，不再只盯着板子本身

上一次 demo 的语音入口主要就是设备自己录音。

现在后端多了 `DesktopVoiceService`，运行态里也已经有 desktop voice / bridge 的状态模型，Chat 和 Home 里也能看到相应状态。

这代表语音链路已经开始往“设备 + 桌面端协同”的方向走，而不是只看板子本身能不能收音。

### 10. 物理交互已经不只是“按住说话”，而是开始做成一套规则

上一次 demo 里，物理交互的重点基本还是按住录音。

现在后端和前端里已经出现一整层 `experience / physical interaction` 逻辑，里面包括：

- shake routing
- tap confirmation
- hold-to-talk 状态
- 交互历史
- debug trigger
- 当前 scene / persona 对物理交互的影响

大白话说：以前是“设备有个触摸板可以说话”，现在是“设备的物理动作开始被当成产品交互逻辑来设计”。

### 11. 底层系统也比上一次成熟很多

这部分用户不一定第一眼能看出来，但很能体现项目是不是在进步。

现在后端已经有比较完整的 `app-v1` 接口和实时事件流，包括：

- HTTP 接口
- WebSocket 事件流
- bootstrap
- resume / replay
- runtime state
- capabilities

`server/config.yaml` 里也已经能看到更系统化的配置，比如：

- transport queue
- outbound lanes
- dual storage
- structured computer control allowlist

这些都说明项目已经不是“把 demo 跑起来就算完”，而是在往更稳定、更像长期产品的结构上走。

## 最能体现“这次真的有进展”的几个点

如果你是想拿给别人看“这段时间不是小修小补，是真的往前走了”，我觉得最值得讲的是这几条：

1. 从“单个硬件 demo”进化成“软硬件一体的桌面 AI 助手系统”。
2. 从“能说话能开程序”进化成“能管理聊天、任务、提醒、设备和电脑动作”的工作台。
3. 从“技术验证演示”进化成“开始有产品形状的 demo”。

## 但也别夸太满，当前还有几件事没完全闭环

为了口径更稳，这几个点建议不要说成“已经彻底做完”：

- Flutter 侧 `voice_capture_service.dart` 现在还是占位，前端原生录音链路还没真正接好。
- `wake word / auto listen` 目前有配置字段，但 runtime 里仍标的是未真正实现。
- 外部日历同步、手机推送这类更高层的产品能力，还没看到完整闭环。
- 根 README 的项目说明还是旧口径，对这次最新 demo 的进展反映得不够完整。

## 最后一句总结

如果上一次 demo 是在证明“这个机器人活了”，那这一次 demo 更像是在证明“这个项目已经开始长成一个真正可操作的 AI 助手产品”。
