# AI-Bot 桌面AI助手 — 系统架构设计

> 创建日期: 2026-02-25
> 状态: 设计阶段（未实现）

---

## 一、架构总览

三端系统：**硬件设备（ESP32-S3）**、**电脑服务端（Python）**、**手机App（Flutter）**。

服务端是唯一的"大脑"，硬件和手机都是"终端"。所有 AI 调用、语音处理、工具执行均在服务端完成。

```
┌──────────────┐    WiFi局域网     ┌───────────────────────────┐    WiFi局域网     ┌──────────────┐
│  ESP32-S3    │◄──WebSocket────►│   电脑服务端 (Python)      │◄──WebSocket────►│  手机App     │
│  硬件设备     │  二进制音频流      │   FastAPI 单进程           │  JSON文本        │  Flutter     │
│              │  JSON控制指令     │                           │  音频数据(后期)   │  iOS/Android │
└──────────────┘                  └───────────────────────────┘                  └──────────────┘
                                    │ 内部调用
                                    ├── Whisper (ASR 语音识别)
                                    ├── Edge-TTS (语音合成)
                                    ├── Claude API (AI对话 + Tool Use)
                                    ├── Nanobot (电脑控制)
                                    └── SQLite (本地数据存储)
```

---

## 二、关键架构决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 后端架构 | 单体 FastAPI，一个 Python 进程 | 部署简单，一条命令启动，个人项目够用 |
| App 平台 | Flutter (iOS/Android) | 跨平台，一套代码两端运行 |
| App 功能定位 | 远程控制 + 配置管理，同等重要 | 不在电脑旁时也能用 |
| 部署方式 | 纯本地部署 | 服务端运行在用户电脑上，手机通过局域网连接 |
| 会话模型 | 硬件与 App 共享对话上下文 | 设备间无缝切换对话 |
| AI 调用 | 统一走服务端 Claude API | 用户只需配一次 API Key |
| 电脑控制 | Nanobot（~4000行轻量 Agent） | 比 OpenClaw 轻量99%，pip install 即用 |
| 数据存储 | 本地 SQLite | 任务、日程日历(events合并表)、对话历史 |
| LLM 工具系统 | 提示词文件(.md) + Claude tool_use | 3个工具（任务队列、日程日历、电脑控制） |
| App 输入方式 | 文字为主，语音为辅（后期添加） | MVP 先做文字 |

## 三、服务端架构

### 3.1 后端风格：单体 FastAPI + asyncio 异步驱动

```
┌─────────────────────────────────────────────────────────┐
│              FastAPI 单进程 (uvicorn)                     │
│                                                         │
│  ESP32 ──WS──► /ws/device ──┐                           │
│                              │    ┌────────────────┐    │
│                              ├──► │ SessionManager  │    │
│                              │    │  (共享会话)      │    │
│  App ────WS──► /ws/app   ──┘    └───────┬────────┘    │
│                                          │              │
│  App ──HTTP──► /api/*    ────────────────┤              │
│                                          ▼              │
│                                 ┌──────────────┐        │
│                                 │ IntentRouter  │        │
│                                 └──┬──┬──┬──┬──┘        │
│                                    │  │  │  │           │
│                            ┌───────┘  │  │  └───────┐   │
│                            ▼          ▼  ▼          ▼   │
│                        ┌──────┐ ┌────┐ ┌────┐ ┌───────┐│
│                        │ ASR  │ │LLM │ │TTS │ │Nanobot││
│                        │线程池 │ │async│ │async│ │subpro ││
│                        └──────┘ └────┘ └────┘ └───────┘│
└─────────────────────────────────────────────────────────┘
```

**异步策略：**
- Whisper 推理是 CPU 密集型 → 放到 `线程池/进程池` 异步执行，不阻塞主循环
- Claude API、Edge-TTS 是 IO 密集型 → 用 `async/await` 原生异步
- Nanobot 电脑控制 → `subprocess` 调用

### 3.2 目录结构

```

│   ├── system.md               #   主系统提示词（AI人格、行为规范）
│   └── tools/                  #   每个工具的使用说明

│       ├── events.md           #     日程+日历工具提示词
```

**三层分离：**
- `api/` — 只管连接和消息收发，不含业务逻辑
- `core/` — 管业务编排（谁该做什么），不依赖具体服务实现
- `services/` — 管具体能力（ASR、TTS、LLM），可独立替换

### 3.3 两条 WebSocket 端点

| 端点 | 连接方 | 传输内容 | 特点 |
|------|--------|----------|------|
| `/ws/device` | ESP32-S3 硬件 | 二进制音频帧 + JSON 控制指令 | 高频、低延迟、混合二进制/文本帧 |
| `/ws/app` | Flutter 手机App | 纯 JSON 消息 | 文本为主，后期可加音频 |

两条 WS 共享同一个 `SessionManager`，硬件上说的话 App 端能看到，App 发的指令硬件端能收到反馈。

### 3.4 REST API

```
GET  /api/health                # 服务端在线检测（App发现服务用）
GET  /api/config                # 读取当前配置
PUT  /api/config                # 修改配置（API Key、音量、TTS语音角色等）
GET  /api/device/status         # 硬件设备状态（在线/电量/WiFi信号）
GET  /api/history               # 对话历史
POST /api/chat                  # App端文字发送消息（HTTP替代方案）
```

## 四、会话模型（SessionManager）

一个用户只有一个 Session，硬件和 App 同时挂在上面：

```
Session
├── conversation: list[Message]       # 对话历史（硬件和App共享）
├── device_ws: WebSocket | None       # 硬件连接（可能离线）
├── app_ws: WebSocket | None          # App连接（可能离线）
├── device_status: DeviceStatus       # 电量、WiFi信号、在线状态
└── config: UserConfig                # 用户配置（引用 config.yaml）
```

**状态同步规则：**
- 任一端发消息 → 写入共享 `conversation` → 另一端实时推送
- 硬件离线时 App 仍可对话，硬件上线后自动同步
- App 离线时硬件正常工作，App 上线后拉取历史

## 五、LLM + Tool Use 架构

### 5.1 核心思路

每个功能是一个独立的 **Tool**，有两个组成部分：
1. **提示词文件** (`prompts/tools/xxx.md`) — 告诉 AI 这个工具是什么、什么时候用、怎么调用
2. **执行代码** (`services/tools/xxx.py`) — 实际执行工具操作（读写数据库、调用外部服务）

### 5.2 工具清单

| 工具名 | 提示词文件 | 执行代码 | 数据存储 | 功能 |
|--------|-----------|----------|----------|------|
| task_queue | task_queue.md | task_queue.py | SQLite tasks 表 | 创建/查看/完成/删除任务 |
| events | events.md | events.py | SQLite events 表 | 日程提醒、日历事件、闹钟（统一管理） |
| computer | computer.md | computer.py | 无（Nanobot执行） | 电脑控制（开应用、文件操作等） |

### 5.3 提示词文件格式

每个 `prompts/tools/*.md` 遵循统一结构：

```markdown
# <tool_name> (<中文名>)

## 描述
一句话说明这个工具做什么。

## 何时使用
- 触发条件1
- 触发条件2

## 可用操作
- action_1: 说明 (参数)
- action_2: 说明 (参数)

## 调用示例
用户说"..."
→ { tool: "...", action: "...", params: { ... } }
```

### 5.4 运行流程

```
用户: "帮我明天下午3点提醒开会"
                │
                ▼
┌───────────────────────────────────────────────┐
│ llm.py 组装请求                                │
│                                               │
│ system_prompt =  prompts/system.md            │
│                + prompts/tools/task_queue.md   │
│                + prompts/tools/events.md       │
│                + prompts/tools/computer.md     │
│                                               │
│ → 发送给 Claude API (messages + tools 参数)    │
└───────────────────────┬───────────────────────┘
                        │
                        ▼  Claude 返回 tool_use
               { tool: "schedule",
                 action: "create",
                 params: { title: "开会",
                           time: "2026-02-26T15:00",
                           remind: true } }
                        │
                        ▼
               schedule.py 执行
               → 写入 SQLite schedules 表
               → 返回执行结果给 Claude
                        │
                        ▼
               Claude 生成最终回复
               → "好的，已经帮你设置了明天下午3点的开会提醒"
                        │
                        ▼
               TTS 合成语音 → 发给硬件播放
               文字回复 → 推送给 App 显示
```

### 5.5 提示词热加载

`llm.py` 支持热加载提示词文件：
- 启动时读取 `prompts/` 目录下所有 `.md` 文件
- 每次 LLM 调用前检查文件修改时间，如有变化则重新加载
- 修改提示词不需要重启服务端

### 5.6 添加新工具流程

只需两步：

1. 创建 `prompts/tools/new_tool.md` — 写清描述、触发条件、可用操作、调用示例
2. 创建 `services/tools/new_tool.py` — 继承 Tool 基类，实现 `execute()` 方法

不需要修改 `llm.py` 或其他代码，系统自动发现并注册新工具。

## 七、配置系统

### 7.1 配置文件 (config.yaml)

```yaml
# LLM 配置
llm:
  provider: claude                    # claude / openai
  api_key: sk-ant-xxx                 # API Key
  model: claude-sonnet-4-20250514               # 模型选择
  max_context_messages: 20            # 保留最近N轮对话

# 语音识别配置
asr:
  model: base                         # whisper模型: tiny / base / small
  language: zh                        # 识别语言

# 语音合成配置
tts:
  voice: zh-CN-XiaoxiaoNeural         # TTS语音角色
  speed: 1.0                          # 语速

# 设备配置
device:
  volume: 80                          # 默认音量 (0-100)
  led_mode: auto                      # LED灯效: auto / off

# Nanobot 配置
nanobot:
  enabled: true                       # 是否启用电脑控制

# 服务端配置
server:
  host: 0.0.0.0
  port: 8080
```

### 7.2 App 通过 REST API 读写配置

- `GET /api/config` → 返回当前配置（API Key 脱敏显示）
- `PUT /api/config` → 修改配置项，服务端热加载生效

### 8.3 页面结构

```
┌─────────────────────────────────────┐
│  App 底部导航栏 (BottomNavigationBar) │
```

**首页 (HomeScreen)**
- 设备状态卡片：在线/离线、电量、WiFi信号强度
- 设备当前状态：待机/录音中/播放中
- 快捷操作按钮：静音、LED开关、重启设备

**对话 (ChatScreen)**
- 聊天气泡界面（类似微信）
- 区分消息来源：设备端(麦克风) / App端(文字)
- 底部输入栏：文字输入 + 发送按钮 + 语音按钮(后期)

- 顶部 Tab 切换：任务列表 / 日历视图
- 任务列表：按优先级分组，支持滑动完成/删除
- 日历视图：月历 + 日视图，标记有事件的日期
- 浮动按钮：手动创建任务/事件（也可通过对话创建）

- 服务端连接：IP地址、端口、连接状态
- LLM配置：API Key（脱敏显示）、模型选择
- 语音配置：TTS语音角色、语速
- 设备配置：音量、LED灯效模式
- 关于：版本信息

### 8.4 App 与服务端通信

```
Flutter App
├── WebSocket 长连接 (/ws/app)
│   ├── 发送: 文字消息、指令
│   ├── 接收: AI回复、状态推送、硬件事件
│   └── 心跳保活 (30秒 ping/pong)
│
└── REST API (HTTP)
    ├── 服务发现: GET /api/health (局域网扫描)
    ├── 配置读写: GET/PUT /api/config
    ├── 设备状态: GET /api/device/status
    └── 对话历史: GET /api/history
```

**通信策略：**
- 实时数据（对话、状态推送）→ WebSocket
- 一次性查询和修改（配置、历史记录）→ REST API
- App 进入后台时断开 WS，回到前台时自动重连

### 8.5 局域网服务发现

App 首次使用时需要找到服务端地址：

1. App 在局域网内广播扫描 `GET /api/health`
2. 服务端响应 → App 记住 IP 和端口
3. 后续直连，断线后重新扫描

备用方案：用户手动输入 IP:端口

### 8.6 状态管理数据流

```
服务端 ──WebSocket──► ws_service.dart
                          │
                          │ 消息分发
                    ┌─────┼──────┬──────────┐
                    ▼     ▼      ▼          ▼
              chat_     device_  task_     event_
              provider  provider provider  provider
                 │        │        │         │
                 ▼        ▼        ▼         ▼
              ChatScreen  Home   Tasks    Events
                         Screen  Screen   Screen
```

`ws_service.dart` 是唯一的 WebSocket 入口，收到消息后按 `type` 字段分发给对应的 Provider，Provider 更新状态后 UI 自动刷新。

## 十、数据库设计 (SQLite)

### 10.1 设计原则

- **SQLite** — 单用户本地项目的最优选择：零配置、Python 内置、一个文件搞定备份、10万条以内毫秒级响应
- 开启 **WAL 模式**（Write-Ahead Logging）— 读写不互相阻塞，WebSocket 写入和 App 读取可并行
- 日程和日历合并为 `events` 表，用 `type` 字段区分，避免 LLM 选错表
- 任务可通过 `event_id` 关联日历提醒（如"这个任务下周五截止"自动创建提醒）
- 对话历史通过 **LLM 摘要压缩** 保留长期记忆，不会简单删除
- 常用查询字段加索引（按时间查询）

### 10.2 建表语句

```sql
-- 启用 WAL 模式（连接数据库后执行一次）
PRAGMA journal_mode=WAL;

-- 任务队列
CREATE TABLE tasks (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    title        TEXT NOT NULL,
    description  TEXT,
    status       TEXT DEFAULT 'pending',     -- pending / in_progress / done
    priority     TEXT DEFAULT 'normal',      -- low / normal / high / urgent
    event_id     INTEGER,                    -- 关联的日历提醒（截止日期）
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    due_at       DATETIME,
    completed_at DATETIME,
    FOREIGN KEY (event_id) REFERENCES events(id) ON DELETE SET NULL
);

-- 日程+日历（合并表）
CREATE TABLE events (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    type          TEXT NOT NULL,              -- 'reminder' 提醒(单时间点)
                                             -- 'event'    事件(有时间段)
    title         TEXT NOT NULL,
    start_time    DATETIME NOT NULL,
    end_time      DATETIME,                  -- reminder 可为 NULL
    all_day       BOOLEAN DEFAULT 0,
    location      TEXT,
    notes         TEXT,
    remind        BOOLEAN DEFAULT 1,
    remind_before INTEGER DEFAULT 10,        -- 提前N分钟提醒
    repeat        TEXT,                      -- none / daily / weekly / monthly
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 对话历史
CREATE TABLE conversations (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    role        TEXT NOT NULL,                -- user / assistant / tool
    content     TEXT NOT NULL,
    source      TEXT DEFAULT 'device',       -- device / app
    tool_name   TEXT,                        -- tool调用时记录工具名
    compressed  BOOLEAN DEFAULT 0,           -- 是否已被压缩为摘要
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 对话摘要（压缩后的长期记忆）
CREATE TABLE summaries (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    summary         TEXT NOT NULL,            -- LLM生成的对话摘要
    source_start_id INTEGER NOT NULL,         -- 摘要覆盖的起始对话ID
    source_end_id   INTEGER NOT NULL,         -- 摘要覆盖的结束对话ID
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_tasks_status       ON tasks(status);
CREATE INDEX idx_tasks_due_at       ON tasks(due_at);
CREATE INDEX idx_events_start       ON events(start_time);
CREATE INDEX idx_events_type        ON events(type);
CREATE INDEX idx_conv_compressed    ON conversations(compressed);
CREATE INDEX idx_conv_created       ON conversations(created_at);
CREATE INDEX idx_summaries_range    ON summaries(source_start_id, source_end_id);
```

### 10.3 对话历史压缩策略

用 LLM 摘要替代简单删除，保留长期记忆：

```
触发条件: 未压缩对话(compressed=0)超过 50 轮

压缩过程:
  1. 查询最早的 50 轮未压缩对话
  2. 发给 Claude: "请用200字以内总结以下对话的关键信息，
     包括用户的偏好、做出的决定、重要事实"
  3. 摘要存入 summaries 表（记录 source_start_id ~ source_end_id）
  4. 原始对话标记 compressed = 1

每次 LLM 调用时组装上下文:
  system_prompt（人格 + 工具说明）
  + 所有历史摘要（按时间顺序，提供长期记忆）
  + 最近 20 轮完整对话（提供精确上下文）
```

**效果**: AI 既能记住"用户上周说过喜欢用中文回复"这样的长期偏好，又不会因为 token 过多导致成本失控。
