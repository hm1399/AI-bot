## 一、当前架构定位

当前后端已经不是“纯设计阶段”，而是一个**可运行的 Demo 级后端**：

- 可接入 ESP32 设备，通过 WebSocket 收发文本、音频和状态消息
- 可调用本地移植的 nanobot AgentLoop 进行 AI 对话与工具调用
- 可使用 SenseVoice 做 ASR，使用 Edge-TTS 做语音合成
- 可通过 Node.js WhatsApp Bridge 收发 WhatsApp 消息
- 可把设备回复同步到设备屏幕、语音播报，以及 WhatsApp

但它目前仍偏向：

- **单机运行**
- **单设备 / 单人 Demo**
- **课程项目与功能验证**

还没有完全演进到：

- 面向 Flutter App 的标准 API 后端
- 强认证、强审计的生产级服务
- 多客户端并发友好的平台型后端

## 二、整体拓扑

当前系统是一个“三段式”本地架构：

```text
ESP32-S3 设备
  └─ WiFi / WebSocket
        ↓
Python 服务端 (aiohttp, port 8765)
  ├─ DeviceChannel
  ├─ ASR / TTS
  ├─ MessageBus
  ├─ AgentLoop
  ├─ Session / Memory
  └─ HTTP API
        ↓
Node.js WhatsApp Bridge (localhost:3001)
  └─ WhatsApp Web / Baileys
```

### 当前各节点职责

#### 1. ESP32 设备

- 采集麦克风 PCM 音频并通过 WebSocket 上传
- 接收服务端返回的 JSON 指令和 PCM 音频流
- 负责屏幕显示、状态反馈、触摸/摇一摇等本地交互

#### 2. Python 服务端

- 作为系统核心编排层
- 负责设备接入、AI 推理编排、ASR/TTS、会话持久化
- 负责设备消息与 WhatsApp 消息之间的统一路由

#### 3. Node.js WhatsApp Bridge

- 专门负责接入 WhatsApp Web
- 与 Python 后端通过本地 WebSocket 通信
- 隔离 WhatsApp 协议复杂度，避免直接在 Python 中处理 Baileys

## 三、Python 服务端内部结构

当前 Python 后端可分为 5 层理解。

### 3.1 启动与装配层

核心文件：

- `server/main.py`
- `server/config.py`
- `server/config.yaml`

职责：

- 初始化日志
- 读取 `config.yaml` 和 `.env`
- 生成 nanobot 运行所需的 `workspace/config.json`
- 初始化 `MessageBus`、`LiteLLMProvider`、`SessionManager`、`AgentLoop`
- 初始化 `ASRService`、`TTSService`、`DeviceChannel`
- 按配置决定是否启用 `WhatsAppChannel`
- 挂载 HTTP 路由与 WebSocket 路由
- 统一管理优雅关闭

### 3.2 接入层

当前已有两类接入：

#### 设备接入

- 路径：`/ws/device`
- 实现：`server/channels/device_channel.py`
- 协议：JSON 控制消息 + 二进制 PCM 音频帧

#### HTTP 接口

当前只有两个轻量接口：

- `/api/health`：健康检查
- `/api/device`：设备状态查询

说明：

- 目前还没有面向 Flutter App 的正式聊天 API
- HTTP 层现在主要服务于本地监控与调试

### 3.3 AI 编排层

核心文件：

- `server/nanobot/agent/loop.py`
- `server/nanobot/agent/context.py`
- `server/nanobot/providers/litellm_provider.py`
- `server/nanobot/bus/queue.py`
- `server/nanobot/bus/events.py`

职责：

- 从 inbound queue 读取消息
- 构建上下文、加载会话历史、注入记忆和技能
- 调用 LLM provider
- 执行工具调用
- 将最终回复写回 outbound queue

### 3.4 语音与设备交互层

核心文件：

- `server/services/asr.py`
- `server/services/tts.py`
- `server/models/protocol.py`
- `server/models/device_state.py`

职责：

- 处理设备音频格式约定
- 将 PCM 音频转成 ASR 可识别格式
- 将文本转成设备可播放的 PCM 音频流
- 维护设备状态机与状态同步协议

### 3.5 持久化与工作空间层

核心目录：

- `server/workspace/config.json`
- `server/workspace/SOUL.md`
- `server/workspace/memory/`
- `server/workspace/sessions/`

职责：

- 保存运行期配置
- 保存人格设定和技能
- 保存长期记忆与历史摘要
- 以 JSONL 形式保存多轮会话

## 四、当前核心运行链路

## 4.1 设备语音链路

这是当前最核心、最完整的一条链路。

```text
ESP32 开始录音
  ↓
向 /ws/device 发送二进制 PCM 帧
  ↓
DeviceChannel 累积 audio_buffer
  ↓
设备发送 audio_end
  ↓
DeviceChannel 调用 ASRService.transcribe()
  ↓
识别文本发布到 MessageBus.inbound
  ↓
AgentLoop 消费消息并调用 LLM / tools
  ↓
生成 OutboundMessage
  ↓
统一 outbound consumer 路由回复
  ├─ 发文字到设备屏幕
  ├─ 若原消息来源是 voice，则触发 TTS
  └─ 可同时转发到 WhatsApp
  ↓
设备收到 audio_play / PCM 二进制流 / audio_play_end
  ↓
播放 AI 回复
```

### 这条链路的关键特征

- 设备与服务端之间使用**原始 PCM 16kHz 16bit 单声道**
- ASR 使用 **SenseVoiceSmall / FunASR**
- TTS 使用 **Edge-TTS + miniaudio 解码**
- 设备状态会在 `IDLE / LISTENING / PROCESSING / SPEAKING / ERROR` 间切换

## 4.2 设备文本与事件链路

设备除语音外，还支持几类事件：

- `text_input`：调试用文字输入
- `touch_event`：单击 / 双击 / 长按 / 长按松开
- `shake_event`：摇一摇触发随机互动
- `device_status`：电量、WiFi、充电状态上报

这些事件由 `DeviceChannel` 直接转成内部消息或状态变化。

## 4.3 WhatsApp 链路

当前 WhatsApp 是“可用但偏 Demo 化”的第二消息通道。

```text
WhatsApp 消息
  ↓
Baileys / Node Bridge
  ↓
Python WhatsAppChannel
  ↓
MessageBus.inbound
  ↓
AgentLoop
  ↓
MessageBus.outbound
  ↓
统一 outbound consumer
  ↓
WhatsAppChannel.send()
```

### 当前行为特点

- 主要服务于 self-chat / 本人调试
- 当前支持文本消息、带 caption 的媒体文本化消息
- WhatsApp 语音消息目前只做占位提示，尚未真正接入 ASR
- 设备回复会被同步转发到最近一个 WhatsApp 会话，便于 Demo 展示

## 五、当前消息总线与路由方式

## 5.1 MessageBus

当前系统采用最简单直接的双队列模型：

- `inbound`：来自设备、WhatsApp、系统任务的输入消息
- `outbound`：AI 处理后的输出消息

优点：

- 解耦 channel 与 AgentLoop
- 简单易懂，便于快速落地

局限：

- 当前路由控制仍较集中
- 扩展到 App、多设备、多通道时需要更清晰的路由层

## 5.2 统一 outbound consumer

当前一个很重要的实现点是：

- Python 服务端没有让设备通道、WhatsApp 通道分别竞争 outbound 队列
- 而是在 `main.py` 中启动了一个**统一 outbound consumer**
- 由它根据 `OutboundMessage.channel` 决定是回设备还是回 WhatsApp

这样做解决了一个实际问题：

- 避免多个通道消费者同时抢占同一个 outbound 队列

但也带来一个现实问题：

- 路由逻辑当前还放在入口层，结构上偏临时、偏 Demo

## 六、当前 AI 核心结构

## 6.1 AgentLoop

当前 AI 核心仍是 nanobot 的精简移植版：

- 会从 `MessageBus` 读取 `InboundMessage`
- 根据 `session_key` 获取或创建会话
- 构建 system prompt、历史消息、runtime context
- 调用 `LiteLLMProvider`
- 如模型返回 tool calls，则依次执行工具
- 生成最终回复后写入 `OutboundMessage`

### 当前能力

- 支持多轮会话
- 支持 tool-use
- 支持 `/stop`、`/new`、`/help` 等指令
- 支持会话压缩与长期记忆归档
- 支持 progress 消息回传

### 当前限制

- 核心处理有全局锁，当前更偏单机、串行、安全保守
- 更适合 Demo 和单用户连续交互，不适合大量并发

## 6.2 LiteLLM Provider

当前 LLM 调用层的特点：

- 统一通过 LiteLLM 对接不同 provider
- 当前配置默认使用 `openrouter + z-ai/glm-4.7`
- 保留了模型解析、参数适配、cache control 等兼容逻辑

因此当前后端的模型层是**可替换的**，并不绑定单一厂商。

## 6.3 工具系统

当前 AgentLoop 默认注册的工具包括：

- 文件读写与编辑
- 目录浏览
- shell / exec
- web_search
- web_fetch
- message
- spawn
- cron（在服务启用时）

说明：

- 代码结构保留了 nanobot 的工具体系
- 当前主用途仍是电脑控制与基础问答
- 其中部分工具虽在底层存在，但未全部围绕本项目场景强化

## 七、当前设备协议与状态模型

## 7.1 设备上行消息

当前设备向服务端发送的核心消息类型：

- `audio_end`
- `touch_event`
- `shake_event`
- `device_status`
- `text_input`

此外，录音中的音频数据通过**二进制帧**直接上传。

## 7.2 服务端下行消息

当前服务端向设备发送的核心消息类型：

- `state_change`
- `display_update`
- `status_bar_update`
- `face_update`
- `audio_play`
- `audio_play_end`
- `text_reply`

说明：

- 协议中预留了 `led_control`
- 但当前主链路仍以屏幕显示、状态栏、表情状态、音频播放为主

## 7.3 状态机

当前设备状态机包括：

- `IDLE`
- `LISTENING`
- `PROCESSING`
- `SPEAKING`
- `ERROR`

状态机的职责：

- 保证录音、识别、播报各阶段切换有约束
- 在设备屏幕和表情状态上同步当前运行状态
- 在异常时回退到可恢复状态

## 八、ASR / TTS 当前实现

## 8.1 ASR

当前实现已从早期规划的 faster-whisper 调整为：

- **FunASR / SenseVoiceSmall**

原因与效果：

- 更贴合当前中英混合语音场景
- 支持情感标签解析
- 采用懒加载 + `asyncio.to_thread()`，减少阻塞主事件循环的风险

## 8.2 TTS

当前 TTS 方案为：

- **Edge-TTS 生成 MP3**
- **miniaudio 解码为 16k PCM**
- 再按 chunk 流式发送到设备

说明：

- 当前“流式”是“先完整合成，再分块发送”的 MVP 方案
- 真正的边生成边播放仍未实现

## 九、会话、记忆与工作空间

## 9.1 会话存储

当前会话以 JSONL 持久化，按 session 保存到：

- `server/workspace/sessions/`

特点：

- 读写简单
- 便于本地查看与调试
- 适合单机 Demo 阶段

不足：

- 不擅长复杂检索
- 不适合后期多端、多用户、高并发场景

## 9.2 记忆体系

当前工作空间内包括：

- `SOUL.md`：人格/系统设定
- `memory/MEMORY.md`：长期记忆摘要
- `memory/HISTORY.md`：可搜索的历史记录
- `skills/`：项目自定义技能

Agent 在构造 system prompt 时会读取这些内容作为上下文的一部分。

## 十、Node.js WhatsApp Bridge 架构

Node 侧的设计已经较清晰，职责也比较专一。

### 10.1 Bridge Server

职责：

- 本地启动 WebSocket Server
- 默认只绑定 `127.0.0.1`
- 可选 `BRIDGE_TOKEN` 认证
- 接收 Python 侧发来的发送指令
- 将 WhatsApp 事件广播回 Python 侧

### 10.2 WhatsApp Client

职责：

- 使用 Baileys 与 WhatsApp Web 建立连接
- 处理二维码登录
- 处理重连
- 读取消息并提取文本内容
- 发送回复消息
- 用 `sentMessageIds` 避免 self-chat 循环

### 10.3 当前局限

- 语音消息只识别为占位文本
- 仍以单 Python 客户端接入为主
- 目前没有更复杂的联系人、会话、媒体管理能力

## 十三、当前架构结论

当前后端可以概括为：

> 一个以 `aiohttp + DeviceChannel + AgentLoop + SenseVoice + Edge-TTS + WhatsApp Bridge` 为核心的本地单机 AI 编排后端。

它的优势是：

- 主链路已经落地
- 模块边界基本成形
- 具备继续扩展的基础

它的现实定位是：

- **能演示**
- **能联调**
- **能继续迭代**

但还不是：

- 面向多端产品的完整后端平台

后续如需继续演进，重点应放在：

- 路由层解耦
- App API 补齐
- 安全与权限
- 自动化测试
- 部署与运维能力
