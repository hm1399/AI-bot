## 结论先行

当前项目已经完成了一个可联调的主干版本：

- 前端已经是一个真实的 Flutter 工程，不再是早期 React 原型
- 后端已经有完整的 `app-v1` HTTP + WebSocket 接口、运行态、会话、设置、任务、事件、通知、提醒、设备控制入口
- 设备侧已经有触摸录音、ASR、屏幕显示、时间/天气状态栏等基础能力

但如果以 `功能实现讨论.md` 里的完整产品目标来衡量，当前状态仍然是：

- `主干链路已打通`
- `中层业务能力部分完成`
- `高层产品功能仍有明显缺口`

由于 `功能实现讨论.md` 已经是旧文件，而项目当前路线已经迁移到 Nanobot，所以这份调研**不再用旧技术方案去否定当前实现**，只看一件事：

- `目标功能有没有对齐`
- `如果没对齐，差在哪一层`

按这个口径看，当前真正的缺口主要集中在：

1. 语音闭环还不完整
2. 提醒 / 待办 / 日历还缺真正的智能编排和调度
3. 人格系统、场景模式、推送通知等高层产品功能还没有完成
4. 前端虽然已有页面和协议层，但不少功能还是“能看、能读、还不能完整操作”

## 功能对齐总判断

如果只按“当前 Nanobot 版本与目标功能的对齐度”来判断，可以概括成下面三层：

- `底层能力对齐度较高`
  - 前后端通信
  - 聊天主链路
  - 设备连接
  - 基础 ASR/TTS
  - 运行态 / 任务 / 事件 / 设置接口
- `中层业务对齐度中等`
  - Tasks / Events / Notifications / Reminders 已有数据结构和接口
  - Home / Control Center / Settings / Chat 已有页面和基础交互
  - 设备显示、时间、天气、触摸录音已经能工作
- `高层产品功能对齐度偏低`
  - AI 人格系统
  - 智能场景模式
  - 智能提醒调度
  - 日历同步
  - 手机推送
  - 完整电脑控制产品化能力

所以更准确的结论不是“架构有没有照旧文档实现”，而是：

- `基础能力已对齐较多`
- `产品功能还没有完全对齐`
- `当前处于中期，不是尾声`

## 前端工作进度总结

### 已完成的前端部分

- 已经是标准 Flutter 工程：
  - `app/android`
  - `app/ios`
  - `app/web`
  - `app/lib`
  - `app/test`
  - `app/pubspec.yaml`
- 页面骨架已经齐全：
  - `Connect`
  - `Home`
  - `Chat`
  - `Tasks`
  - `Control Center`
  - `Settings`
  - `Demo Mode`
- 已接入后端主协议：
  - `GET /api/health`
  - `GET /api/app/v1/bootstrap`
  - `GET /api/app/v1/runtime/state`
  - `GET/POST sessions/messages`
  - `GET/PUT settings`
  - `GET tasks/events/notifications/reminders`
  - `/ws/app/v1/events`
- 已有连接信息本地保存、事件流重连、`last_event_id` 恢复
- 已有聊天 pending/progress/completed 事件流模型
- 已有 Demo Mode，本地假数据与真实后端模式分离

### 前端仍未完成或只完成一半的部分

- `VoiceCaptureService` 目前直接返回 `false/null`，手机端语音输入没有真正实现
- Chat 页虽然能发消息，但当前 UI 没有真正把“多会话管理”做完整，已看到 provider 中有 `selectSession`，页面层未完成切换/创建入口
- Tasks / Events / Notifications / Reminders 当前主要是读取展示
  - service 层已经定义了部分 CRUD 方法
  - 但页面层并没有把这些新增/编辑/删除能力真正做出来
- Device command API 已有常量和 service，但前端没有完成音量、灯光、wake/sleep 等完整控制面板
- 没有手机推送通知能力
- 没有局域网自动发现，连接页明确写着后续再做 mDNS/zeroconf
- 自动化验证很弱：当前只有 `app/test/widget_test.dart` 一个基础启动测试

### 当前前端验证状态

- `flutter test` 未能在当前环境跑通
- 原因不是代码语法本身直接报错，而是本机 Flutter/Dart 版本不满足工程要求：
  - 当前环境：`Flutter 3.38.7 / Dart 3.10.7`
  - `pubspec.yaml` 要求：`sdk: ^3.11.1`
- 因为 SDK 版本不匹配，依赖解析失败，前端分析和测试都无法完成

### 对前端进度的简单判断

- `工程框架`: 已完成
- `协议对接主链路`: 已完成度较高
- `产品功能完整度`: 中等偏低
- `测试与验收状态`: 偏弱

一句话概括：`前端主干已经搭好，但深功能和可验证性还没有补齐。`

- 服务端入口和主框架已经稳定存在：
  - `server/main.py`

- 已有完整 `app-v1` 接口注册：

  - `Authorization: Bearer`
  - `X-App-Token`
  - WebSocket `token`
- 已有事件流恢复：
  - `last_event_id`
  - `replay_limit`
  - `system.hello`
  - replay / refetch 机制
- 聊天后端主链路已经成型：
  - App 发消息到 session
  - MessageBus 入队
  - AgentLoop 处理
  - 事件流返回 `created/progress/completed/failed`
- `settings/tasks/events/notifications/reminders` 已是真实后端能力，不再是“未提供”
- 这些业务数据当前由 `server/services/app_api/*.py` 和 JSON store 持久化
- 已有 Todo / Calendar summary 读写和事件广播
- 已有设备状态快照、设备播报、设备命令入口
- 已有设备 WebSocket 通道、状态机、ASR、TTS、天气推送、时间推送
- 已有 WhatsApp bridge

### 后端仍未完成或需要注意的部分

- `reminders` 目前只是 CRUD 数据，未接入真实调度执行
  - 仓库里虽然有 `nanobot/cron/service.py`
  - 但当前 runtime/bootstrap 没有把 cron service 接进主运行链路
- `wake_word` 和 `auto_listen` 目前只是设置字段，没有看到真正的唤醒词检测链路
- 人格系统没有独立实现层
- 场景模式（专注 / 下班 / 会议）没有专门编排服务
- 外部日历同步没有实现
- 手机推送通知没有实现
- 电脑控制虽然底层 agent 工具存在，但还没有被封装成稳定、明确、可验收的产品功能链路

### 设备与固件相关的真实进度

- 固件 demo 已有：

## 后续工作建议

### 第一优先级：补功能闭环，不再纠结旧方案

当前项目已经是 Nanobot 路线，后续建议直接按“功能对齐”推进，而不是再拿旧的 OpenClaw / FastAPI 方案做比较。

- 升级本机 Flutter/Dart 到满足 `pubspec.yaml` 的版本
- 完成 `flutter pub get`
- 跑通 `flutter test`
- 给 provider/service 层补真实测试，不要只保留 1 个 smoke test

- Chat 补会话切换/创建 UI

- 把 device commands 做成真正的控制面板
- 完成手机端语音输入
- 视需求决定是否做局域网发现和手机推送

### 第四优先级：补后端高层业务逻辑

- 把 reminders 接到真实调度执行链路
- 明确 Todo / Calendar / Reminder 之间的数据关系
- 完成设备提醒、通知、播报联动
- 决定是否保留 JSON store，或者后续迁到更稳定的数据层

### 第五优先级：补产品级功能

- 人格系统
- 场景模式
- AI 日历自然语言处理与外部同步
- 电脑控制的正式产品化能力
- 设备端完整 TTS 播放与更多传感器交互
