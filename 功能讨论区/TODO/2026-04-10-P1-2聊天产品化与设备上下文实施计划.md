## 调研结论摘要

### 前端现状

- `app/lib/screens/chat/chat_screen.dart`、`app/lib/widgets/chat/chat_session_panel.dart`、`app/lib/widgets/chat/message_bubble.dart` 已有基础聊天界面，但只支持新建/切换会话，没有 pin、archive、归档筛选，也没有来源徽标。
- `SessionModel` 已有 `pinned / archived` 字段，但 `ChatService` 只暴露 `list/create/get/postMessage`，前端没有 session 更新 API。
- `MessageModel` 目前主要解析 planning metadata，消息头只显示 `You / Assistant`，没有把 `capture_source / interaction_surface / source_channel` 产品化。
- `NotificationModel` 和 `ReminderModel` 已经能解析 `source_channel / source_session_id / source_message_id`，但 `notification_panel.dart` 和 `reminder_panel.dart` 没有展示来源。

### 后端现状

- `server/services/app_runtime.py` 目前只有 `GET/POST sessions` 和 `GET/POST messages`，没有 session patch，因此 `pin/archive/title` 只存在于 session metadata，未对外可操作。
- `handle_list_sessions()` 已支持 `archived` 查询参数，但前端没有入口，说明后端“归档视图”只做了一半。
- assistant 完成消息在 `on_outbound_published()` 里用 `metadata={"task_id": task_id}` 广播，导致原本在 `msg.metadata` 里的 `capture_source / interaction_surface / reply_language / tool_results / app_session_id` 没有完整透传到 App。
- `_extract_message_metadata()` 只提取少量字段，session 序列化出来的历史消息也会丢掉更细的来源上下文。

### 设备上下文问题定位

- `DesktopVoiceService` 虽然会给桌面麦克风语音写 `app_session_id`，但默认永远落到 `default_app_session_id`，也就是 `app:main`；当前前端切到其他会话时，设备按住说话仍不会跟着当前会话走。
- `DeviceChannel` 直采 `device_mic` 语音时完全没有 `app_session_id`，因此会进入 `device:esp32` 这条独立 session，和 App 多会话上下文断开。
- 结论：这不是单纯 UI 问题，而是“active app session 没有同步到后端”和“设备入口没有绑定 active app session”的组合问题。

## 产品决策

### 会话管理

- 保留当前 `/app/chat` 单页结构，不新增独立“归档页”路由。
- 在 Chat 左侧会话面板内增加 `Active / Archived` 视图切换。
- pin 只影响排序，不强行单独拆出新的信息架构。
- archive 后默认从 Active 列表消失，但仍可在 Archived 视图恢复。

### 标题自动生成

- 新会话保持用户手填标题优先。
- 若会话标题仍是默认占位标题（如 `新对话 / New conversation / 主对话衍生占位`），则在首条有意义用户消息进入后自动生成标题。
- 自动标题只在“尚未人工改名”时覆盖，避免把用户手工标题改掉。

### 来源标识口径

- `device_press + desktop_mic`：显示为“设备按住说话”
- `device_press + device_mic`：显示为“设备语音直采”
- `app + text`：显示为“App 文本”
- `external channels`：显示为“外部渠道”，并优先落具体 channel 名称，如 `WhatsApp / Telegram / Slack`
- assistant 回复沿用触发它的来源上下文，保证用户能看懂“这条回复是接着哪条入口来的”

### 设备上下文

- 后端维护一个当前 active app session。
- App 在连接、切换会话、创建会话后同步 active session 到后端。
- 桌面麦克风和设备直采在缺省情况下都绑定 active app session，而不是各走各的会话。
- 如果没有 active app session，才回退到 `default_session_id`。
