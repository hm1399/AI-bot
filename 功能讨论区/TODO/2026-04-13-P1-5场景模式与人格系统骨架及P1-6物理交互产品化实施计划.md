# P1-5 场景模式与人格系统骨架 + P1-6 物理交互产品化 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在当前 `Flutter App + app-v1 + desktop_mic + DeviceChannel + Planning + ComputerControl` 主干上，补一层真正可落地的 `scene mode + persona` 运行时骨架，并把 `shake / tap / hold` 物理交互做成产品能力，同时不破坏现有 `按住设备说话 + desktop_mic` 主路线，也不误报 WeChat 已支持语音直接发送。

**Architecture:** 后端新增一层轻量 `experience` 服务，统一负责 `scene/persona` 默认值、会话级覆盖、运行态解析、physical interaction 路由和结果结构化；前端只在 `Settings / Chat / Home / Control Center` 上加轻量入口与只读摘要，不新开主导航；固件继续做“薄事件层 + 反馈执行层”，只负责上报 `hold/tap/shake` 与消费已有屏幕/表情/音频反馈，玩法判断和安全边界全部留在服务端。

**Tech Stack:** Flutter + Riverpod, Python `aiohttp`, existing `AppRuntimeService`, `PlanningBundleService`, `ComputerControlService`, `DeviceChannel`, `DesktopVoiceService`, Nanobot `AgentLoop`, ESP32 Arduino demo firmware, macOS `open` / Shortcuts / Apple Events / Accessibility fallback, existing WebSocket device protocol

## 范围与约束

- 本计划只覆盖总 TODO 里的 `P1-5` 与 `P1-6`，外加与它们直接相关的 WeChat 发送边界回填。
- 不顺手做这些范围外事项：
  - Flutter App 自己的直接麦克风采集
  - `device_mic` 恢复为主路线
  - 原生 macOS Focus Filter / App Intents 扩展
  - 微信自动发送消息能力
  - 新的主导航页或大规模 UI 重构
- 前端必须保持当前 `Connect / Home / Chat / Agenda / Tasks / Control / Settings` 的 IA 和现有 Linear 风格，不破坏现有 UI/UX。
- 第一版优先交付“骨架和产品契约”，不是做一个复杂学习型系统：
  - `scene mode` 先做固定规则与手动切换
  - `persona` 先做四个核心维度与少量 preset
  - `physical interaction` 先做 `hold / tap / shake` 三类动作
- 当前用户约束继续生效：本轮只出计划和文档回填，不跑测试，不做 git 提交。

## 调研结论摘要

### 代码现状

- 前端已经有足够集中的全局状态和能力模型，适合承接 `scene/persona/physical interaction`：
  - `AppState` 已聚合 `capabilities / runtimeState / sessions / settings / planning*`
  - `Home / Chat / Control Center / Settings` 已是明确的运行态、配置态、操作态入口
  - `Message / Reminder / Notification / Planning bundle` 已带一部分来源 metadata
- 后端已经具备可复用的三条主链：
  - `AppRuntimeService`：统一 HTTP + WebSocket 聚合层
  - `PlanningBundleService`：统一 `bundle_id / source_* / interaction_surface / capture_source / voice_path`
  - `ComputerControlService`：统一结构化动作、风险分级、确认流和审计
- 设备侧并不是“空白”：
  - demo 固件已经上报 `device_status` 和 `device_command_result`
  - `long_press / long_release` 已经接到 `desktop_mic`
  - 设备能消费 `state_change / display_update / face_update / status_bar_update / audio_play`
- 真正的缺口在于“产品骨架没建立”：
  - `scene / persona` 没有正式 schema、settings 字段、runtime state、session override
  - `shake` 目前还是硬编码 prompt
  - `tap` 的产品语义没有统一协议
  - `wake_word / auto_listen` 仍只是配置占位

### WeChat 当前边界

- 当前仓库 `server/services/computer_control/adapters/wechat.py` 的真实能力仍是：
  - 打开 WeChat
  - 复制消息草稿到剪贴板
  - 返回 `manual_step_required` 与 `manual_steps`
- `wechat_send_prepared_message` 目前会直接抛 `adapter_unavailable`，不是前端漏接，而是后端明确禁用。
- 本机验证结果：
  - `osascript -e 'tell application "WeChat" to activate'` 可用
  - `sdef /Applications/WeChat.app` 返回 `error -192`
- 结论：
  - 本轮必须继续维持 `draft-first + confirm/manual-step-required`
  - `P1-6` 的 tap 确认流不能被误解成“微信语音直发已可用”

## 产品决策

### 1. P1-5 采用“场景模式 + 人格配置”两层模型

- `scene mode` 负责“什么时候怎么做”：
  - `focus`
  - `offwork`
  - `meeting`
- `persona profile` 负责“说成什么样”：
  - `tone_style`
  - `reply_length`
  - `proactivity`
  - `voice_style`
- 第一版不要做开放式人格编辑器，先做 `preset + 最小参数`：
  - preset 只是便捷入口
  - 最终后端总是存四个解析后的核心字段
- 优先级固定为：
  - `session override`
  - `runtime temporary override`
  - `scene default`
  - `global default`

### 2. Scene 与 Persona 的产品影响范围

- Scene 第一版必须真实影响这些运行面：
  - Chat 回复简洁度和主动性
  - TTS 是否允许播报、偏简短还是偏陪伴
  - Reminder/notification 的打断级别
  - `shake / tap` 的可触发性和结果风格
- Persona 第一版必须真实影响这些输出面：
  - 文本语气
  - 回复长度
  - 主动建议强度
  - 语音播报风格标签
- 第一版不做这些高成本项：
  - 自动学习用户性格
  - 按历史统计自动切 scene
  - 不同 persona 的独立长期记忆仓

### 3. P1-6 采用“固件薄事件层 + 后端路由 + 前端可视化”模式

- `hold`：
  - 继续保留为 `按住说话`
  - 不恢复 `device_mic` 主路线
- `tap`：
  - 只在存在明确待确认上下文时启用产品语义
  - `1` 次允许
  - `2` 次拒绝
  - `3` 次打断
- `shake`：
  - 不再走固定 prompt
  - 改成结构化玩法路由：
    - `fortune`
    - `decision`
    - `random`
- 所有物理交互第一版都遵循这组 guard：
  - `listening / transcribing / responding / speaking` 时禁止 `shake`
  - 高优先级 reminder 等待确认时禁止彩蛋型 `shake`
  - 设备离线、bridge 不可用、命令 pending 时禁止冲突动作
  - 连续短时间重复触发必须节流

### 4. Scene 与 Physical Interaction 要联动，但只做固定规则

- `focus`：
  - 回复更短
  - 关闭陪伴型随机摇一摇
  - 非高优先级提醒尽量少打断
- `offwork`：
  - 允许陪伴型和趣味型 shake
  - 允许相对更暖的 persona
- `meeting`：
  - 回复极短
  - 尽量静默
  - 关闭随机娱乐型 shake
  - tap 只保留确认/拒绝/打断
- 不做学习逻辑；只做可测试的固定映射表。

### 5. WeChat 继续保持硬边界

- `scene/persona` 只能影响“怎么提示和确认”，不能伪装成“已自动发送”。
- `tap` 可以确认“允许执行某个结构化动作”，但不能绕过 `wechat_send_prepared_message` 当前不可用的事实。
- 所有涉及消息外发的产品 copy 继续维持：
  - `draft_ready`
  - `awaiting_confirmation`
  - `manual_step_required`

## 第一版验收标准

- 用户能在不新增主页面的前提下看到并切换当前 `scene mode` 和 `persona`。
- 后端 runtime 能稳定返回 `active_scene_mode / active_persona / physical_interaction` 状态。
- Agent 回复和 planning / computer action / message metadata 能留下 `scene/persona/interaction` 审计痕迹。
- `shake` 不再返回固定笑话 prompt，而是返回结构化 `fortune / decision / random` 结果。
- `tap` 在待确认上下文里能稳定映射为 `允许 / 拒绝 / 打断`，且不会误触发旧 `single / double` 录音 toggle 语义。
- 任何物理交互都不破坏当前 `按住设备说话 + desktop_mic` 主路线。
- WeChat 相关展示继续明确为 `manual_step_required`，不误报“已发送”。
