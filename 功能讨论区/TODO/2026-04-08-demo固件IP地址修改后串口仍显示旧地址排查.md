# 2026-04-08 demo 固件 IP 地址修改后串口仍显示旧地址排查

## 任务目标

排查为什么修改了 `firmware/arduino/demo/demo.ino` 中的 IP 地址相关内容后，设备串口依然输出：

- `WiFi connected! IP: 172.20.10.2`

明确串口输出的真实来源，是代码逻辑、编译/烧录流程、运行时网络分配，还是读取了其他配置。

## 排查计划

1. [x] 核对 `firmware/arduino/demo/demo.ino` 中与 Wi‑Fi、IP、串口打印相关的代码路径。
2. [x] 查找仓库内是否还有其他 demo 固件、构建产物或脚本会覆盖该串口输出。
3. [x] 对比当前串口日志与代码文案，确认日志中的 IP 是静态配置值、DHCP 分配值，还是 `localIP()` 实时读取值。
4. [x] 如发现异常链路，整理根因与建议，不擅自改动固件逻辑。
5. [x] 按仓库要求回填本计划文件执行结果，并等待你确认后再决定是否继续后续动作。

## 执行结果

- 已确认当前串口日志中的 `=== AI-Bot Demo (desktop voice + speaker playback) ===` 对应 `demo.ino` 现有 `setup()` 文案，说明设备运行的是当前这版新固件，不是旧版草图。
- 已确认 `firmware/arduino/demo/demo.ino` 中真正被你修改的服务端地址是 `WS_HOST`，当前位于 `webSocket.begin(WS_HOST, WS_PORT, WS_PATH)`。
- 已确认串口输出 `WiFi connected! IP: ...` 的来源是 `WiFi.localIP().toString()`，它打印的是设备连上当前 Wi‑Fi 后由路由器或热点分配到的本机局域网地址，不是 `WS_HOST`。
- 已确认当前草图内没有 `WiFi.config(...)`、`IPAddress(...)` 等静态 IP 配置逻辑，因此 `172.20.10.2` 属于运行时网络分配结果。
- 已检查 `firmware/arduino/demo` 目录下没有本地构建产物或其他 demo 草图覆盖该串口文案的证据。
- 结论：你修改 `demo.ino` 里的 IP 地址后，串口依然看到 `WiFi connected! IP: 172.20.10.2`，并不表示改动未生效；它只是设备自己的 Wi‑Fi 地址碰巧仍是 `172.20.10.2`。真正被修改的 `WS_HOST` 当前没有被串口打印出来。
