# 机器人首次配对与前端配网 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把 `firmware/arduino/demo/demo.ino` 中烧录期写死的 `WIFI_SSID / WIFI_PASS / WS_HOST / WS_PORT / WS_PATH` 迁移成桌面前端驱动的运行时配对配置，并把“第一次连接机器人时，用户用线连接电脑并长按触摸盘完成配对”做成正式产品流程。

**Architecture:** 保持现有 `Flutter App -> AI-bot server -> /ws/device -> ESP32-S3 demo firmware` 主链路不变，只新增一条桌面限定的 `Flutter App -> USB CDC Serial -> ESP32` 首配/重配链路。固件通过 Arduino `Preferences` 持久化 WiFi、WebSocket 与设备认证配置；前端在 `Connect` 入口页增加符合 `app/DESIGN.md` 的配对面板；后端只补最小的配对 bundle 接口，避免把现有 app 连接契约和机器人配网状态混成一层。

**Tech Stack:** Flutter desktop + Riverpod + `flutter_libserialport`, Python aiohttp backend, ESP32-S3 Arduino demo firmware + `Preferences` (NVS) + USB CDC/Serial

- 前端 UI 必须遵守 [`app/DESIGN.md`](/Users/mandy/Documents/GitHub/AI-bot/app/DESIGN.md)：

  - 不恢复“假 LAN 扫描”按钮；若需要本机候选地址，只能做本机网卡候选提示，不做网络扫描叙事。
  - 维持 Linear 风格的暗色、紧凑、工具化面板，不破坏现有 shell 视觉语言。
- 本轮计划默认“前端配置”指 `app/` 下 Flutter 桌面端，而不是旧 `software/flutter_application_1/` 或 Web 专用流程。
- 本轮不直接引入 BLE / SoftAP 配网，不把 Arduino demo 固件整体迁移到 ESP-IDF。
- 当前只写计划，不跑测试；下面所有验证命令都只是后续执行清单，必须等你确认代码修改后再跑。

## 关键假设

- 用户会在运行 AI-bot server 的同一台电脑上打开桌面 App，并通过 USB-C 把机器人接到这台电脑完成首配。
- 机器人首次入网依赖 USB CDC 串口，而不是 BLE、SoftAP 或手机 App。
- 若服务端启用了 `DEVICE_AUTH_TOKEN`，配对流程必须把设备接入 token 一并安全地下发给设备；否则会出现“已连 WiFi 但进不了 `/ws/device`”的假成功。
- 首个版本优先解决“首次配对 + 可恢复重配”，不在本轮扩展“远程服务器跨机器配对”。

- 前端现状：
  - [`app/lib/screens/connect/connect_screen.dart`](/Users/mandy/Documents/GitHub/AI-bot/app/lib/screens/connect/connect_screen.dart) 已是独立入口流，负责 `App -> server` 的 host/port/token 校验，不负责机器人配网。
  - [`app/lib/services/storage/auth_storage_service.dart`](/Users/mandy/Documents/GitHub/AI-bot/app/lib/services/storage/auth_storage_service.dart) 当前只持久化 operator console 的连接配置；若直接把机器人 WiFi 凭据塞进这份模型，会污染现有 contract。
  - 当前 UI 明确写了“LAN scan is intentionally removed”，所以配对设计不能退回“扫一扫找设备”的假捷径。
- 后端现状：
  - [`server/services/app_runtime.py`](/Users/mandy/Documents/GitHub/AI-bot/server/services/app_runtime.py) 已暴露 `/api/app/v1/device`、`/api/app/v1/device/commands`，但还没有只为配对服务的 bundle 接口。
  - [`server/channels/device_channel.py`](/Users/mandy/Documents/GitHub/AI-bot/server/channels/device_channel.py) 的设备接入认证支持 `Authorization: Bearer ...`、`X-Device-Token` 或 query `token`，因此只要固件能拿到 token，就无需改动设备端鉴权模型。
- 官方与依赖资料：
  - Espressif 官方 `Wi-Fi Provisioning` 文档说明 `wifi_prov_mgr` 支持 `BLE / SoftAP / Console(serial)` 三种 transport，但这是 ESP-IDF 配网栈，不是当前 Arduino demo 的最小改法。来源：<https://docs.espressif.com/projects/esp-idf/en/v5.1.3/esp32/api-reference/provisioning/wifi_provisioning.html>
  - Espressif 官方 `Preferences` 文档说明该库使用 ESP32 的 NVS，数据在重启和断电后仍保留，适合存储多组小配置值。来源：<https://docs.espressif.com/projects/arduino-esp32/en/latest/api/preferences.html>
  - `flutter_libserialport` 当前支持 `Linux / macOS / Windows / Android`，适合桌面 Flutter 直接做串口首配。来源：<https://pub.dev/packages/flutter_libserialport>

## 方案选择

### 推荐方案：桌面前端 USB 串口配对 + 固件 NVS 持久化 + 连接页双面板接入

推荐理由：

- 最贴合你的首要约束：第一次连接机器人时，用户就是插线到电脑，并长按触摸盘触发 pairing mode。
- 与仓库现状最兼容：现有固件是 Arduino sketch，不需要为配网单点需求把整条固件工程切去 ESP-IDF。
- 与 `DESIGN.md` 最一致：把配对放在 `Connect` 页作为“进入主壳层前的真实设备准备动作”，不会把主壳层污染成 setup wizard。
- 安全边界更清晰：WiFi 密码与设备 token 只在“已验证的 App 会话 -> 一次性串口下发 -> 设备 NVS”这条链路中出现，不必长期落在前端明文配置里。

### 不采用的方案

1. **继续烧录期写死常量**
   - 缺点：每换 WiFi、换电脑 IP、换端口都要重编译重烧录，和“前端配置”目标直接冲突。

2. **直接上 BLE / SoftAP 统一配网**
   - 优点：官方有现成能力。
   - 当前不采用的原因：当前固件是 Arduino demo，直接接入 ESP-IDF provisioning manager 会引入较大的框架迁移、内存和调试成本；同时你已经明确首配必须“插线 + 长按触摸盘”，没必要先走无线 transport。

3. **只做“设备在线后从后端下发新配置”**
   - 缺点：解决不了首次入网和首次指向电脑局域网 IP 的冷启动问题；一旦 WiFi 配错，设备根本无法上线收新配置。

## 设计收口

### 配对 UX 落点

- 首配入口放在 `Connect` 页，不新增独立 route。
- 版式延续当前双栏结构：
  - 左栏继续是 operator console 的连接说明与状态摘要。
  - 右栏在现有 `Connection Setup` 下方或同级增加 `Robot Pairing` 面板。
- `Robot Pairing` 面板保持紧凑控制台风格，只展示必要字段：
  - 串口设备选择
  - WiFi SSID
  - WiFi Password
  - 机器人连接的服务端地址（默认取当前已验证连接；若是 `localhost / 127.0.0.1 / 0.0.0.0`，要求用户显式改成局域网 IPv4）
  - 端口与路径（默认 `8765` 和 `/ws/device`，通常只读）
  - 状态提示：`USB Connected / Pairing Armed / Writing Config / Waiting Device Online / Paired`
- 交互步骤：
  1. 用户先让 App 成功连接 server。
  2. 用户用 USB 连接机器人到电脑。
  3. 用户长按触摸盘进入 pairing mode。
  4. 前端识别到串口握手事件后允许发送配置。
  5. 写入成功后设备保存到 NVS、重启并尝试连 WiFi + `/ws/device`。
  6. App 通过 runtime/device API 或事件流等待设备上线，完成配对闭环。

### 配置归属原则

- `ConnectionConfigModel` 继续只负责 `App -> server`，不要混入机器人 WiFi 字段。
- 新增单独的 `DevicePairingDraftModel` / `DevicePairingBundleModel`，专门负责 `App -> robot(serial)`。
- 前端只持久化非敏感默认值：
  - 上次 SSID
  - 上次 LAN host
  - 上次串口名
  - 端口 / path
- 默认不把 WiFi 密码和设备 token 长期存进 `SharedPreferences`。

### 串口协议建议

- 使用 newline-delimited JSON，避免引入二进制 framing 复杂度。
- 固件上报：

```json
{"type":"pairing.status","data":{"state":"idle","reason":"await_long_press","device_id":"esp32s3-xxxx","firmware":"demo-2026-04-11"}}
{"type":"pairing.status","data":{"state":"armed","reason":"touch_long_press"}}
{"type":"pairing.result","data":{"ok":true,"requires_restart":true}}
```

- 前端下发：

```json
{
  "type": "pairing.apply",
  "data": {
    "schema_version": 1,
    "wifi": {
      "ssid": "AAAAA",
      "password": "92935903"
    },
    "server": {
      "host": "192.168.1.23",
      "port": 8765,
      "path": "/ws/device",
      "secure": false
    },
    "auth": {
      "device_token": "device-local-123"
    }
  }
}
```

- 设计要求：
  - 固件先校验 payload 完整性，再写入 NVS。
  - 配置写入成功前，不覆盖旧配置。
  - 支持 `pairing.clear` 或“启动期长按清空配置并重入配对态”的恢复路径。

### 设备端状态机建议

- `UNPROVISIONED`
  - 无已保存配置时默认进入，屏幕提示“请插线并长按配对”。
- `PAIRING_ARMED`
  - 检测到 USB 串口在线且触摸盘长按满足阈值后进入。
- `PAIRING_APPLYING`
  - 正在校验并写入 NVS。
- `PAIRING_RESTARTING`
  - 写入成功后准备重启。
- `PROVISIONED_BOOT`
  - 正常从 NVS 读取配置，连接 WiFi 与 WebSocket。
