# P1 智能待办 / 提醒 / 日历（macOS App）Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 完成 `P1-1 智能待办 / 提醒 / 日历` 的本地工作台闭环：在现有 `app-v1 + WebSocket + JsonCollectionStore` 基线上，让用户能通过语音或文本创建结构化 `Task / Reminder / Event`，并在现有 macOS App 5-tab 壳层内看到统一的 planning workbench、自动摘要、真实 reminder 调度与本地冲突视图。本轮明确不做原生 macOS `Calendar / Reminders` 接入，也不做外部日历同步。

**Architecture:** P1 继续站在现有 `Flutter App + aiohttp AppRuntimeService + JsonCollectionStore + ReminderScheduler + WebSocket event stream` 主干上，不推翻现有 CRUD 和壳层。推荐做法是在现有 `Task / Event / Reminder / Notification` 四类资源之上增加统一的 `planning bundle / projection / summary` 派生层：用同一个 `bundle_id` 串起一次语音/文本创建出的资源，用 `planning projection` 统一计算时间轴、冲突与工作台摘要；前端则在 `AppState + AppController` 上增加 planning workbench 状态与服务，由 `/app/tasks` 承接桌面工作台，`/app/home` 保留概览，`/app/chat` 回显结构化创建结果。

**Tech Stack:** Python `aiohttp` 后端、Nanobot `AgentLoop` + 自定义 agent tool、`JsonCollectionStore`、`ReminderScheduler`、Flutter + Riverpod + GoRouter macOS App。

---

## 范围边界

- 本轮计划只覆盖 `P1-1 智能待办 / 提醒 / 日历` 的“本地 JSON 工作台版”。
- 这轮明确按 `macOS App` 作为产品载体思考，不再按手机端信息架构设计。
- `P1-5 场景模式` 本轮不纳入。
- 不接 `EventKit`、不做原生 `Calendar / Reminders` 集成，也不提前做外部日历同步；先把本地模型、冲突检测、摘要和工作台做稳。
- 不破坏现有前端 UI/UX 风格，不新增底部导航，不重做前端桌面适配壳层。
- 不新增 `/app/planning` 或 `/app/calendar` 路由，继续沿用现有 5-tab 壳层。
- 不移除 `runtime/todo_summary.json` 与 `runtime/calendar_summary.json` 这套兼容输出；P1 只能把它们改成“自动派生后的兼容写回”。
- 不把 `runtime task` 和 `todo task` 混为一谈，后续实现里必须明确区分。
- 本任务只更新计划与调研口径；实现完成后仍需等用户确认，再跑测试、再 git。

## 调研结论

### 当前前端事实

- 当前壳层已经固定为 5-tab 桌面路由：`/app/home`、`/app/chat`、`/app/tasks`、`/app/control`、`/app/settings`，并由 `AppScaffold` 统一承接 sidebar / bottom dock；P1 不应新增 `planning` 新入口。
- `app/lib/screens/tasks/tasks_screen.dart` 现在已经具备 Task / Event CRUD，但主视图仍是 `Tasks / Events` 二选一列表页，适合作为工作台主入口，不适合继续堆散功能。
- `app/lib/screens/home/home_screen.dart` 已经有 `todo_summary`、`calendar_summary`、当前运行任务、任务队列、通知与提醒概览，适合继续承接概览层。
- `app/lib/screens/control_center/control_center_screen.dart` 当前同时承载设备控制、Notifications、Reminders，说明“计划类信息”现在分散在多个页面；它更适合保留为兼容入口而不是主工作台。
- `app/lib/screens/chat/chat_screen.dart` 已经是 App 文本入口，未来用户通过自然语言创建 Task / Reminder / Event 会先经过这里；但 `app/lib/widgets/chat/message_bubble.dart` 现在只渲染文本和状态，没有把 `message.metadata` 里的结构化创建结果展示出来。
- `app/lib/providers/app_state.dart` 与 `app/lib/providers/app_providers.dart` 仍是前端真实状态中心；`home_provider.dart`、`tasks_provider.dart`、`events_provider.dart`、`control_center_provider.dart` 目前都只是薄 selector，不是各自独立的状态机。
- `app/lib/models/connect/bootstrap_model.dart` 里的 `runtime` 与 `app/lib/models/home/runtime_state_model.dart` 共同承载当前“工作台初始快照”，但前端还没有单独的 `PlanningModel / PlanningService / WorkbenchState`。
- `app/lib/services/home/runtime_service.dart` 虽然已经有 `fetchTodoSummary()` 和 `fetchCalendarSummary()`，但现状里的摘要刷新仍主要依赖 `fetchRuntimeState()` / `refreshRuntime()`。
- 当前前端实时事件处理缺口明显：
  - 已处理 `task.* / event.* / reminder.created|updated|deleted`
  - 未处理 `notification.created`
  - 未处理 `reminder.triggered`
  - `system.hello.resume.should_refetch_bootstrap` 场景下只会 `refreshRuntime()`，不会自动重拉 `bootstrap + planning` 数据

### 当前后端事实

- `server/services/app_runtime.py` 已经是本地工作台后端：它注册了 bootstrap、sessions/messages、tasks/events/notifications/reminders CRUD、runtime state/stop、todo/calendar summary、capabilities 和 WebSocket 事件流。
- `server/services/app_api/resource_service.py` 里 `task / event / reminder / notification` 仍是四个独立 `JsonCollectionStore`，数据真实落在 `runtime/*.json`；当前没有共享关系模型，也没有跨资源事务。
- `server/services/app_runtime.py` 里的 `todo_summary / calendar_summary` 仍是“接收 payload -> 校验 -> 落盘 -> 广播”的兼容摘要接口，不是自动派生。
- `server/services/reminder_scheduler.py` 已经是唯一真实会执行的时间驱动模块，并且后端 reminder 已经有一部分运行时字段：
  - 已有 `next_trigger_at`
  - 已有 `last_triggered_at`
  - 已有 `last_error`
  - 仍缺 `snoozed_until / completed_at / status`
  - 仍没有延后、完成、过期与冲突检测的统一行为中心
- `server/services/app_runtime.py` 当前会在提醒触发时广播 `reminder.updated -> reminder.triggered -> notification.created`，但并没有 `planning.*` 路由或 projection service。
- `server/nanobot/agent/loop.py` 当前只注册通用默认工具，没有 `Task / Reminder / Event` 的结构化 planning tool；同时 `server/bootstrap.py` 现在也没有把 `cron_service` 或 app-resource service 注入到 AgentLoop，因此“语音直接创建计划项”不能只改 `loop.py`，还必须补依赖注入接线。
- 现有测试主要覆盖 CRUD、事件流 replay、summary 手动写入、会话锁和 `/stop`，还没有证明“智能待办 / 提醒 / 日历”闭环。

### 产品层结论

- 如果继续在 `TasksScreen + ControlCenter + Home` 上分散补 UI，P1 会做成“能用但不统一”的半成品。
- 如果现在就接原生 macOS 日历/提醒或外部同步，会把 P1 拉成 P2 级别的系统集成任务，节奏过重。
- 最合理的 P1 路线是：
  - 保留现有三套资源和固定 5-tab 壳层
  - 增加统一的 `planning bundle + projection + summary` 派生层
  - 给 agent 补一层稳定的 app-resource 结构化工具
  - 让 `/app/chat` 继续承担文本创建入口
  - 让 `/app/tasks` 汇总成桌面工作台
  - 让 `/app/home` 保留概览，`/app/control` 保留提醒与通知兼容入口

## 方案对比

### 方案 A：直接在现有页面补功能

- 优点：改动最少，最快看到页面变化。
- 缺点：Task / Reminder / Event 的关系仍然散；冲突检测、自动摘要、语音创建都只能写成补丁。
- 结论：不推荐，后面还得返工。

### 方案 B：保留三套资源，新增统一 planning 投影视图层

- 优点：改动范围可控，兼容现有路由和 JSON store；能先完成联动、工作台、自动摘要和冲突检测。
- 缺点：需要新增派生层，`app_runtime.py` 与 `app_providers.dart` 两个热点文件必须串行收口。
- 结论：推荐，适合本次 P1。

### 方案 C：现在就接原生 macOS / 外部日历同步

- 优点：长期产品形态更完整。
- 缺点：当前代码没有系统日历、系统提醒、系统通知的接入基础，会直接把本轮变成系统集成项目。
- 结论：本轮不做。

## 推荐实现口径

### 统一关系模型

- 不取消 `Task / Reminder / Event` 三种资源。
- 不绕开现有 `JsonCollectionStore`；P1 先在现有资源上补 planning 元数据与 bundle service。
- 新增跨资源公共字段，至少包括：
  - `bundle_id`
  - `created_via`
  - `source_channel`
  - `source_message_id`
  - `source_session_id`
  - `linked_task_id`
  - `linked_event_id`
  - `linked_reminder_id`
- 一次语音/文本创建出来的一组资源共享同一个 `bundle_id`。
- `bundle_id` 是 P1 的最小统一键，比直接重构单表更稳。
- 跨资源联动更新必须经过 `planning_bundle_service`，不能让页面层或 tool 各自写不同 JSON 文件。

### 时间轴与冲突检测

- 新增 `planning projection`，统一把三类资源投影成时间轴项。
- P1 冲突检测先做本地规则，不做外部同步：
  - `event` 与 `event` 时间重叠
  - `reminder.next_trigger_at` 落进 `event` 时间窗
  - `task.due_at` 与高优先级事件/提醒撞时
- 时间排序统一改为“按业务时间排序”，不再按 `updated_at` 近似代替。

### 自动摘要

- `todo_summary` 从任务与提醒派生，不再依赖人工 POST 写入内容。
- `calendar_summary` 从事件与最近提醒派生，不再依赖人工 POST 写入内容。
- 对外继续保留现有 summary 字段和 `runtime/*.json` 兼容文件，避免前端 Home 卡片和现有测试先被打断。
- 允许追加新字段，但必须向后兼容当前 `HomeScreen`。

### 语音/文本直创计划项

- 语音和 App 文本都复用同一条 agent tool 路线。
- 不在 ASR 层做意图判断，意图判断放到 Agent 的 tool 决策层。
- 新增 planning tool，让 Agent 结构化创建：
  - `create_task`
  - `create_event`
  - `create_reminder`
  - `complete_task`
  - `snooze_reminder`
  - `list_today`
- planning tool 需要通过 `server/bootstrap.py` / `AgentLoop` 注入 app-resource service 或 app runtime facade，不能在 tool 内部直接 new `JsonCollectionStore`。
- 对时间信息不完整或歧义较大的输入，P1 应要求 agent 先澄清，不应静默创建错误提醒。

### macOS App 可视化

- 不新增新导航页，也不新增新 tab；继续沿用现有 5-tab 桌面壳层。
- `/app/tasks` 直接升级为桌面 Planning Workbench，承接统一的计划管理与时间轴。
- `/app/home` 继续承担总览页，只显示摘要、运行态和今日关键变化，不再承担完整计划编辑。
- `/app/chat` 继续承担自然语言文本入口，负责把“我明天 9 点提醒我开会”这类输入送进同一条 planning tool 路线，并把结构化创建结果明确回显给用户；它不是主工作台。
- `/app/control` 继续保留 `Reminders / Notifications` 兼容入口和运维视角，不另起一套 planning 逻辑。
- 建议布局按桌面工作台思路拆成 4 个区域，并在窄宽度下保持现有样式的垂直堆叠：
  - 今日概览
  - 待办列表
  - 日历时间轴
  - 提醒与冲突
- `HomeScreen` 继续保留为全局总览，但不再承载完整计划管理。
- `ControlCenter` 中的 Reminders/Notifications 可先保留兼容入口，等 P1 稳定后再决定是否收口。

## 风险与回避策略

- `AppController` 仍是前端核心热点文件：虽然现在有 `home_provider / tasks_provider / events_provider / control_center_provider` 这些 selector，但主要异步逻辑仍集中在 `app_providers.dart`，并行开发时这里依然最容易冲突。
- 当前还没有统一的 `PlanningModel / WorkbenchState`：任务、事件、提醒、通知、summary 仍分散在多套模型里，P1 实现前必须先定义“工作台真相源”。
- `AppRuntimeService` 已经很大：P1 新逻辑尽量下沉到 `planning services`，不要继续把业务规则塞回单文件。
- `JSON store` 没有事务：跨资源联动更新必须通过 bundle service 封装，避免页面层各写各的。
- `JsonCollectionStore` 还是整文件读写且缺少进程内锁：P1 只能在当前规模内使用，不能假设它天然适合高并发与大数据量。
- `runtime task` 与 `todo task` 命名冲突：实现时必须提前重命名口径，否则会在消息元数据和 UI 回显里混乱。
- `todo_summary / calendar_summary` 当前是 runtime snapshot 的内嵌字段，不是独立 summary service；P1 不能按旧文档再拆出一套重复拉取路径。
- `todo_summary / calendar_summary` 现有测试依赖其持久化文件存在；自动派生实现必须保留兼容写回。
- `notifications` 当前更接近读、已读、删除、清空的资源，不是完整创建型 CRUD；工作台实现不能误把它当成 Task / Event / Reminder 一样的主资源。
- 现有 `sortedTasksProvider`、`upcomingEventsProvider` 这类派生 provider 只是页面视图切片，不是通用 planner 查询模型；如果直接拿来当工作台数据源，会把排序和过滤规则写死。
- Reminder 的真实调度已经上线一部分，但前端实时事件消费不完整：实现时必须先补事件处理，再说工作台体验。
- `resource_service.py` 对 `repeat` 只校验“非空字符串”，`reminder_scheduler.py` 却会把未知值静默降级成 `daily`；P1 实现时需要先统一契约，避免前后端口径分叉。
- `AgentLoop` 目前没有 app-resource tool 的依赖注入路径；如果不先改 `server/bootstrap.py`，planning tool 会变成只存在于文件系统里的死代码。

## 计划完成定义

- 用户说一句自然语言，系统能稳定识别并创建 Task / Reminder / Event 中的一项或多项。
- 同一次创建出来的资源在数据上可关联，在 app 工作台里能看出关联。
- 这套闭环明确建立在本地 JSON 工作台之上，不依赖原生 macOS Calendar / Reminders 集成。
- 提醒支持重复、延后、完成、过期这四类真实行为。
- `todo_summary / calendar_summary` 自动刷新，不再依赖手动写入。
- `notification.created`、`reminder.triggered` 与 planning 级事件能被前端正确消费。
- 本地冲突检测和时间排序可见。
- macOS App 的 `/app/tasks` 成为统一可视化工作台。
