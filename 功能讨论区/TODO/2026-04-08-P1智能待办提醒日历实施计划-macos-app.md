# P1 智能待办 / 提醒 / 日历（macOS App）Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 完成 `P1-1 智能待办 / 提醒 / 日历`，让用户能通过语音或文本直接生成结构化 Task / Reminder / Event，并在 macOS App 里看到统一的可视化工作台，同时补齐联动、真实调度、自动摘要、本地冲突检测与时间排序。

**Architecture:** P1 不推翻现有 `app-v1 + WebSocket + JSON store` 主干，也不直接上外部日历同步。推荐做法是在现有 `Task / Event / Reminder` 三套资源之上增加一个统一的 `planning` 领域层与投影视图层：用同一个 `bundle_id` 把一次语音/文本创建出来的资源串起来，用 `planning projection` 统一计算工作台时间轴、冲突、摘要，再把结果通过现有 App 路由 `/app/tasks` 呈现为桌面工作台。

**Tech Stack:** Python `aiohttp` 后端、Nanobot `AgentLoop` + 自定义 agent tool、`JsonCollectionStore`、`ReminderScheduler`、Flutter + Riverpod + GoRouter macOS App。

---

## 范围边界

- 本轮计划只覆盖 `P1-1 智能待办 / 提醒 / 日历`。
- 这轮明确按 `macOS App` 作为产品载体思考，不再按手机端信息架构设计。
- `P1-5 场景模式` 本轮不纳入。
- 不提前做外部日历同步；先把本地模型、冲突检测、摘要和工作台做稳。
- 不破坏现有前端 UI/UX 风格，不新增底部导航，不重做前端桌面适配壳层。
- 不把 `runtime task` 和 `todo task` 混为一谈，后续实现里必须明确区分。

## 调研结论

### 当前前端事实

- `app/lib/screens/tasks/tasks_screen.dart` 现在只是 `Tasks / Events` 二选一列表页，适合继续复用为工作台主入口，但不适合继续堆散功能。
- `app/lib/screens/home/home_screen.dart` 已经有 `todo_summary`、`calendar_summary`、当前运行任务和任务队列卡片，适合继续承接概览层。
- `app/lib/screens/control_center/control_center_screen.dart` 目前还承载 `Reminders / Notifications`，说明“计划类信息”现在被拆散在多个页面。
- `app/lib/providers/app_state.dart` 与 `app/lib/providers/app_providers.dart` 已经把 `tasks / events / reminders / notifications / runtimeState` 放进同一个全局状态容器，适合在不重搭状态层的前提下做统一工作台。
- 当前前端实时事件处理缺口明显：
  - 已处理 `task.* / event.* / reminder.created|updated|deleted`
  - 未处理 `notification.created`
  - 未处理 `reminder.triggered`
  - 断线恢复时只会 `refreshRuntime()`，不会自动重拉完整 planning 数据

### 当前后端事实

- `server/services/app_api/resource_service.py` 里 `task / event / reminder / notification` 仍是四个独立 JSON store，当前没有共享关系模型。
- `server/services/app_runtime.py` 已经有完整 CRUD 路由、事件流和运行态快照，但 `todo_summary / calendar_summary` 仍是“手动写入的运行态摘要”，不是自动派生。
- `server/services/reminder_scheduler.py` 已经是唯一真实会执行的时间驱动模块，但它现在只会：
  - 计算 `next_trigger_at`
  - 到点创建一条 notification
  - 广播 `reminder.triggered`
  - 不会回写任务状态、不会延后、不会完成提醒、不会做冲突检测
- 现有 `AgentLoop` 只有 `filesystem / shell / web / message / spawn / cron` 等默认工具，没有 `Task / Reminder / Event` 的结构化工具，因此“语音直接创建计划项”现在没有合适的 agent 调用接口。

### 产品层结论

- 如果继续在 `TasksScreen + ControlCenter + Home` 上分散补 UI，P1 会做成“能用但不统一”的半成品。
- 如果直接重做底层存储成数据库，会把 P1 拉成 P2 级别重构，节奏过重。
- 最合理的 P1 路线是：
  - 保留现有三套资源
  - 增加统一的 `planning bundle + projection + summary` 层
  - 给 agent 补一层稳定的结构化工具
  - 在 `/app/tasks` 汇总成桌面工作台

## 方案对比

### 方案 A：直接在现有页面补功能

- 优点：改动最少，最快看到页面变化。
- 缺点：Task / Reminder / Event 的关系仍然散；冲突检测、自动摘要、语音创建都只能写成补丁。
- 结论：不推荐，后面还得返工。

### 方案 B：保留三套资源，新增统一 planning 投影视图层

- 优点：改动范围可控，兼容现有路由和 JSON store；能先完成联动、工作台、自动摘要和冲突检测。
- 缺点：需要多一个派生层，`app_runtime.py` 也必须继续收口职责。
- 结论：推荐，适合本次 P1。

### 方案 C：现在就切数据库并合并成单表

- 优点：长期最整洁。
- 缺点：范围过大，和当前代码基线不匹配，会阻塞 P1 交付。
- 结论：本轮不做。

## 推荐实现口径

### 统一关系模型

- 不取消 `Task / Reminder / Event` 三种资源。
- 新增跨资源公共字段，至少包括：
  - `bundle_id`
  - `created_via`
  - `source_channel`
  - `source_message_id`
  - `source_session_id`
  - `linked_task_id`
  - `linked_event_id`
  - `linked_reminder_id`
- 一次语音/文本创建出来的一组资源共享同一个 `bundle_id`。
- `bundle_id` 是 P1 的最小统一键，比直接重构单表更稳。

### 时间轴与冲突检测

- 新增 `planning projection`，统一把三类资源投影成时间轴项。
- P1 冲突检测先做本地规则，不做外部同步：
  - `event` 与 `event` 时间重叠
  - `reminder.next_trigger_at` 落进 `event` 时间窗
  - `task.due_at` 与高优先级事件/提醒撞时
- 时间排序统一改为“按业务时间排序”，不再按 `updated_at` 近似代替。

### 自动摘要

- `todo_summary` 从任务与提醒派生，不再依赖手动写入。
- `calendar_summary` 从事件与最近提醒派生，不再依赖手动写入。
- 对外继续保留现有 summary 字段，避免前端 Home 卡片先被打断。
- 允许追加新字段，但必须向后兼容当前 `HomeScreen`。

### 语音/文本直创计划项

- 语音和 App 文本都复用同一条 agent tool 路线。
- 不在 ASR 层做意图判断，意图判断放到 Agent 的 tool 决策层。
- 新增 planning tool，让 Agent 结构化创建：
  - `create_task`
  - `create_event`
  - `create_reminder`
  - `complete_task`
  - `snooze_reminder`
  - `list_today`
- 对时间信息不完整或歧义较大的输入，P1 应要求 agent 先澄清，不应静默创建错误提醒。

### macOS App 可视化

- 不新增新导航页，`/app/tasks` 直接升级为桌面 Planning Workbench。
- 建议布局按桌面工作台思路拆成 4 个区域：
  - 今日概览
  - 待办列表
  - 日历时间轴
  - 提醒与冲突
- `HomeScreen` 继续保留为全局总览，但不再承载完整计划管理。
- `ControlCenter` 中的 Reminders/Notifications 可先保留兼容入口，等 P1 稳定后再决定是否收口。

## 风险与回避策略

- `AppRuntimeService` 已经很大：P1 新逻辑尽量下沉到 `planning services`，不要继续把业务规则塞回单文件。
- `JSON store` 没有事务：跨资源联动更新必须通过 bundle service 封装，避免页面层各写各的。
- `runtime task` 与 `todo task` 命名冲突：实现时必须提前重命名口径，否则会在消息元数据和 UI 回显里混乱。
- 前端当前大量直接 `watch(appControllerProvider)`：P1 页面改造时应顺便多用 selector provider，避免工作台页面过度重建。
- Reminder 的真实调度已经上线一部分，但前端实时事件消费不完整：实现时必须先补事件处理，再说工作台体验。

## 计划完成定义

- 用户说一句自然语言，系统能稳定识别并创建 Task / Reminder / Event 中的一项或多项。
