# 语音删除行程未真正执行修复计划

> 仅覆盖本轮新增问题：用户通过语音说“删除这个行程 / 取消 18 号考试行程”后，助手口头确认已取消，但实际 event 仍存在，前端个人行程也继续显示。要求保持现有 Chat / Agenda UI 风格，不擅自改视觉样式，只修 planning 工具链与删除后的状态同步口径。

## 目标

- [x] 让语音侧“删除 / 取消行程”具备真实的 `event` / `reminder` 删除能力，不再只口头确认。
- [x] 删除成功后，通过现有 realtime 事件链把前端 Agenda / Tasks / Chat 状态同步更新到位。
- [x] 让聊天结构化摘要正确识别删除动作，避免删除成功后仍显示成误导性的 `Planning update / 1 task`。

## 调研结论

- [x] `server/logs/server_2026-04-15.log` 在 `2026-04-15 16:55:40` 识别到“删除这个星程。”后，到 `16:55:46` 直接回复“好的，取消了18号考试行程。”，中间没有任何 `planning(...)` tool call 记录。
- [x] 当前 planning tool [server/nanobot/agent/tools/planning.py](/Users/mandy/Documents/GitHub/AI-bot/server/nanobot/agent/tools/planning.py) 只支持 `create_task / create_event / create_reminder / complete_task / snooze_reminder / list_today`，没有 `delete_event` 或 `delete_reminder`。
- [x] 后端资源层与 App API 实际已经具备删除能力：
  - [server/services/app_api/resource_service.py](/Users/mandy/Documents/GitHub/AI-bot/server/services/app_api/resource_service.py) 已实现 `delete_event()` / `delete_reminder()`
  - [server/services/app_runtime.py](/Users/mandy/Documents/GitHub/AI-bot/server/services/app_runtime.py) 已暴露 DELETE 路由并广播 `event.deleted / reminder.deleted`
  - [app/lib/providers/app_controller_realtime.dart](/Users/mandy/Documents/GitHub/AI-bot/app/lib/providers/app_controller_realtime.dart) 已消费 `event.deleted / reminder.deleted`
- [x] 因此当前根因不是“前端没刷新”，而是 agent 根本没有可调用的删除工具能力，最终产生了口头幻觉式确认。

## 修复方案

- [x] 后端 planning tool：
  - 新增 `delete_event`、`delete_reminder` 动作，并补对应 backend protocol / bootstrap 委托。
  - 动作说明里显式补充“当用户说取消 / 删除某条行程或提醒时，应调用删除动作，不可仅自然语言确认”。
  - 当前实现优先支持显式 `event_id / reminder_id`，让已有 planning 上下文中的结构化资源 id 可以被删除动作真实消费。
- [x] 聊天结构化摘要：
  - 为新动作补充 action label，让删除成功后的卡片语义正确。
  - 不改现有卡片样式，只修动作识别口径。
- [x] 回归测试：
  - 补 planning tool 单测，覆盖 `delete_event / delete_reminder` 的真实返回与资源移除。
  - 补聊天 metadata 单测，覆盖 `delete_event / delete_reminder` 的动作标签识别。
