# P1-6 物理交互产品化剩余项确认与收口实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在不破坏现有前端 UI/UX、不中断 `desktop_mic` 主路线的前提下，收口 `P1-6 物理交互产品化` 中软件侧真实未完成的能力，并把“文档未回填”和“功能未落地”彻底分开。

**Architecture:** 继续复用现有 `experience runtime + device_channel + Flutter runtime panel + demo 固件` 骨架，不推翻现有设计。后端补齐 `hold` 统一路由、`shake` 上下文化玩法选择与每日状态；前端补调试入口与只读状态；固件补 `tap_count` 检测与 `animation_hint` 消费。当前 demo 固件明确声明 `LED_HARDWARE_AVAILABLE = false`，因此本轮把 LED 视为硬件阻塞项，不把“灯效真实可用”作为 `P1-6` 完成门槛。

**Tech Stack:** Python aiohttp backend, Flutter app, Arduino demo firmware, existing `experience` runtime/store/event stream, existing `m5stack-avatar` 风格脸部显示。

## 0. 审计结论

### 0.1 已有代码、但总 TODO 口径落后的部分

- `scene / persona / runtime experience` 前后端骨架已落地，前端已经可在 `Chat` 和 `Settings` 中切换场景与人格。
- `tap 1 / 2 / 3` 的后端语义已落地，分别对应允许 / 拒绝 / 打断。
- `shake` 已从旧 prompt 变成结构化结果，并会进入 runtime history。
- `device_status`、`device_command_result`、`desktop_mic` 主路线已经在主 demo 固件里，不属于本轮“从零搭基础链路”。

### 0.2 真正还没完成、且会阻塞 P1-6 完成口径的部分

- 后端 `hold` 没有进入统一 physical interaction route/history，只是直接操作了 `desktop_mic bridge`。
- 后端 `shake` 还是按 `scene_mode` 的简化路由，没做到总 TODO 里要求的“待决策优先 / 当天首摇运势 / 其余随机内容”。
- `fortune / decision / random` 结果目前还是固定单条文案，不是最小产品化内容池。
- 前端 `Physical Interaction` 面板仍是只读，没有主动调试入口，也没有“今日首摇是否已使用 / 当前玩法模式”只读状态。
- 固件没有真实 `tap_count` 来源，当前只有 `long_press / long_release` 和 `shake_event`。
- 固件没有消费后端下发的 `animation_hint` 语义，`focus / celebrate / affirm / deny / interrupt` 这些反馈目前不会在屏幕上体现。

### 0.3 本轮不计入 P1-6 完成门槛的项

- `LED_CONTROL` 真正驱动硬件灯效。
- 恢复 `device_mic` 主路线。
- 恢复 App 直接录音。
- 重做聊天页气泡 UI 或大改前端交互结构。

## 1. 收口方案对比

### 方案 A：最小软件收口

- 只补 `hold` 路由、`shake` 规则、固件 `tap_count`、前端调试入口。
- 优点：实现成本最低，能较快把 P1-6 软件主线闭环。
- 风险：`shake` 内容质量和设备反馈层次偏薄，后续还要再补一次。

### 方案 B：产品化收口，推荐

- 在方案 A 的基础上，再补 `shake` 最小内容池、设备侧 `animation_hint` 消费、前端“今日首摇 / 最近模式”状态展示。
- 优点：完成后 `P1-6` 的体验、审计、调试三条线基本闭环，和总 TODO 的产品定义对齐度高。
- 风险：涉及前后端 + 固件三端联动，改动面比方案 A 大，但仍然在当前骨架上收口，不会推翻已有实现。

### 方案 C：把 LED 一并做到“完全体”

- 在方案 B 基础上继续把 LED 实体控制也做掉。
- 优点：物理反馈最完整。
- 风险：当前 demo 固件明确标记 LED 硬件不可用，缺少硬件前提，容易把软件任务拖成硬件排障任务。

**建议采用方案 B。**

- 原因：它是当前唯一能把 `P1-6` 真正收口、又不被硬件阻塞拖住的方案。
- 口径：`P1-6` 完成标准改为“软件可交付闭环完成，LED 作为硬件恢复项继续留在后续硬件任务中”。

## 2. 目标交付口径

- 设备长按说话仍走现有 `desktop_mic` 主链路，但 `hold` 会进入统一 physical interaction history/event。
- 摇一摇按固定优先级自动切到 `decision / fortune / random`：
  - 有待确认上下文：`decision`
  - 当天首次有效摇动：`fortune`
  - 其他有效摇动：`random`
- 设备屏幕能对 `animation_hint` 做出可见反馈，不再只会 `ACTIVE / IDLE` 两态。
- 前端 `Control Center` 能手动模拟 `shake` 与 `tap 1 / 2 / 3`，用于验收与回归，但仍然走真实后端 guard，不伪造成功结果。
- 前端能看到“当前 shake 倾向模式”“今日是否已首摇”“最近一次 shake 模式”“最近物理交互结果/历史”。
- 总 TODO 与实施计划会按真实完成度回填，不再误把未完成项标成已完成。

#### 路径二：待确认动作 + 拍一拍

1. 先制造一个待确认电脑动作。
2. 拍 1 次，预期允许。
3. 再制造一个待确认动作，拍 2 次，预期拒绝。
4. 当前存在可打断流程时拍 3 次，预期打断。

#### 路径三：Control Center 调试按钮

1. 打开 `Control Center`。
2. 分别触发 `模拟摇一摇`、`拍 1 / 2 / 3 次`。
3. 预期面板不本地伪造结果，而是在后端真实返回后刷新 history/last result。

#### 路径四：长按说话主路线

1. 长按设备。
2. 预期仍走 `desktop_mic`。
3. 松开后识别流程继续。
4. 同时 runtime history 新增 `hold` 记录。

## 6. 执行约束

- 当前用户要求：改动前先写计划，测试需等用户确认后再跑。
- 因此执行时先落代码与测试文件，再等用户确认后统一运行验证命令。
- 继续在 `main` 上工作，不使用 worktree。
- 每完成一个独立任务后都要 `git commit`，但必须先向用户汇报进度并等其确认文件内容。

## 7. 完成定义

当以下条件全部满足时，`P1-6` 可标记为“软件侧完成”：

- `hold / tap / shake` 都进入统一 interaction route/history。
- `shake` 具备 `decision / fortune / random` 的固定规则自动切换。
- 固件具备真实 `tap_count` 来源。
- 设备屏幕能消费 `animation_hint`，让不同 physical result 至少有可见表情差异。
- 前端具备调试入口和今日状态只读面。
- 总 TODO 与旧实施计划完成口径已回填。

LED 若仍不可用，不阻塞以上完成定义，但必须在文档中明确标注为硬件阻塞项。
