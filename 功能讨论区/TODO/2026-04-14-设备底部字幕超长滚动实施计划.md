# 设备底部字幕超长滚动实施计划

> 仅覆盖本轮新增规则：设备底部字幕改为“短句静止，超长才滚动”，并保持现有 `status > AI reply` 显示优先级不变。

## 目标

- [x] 底部字幕在内容可完整显示时保持静止，不做无意义运动。
- [x] 底部字幕在内容超出可显示宽度时自动横向滚动，尾部可被看到。
- [x] `statusText > replyText` 的优先级保持不变，滚动逻辑对两者统一生效。
- [x] 现有眨眼、说话、processing 点点动画继续正常工作，不被字幕滚动打断。
- [x] 不修改前后端协议，只在固件显示层完成滚动能力。

## 调研结论

- 当前滚动缺失的根因在 `firmware/arduino/demo/face_display.cpp`：
  - `drawSubtitle()` 只支持固定两行排版，超出直接 `...` 截断。
  - `faceUpdate()` 只驱动眨眼、说话、processing 点点，没有字幕滚动计时器。
- `demo.ino` 仅负责持续调用 `faceUpdate()` 和向 `faceSetReplyText / faceSetStatusText` 写入文本，不需要改协议层。
- 因此本轮最小可行方案是：只在 `face_display` 内补充字幕滚动状态与重绘条件。

## 方案对比

- 方案 A：保留两行排版，超长仍用省略号。
  - 优点：实现最简单。
  - 缺点：用户看不到完整句子，不满足本轮需求。
- 方案 B：统一改成单行横向跑马灯，短句静止，超长才滚动。
  - 优点：实现边界清晰，稳定性高，能完整看到内容。
  - 缺点：显示密度低于两行模式。
- 方案 C：两行分页滚动或逐字滚动。
  - 优点：信息密度更高。
  - 缺点：状态复杂，容易和眨眼/说话动画抢刷新时机。

## 推荐方案

- [x] 采用方案 B。
- [x] 底部字幕统一收敛为单行显示区域。
- [x] 只有当文本总宽度超过可视宽度时才启动滚动。
- [x] 每次字幕内容切换时重置滚动位置，先从起点停留，再开始移动。
- [x] 滚动到末尾后停留一小段时间，再回到起点循环。

## 实施范围

- `firmware/arduino/demo/face_display.h`
- `firmware/arduino/demo/face_display.cpp`

## 落地步骤

- [x] 在 `face_display.cpp` 增加字幕滚动所需的状态变量：
  - 当前滚动像素偏移
  - 上次滚动推进时间
  - 文本切换后的起点停留时间
  - 尾部停留状态
- [x] 将 `drawSubtitle()` 从“两行截断排版”改为“单行窗口裁切 + 按偏移绘制”。
- [x] 在 `faceUpdate()` 中增加字幕滚动推进逻辑，仅在超长文本时触发重绘。
- [x] 在 `faceSetReplyText / faceSetStatusText / faceClearStatusText` 触发字幕切换时重置滚动状态。
- [x] 保持 `status > reply`、英文状态提示、上一句 AI 回复回退逻辑不变。

## 验收口径

- [ ] 短句如 `WiFi connected`、`Processing` 显示静止，不滚动。
- [ ] 超长 AI 回复会自动横向滚动，能看到后半句。
- [ ] status 提示出现时，优先显示 status，并按同样规则静止/滚动。
- [ ] status 清空后，恢复显示上一句 AI 回复，并从起点重新开始。
- [ ] 眨眼、说话、processing 点点动画仍正常，不出现明显闪烁或字幕残影。
