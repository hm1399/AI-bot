## 1. 根因总览：三个 P0 其实是同一个问题

- 当前系统对“事实”存在三套不同口径：
  - 代码声明自己支持哪些 runtime 工具；
  - 脚本用 `pid` 推断服务是否真的启动；
  - 前端同时在 `ConnectionConfigModel` 与 `Settings.server_url/server_port` 中表达服务地址。
- 这三套口径共同导致：**静态配置、历史字段、脚本 heuristics** 代替了 **当前已验证运行事实**。
- 因此本方案的统一原则不是分别打补丁，而是建立三条单一真源：
  - **能力真源：** `ResolvedAgentRuntimeOptions`
  - **就绪真源：** `/api/health` 的 `ready`
  - **地址真源：** `ConnectionConfigModel / AppState.connection`

## 2. 单一真源原则

### 2.1 Runtime 能力真源

- 只有被 bootstrap 实际解析并传进 `AgentLoop` 的能力，才算当前 build 已启用能力。
- 本轮以 backend 输出的 effective runtime 结构为唯一真源：
  - backend / 调试接口直接输出这份真相层；
  - Flutter bootstrap 模型按需解析这份附加字段；
  - `SOUL.md` 在本轮先改成保守描述，去掉默认未启用工具的硬编码宣称，不强行在这一轮做动态 prompt 注入。

### 2.2 服务就绪真源

- 只有 `/api/health` 返回 `ready=true`，才表示后端真正可用于 Flutter 连接、WS 建连和设备配网。
- `pid 已写入`、`进程还活着`、`端口偶尔监听` 都只能算中间态，不能算启动成功。

### 2.3 服务地址真源

- Flutter 当前连接 backend 的唯一真源是 `ConnectionConfigModel / AppState.connection`。
- `Settings` 不再承载“App 连哪个 backend”这一职责；如需展示服务器地址，只能展示从 `state.connection` 派生的**只读诊断信息**。
- `Pairing` 的回连端点应来自“当前已验证成功的连接事实”，而不是另一套静态 `server_url/server_port` 或后端配置猜测。

## 3. 推荐实施顺序

- **先做 Track B：后端启动 ready 闭环**
  - 最隔离、最容易先落地；
  - 能立刻减少后续联调误判。
- **再做 Track A：runtime 能力与权限边界收口**
  - 收紧 backend 真相层；
  - 为前端提示与后续验收提供统一口径。
- **最后做 Track C：前端服务地址真源统一**
  - 基于稳定 backend 与 ready 信号收口前端与设备配网链路。

## 4. 关键设计决策

### 4.1 为什么不继续让 `Settings` 持有 `server_url/server_port`

- 这两个字段不是当前 App live 连接的真源；
- Flutter 已经不再通过 `SettingsUpdate` 回写它们，继续保留只会制造“看得到、改不到、还不一定准”的伪能力；
- 把它们留作只读 API 回包也没有产品价值，不如把同一位置的展示直接改成 `state.connection` 派生诊断，既不改样式，也消除双数据源。

### 4.2 为什么 Pairing 不能继续读后端静态 `server.port`

- 机器人需要的是“当前这次成功连接里，真正可达的回连端点”，不是“后端进程理论监听端口”；
- 一旦 App 连的是非默认端口、代理入口、远程 tunnel 或未来可切换环境，静态 `cfg.server.port` 就会继续漂移；
- 让 bundle 从当前请求派生 `port/secure`，才符合“已验证事实 > 静态猜测”。

### 4.3 为什么 `backend-restart` 要走 `/api/health.ready`

- 这是唯一能跨语言、跨脚本、跨前后端共享的 readiness 合同；
- `pid` 与 `kill -0` 只适合进程管理，不适合产品层“服务已可用”判断；
- 一旦 readiness 合同稳定，后续 Flutter 自检、部署脚本、CI smoke check 都能复用同一标准。

## 5. 风险与回滚策略

- **风险 1：** `request.scheme / request.url.port` 在代理场景下不可靠  
  **策略：** Pairing bundle 先取请求事实，失败时 fallback 到 `cfg.server`，并写日志说明 fallback 原因。
- **风险 2：** `restrict_to_workspace` 一次性打开会改变现有 agent 行为  
  **策略：** 第一版先把配置链路与可观测性接好，默认行为保持兼容；是否真正默认收紧，由后续单独开关决定。
- **风险 3：** `/api/health.ready` 引入后，脚本等待时间增加  
  **策略：** 采用短轮询 + 明确超时（例如 20~30 秒），并输出阶段性日志，避免“卡住无反馈”。
- **风险 4：** 前端同时改 Settings 与 Pairing 容易引起误会  
  **策略：** 保持 UI 结构不变，只改数据源与标签，不新增复杂交互。
