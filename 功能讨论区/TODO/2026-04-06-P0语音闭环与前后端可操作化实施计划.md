## 调研结论

### 后端

- `server/channels/device_channel.py` 已有设备状态机、触摸事件、ASR/TTS 和设备命令发送能力，但当前语音链路默认依赖设备上传 PCM。
- `server/services/app_runtime.py` 已有 `sessions / tasks / events / notifications / reminders / runtime / device / capabilities` 接口和事件广播框架，适合继续承接桌面语音桥接状态与提醒事件。
- `server/services/app_api/resource_service.py` 已有任务、事件、通知、提醒的 JSON CRUD，可在其上补提醒调度与通知落地。
- 当前没有本地桌面麦克风实时入口，也没有提醒真实调度执行器。

### 前端

- `app/lib/providers/app_providers.dart` 统一承接连接、聊天、任务、事件、通知、提醒和设备状态，但目前缺少增删改动作和更细粒度语音可用性判断。
- `app/lib/screens/chat/chat_screen.dart` 只有消息列表和输入框，没有会话切换/创建入口。
- `app/lib/screens/tasks/tasks_screen.dart`、`app/lib/screens/control_center/control_center_screen.dart` 目前基本是只读展示。
- `app/lib/services/chat/voice_capture_service.dart` 仍是 stub，不能继续把按钮当成真实语音入口。

### 本轮正式方案

- 语音接入采用 `P0-3 方案 B`：新增独立的本地语音客户端 WebSocket 入口，不复用 `/ws/device`。
- 当前稳定输出采用 `P0-1 备选方案 B`：设备负责 `listening / transcribing / responding` 状态和文字反馈，暂不在本轮强行接入电脑本地扬声器 TTS。
- Reminder 先落到“真实调度 + App 事件流通知/通知中心落单”，先把主闭环跑通。
