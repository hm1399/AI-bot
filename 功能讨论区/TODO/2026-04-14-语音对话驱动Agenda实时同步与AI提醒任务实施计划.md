# 语音对话驱动 Agenda 实时同步与 AI 提醒任务 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans。步骤使用 checkbox (`- [ ]`) 语法跟踪。

**Goal:** 让用户能通过语音/对话要求 AI 添加行程或提醒；行程实时进入前端 `Agenda` 日历；提醒类需求转成 AI 可执行任务并显示在前端 `Tasks` 页面；到点后设备发出语音提醒；AI 还能读取用户当日/次日规划并回答“今天/明天有什么要做”。

**Architecture:** 继续沿用现有 `Flutter Agenda/Tasks/Chat + WebSocket app events + aiohttp AppRuntimeService + AppResourceService + PlanningTool + ReminderScheduler + DeviceChannel TTS` 主干，不新增新的产品壳层，也不破坏现有 Linear 风格 UI。核心不是重写日历/提醒系统，而是把“自然语言意图 -> 结构化 planning 资源 -> 实时投影 -> 设备到点播报”这条链路收口，并补一层 `planning_surface / owner_kind / delivery_mode` 的产品语义。

**Tech Stack:** Flutter + Riverpod + GoRouter，Python `aiohttp`，`JsonCollectionStore`，`PlanningProjectionService`，`PlanningTool`，`ReminderScheduler`，`DeviceChannel`，WebSocket app events。

## 需求拆解与产品口径

| 用户表达 | 应落资源 | 前端主落点 | 到点行为 |
| --- | --- | --- | --- |
| “帮我把明天下午三点牙医预约加进行程” | `Event` | `Agenda` | 默认不播报；只有用户明确要求提醒时才补 `Reminder` |
| “下午 5 点提醒我要开会” | `Task` + 一个隐藏的 delivery `Reminder` | `Tasks` 显示 AI task；`Agenda` 不把它当成普通行程主项 | 17:00 设备 TTS 播报，并生成 notification |
| “我今天/明天有什么要做” | AI 读取 planning 数据并总结 | `Chat` 回复 | 按当天/次日过滤 `Agenda + Tasks + active reminders`，不靠幻觉回答 |

### 推荐语义

- “行程 / schedule / event / calendar” 归入 `Event`，默认面向 `Agenda`。
- “提醒我 / remind me / 到点叫我” 归入“AI 跟进任务”语义：前台展示一个 `Task`，后台挂一个真正可触发的 `Reminder` 负责播报。
- `Reminder` 在这个需求里更像“交付机制”，不应该把所有提醒都直接污染成 `Agenda` 行程块。
- AI 回答“今天/明天有什么要做”时，应该分组回答：
  - `Agenda` 行程
  - `Tasks` 里的 AI 跟进任务
  - 仍处于激活态的提醒

## 调研结论摘要

### 前端现状

- `app/lib/screens/agenda/agenda_screen.dart` 已经存在，并且会主动拉取 `tasks / events / reminders / planning workbench`。
- `app/lib/providers/app_providers.dart` 已处理 `task.* / event.* / reminder.* / planning.changed / session.message.*` 等 WebSocket 事件；只要后端广播正确，前端本身已经具备实时刷新基础。
- `app/lib/screens/tasks/tasks_screen.dart` 已经是 planning workbench，天然更适合承接“AI 要做的 task 列表”。
- `app/lib/widgets/chat/message_bubble.dart` + `app/lib/widgets/chat/chat_structured_summary.dart` 已经能展示 planning tool 的结构化摘要，说明聊天页并不缺“结构化结果回显”的 UI 容器。
- 当前缺口不是“没有页面”，而是没有稳定的产品语义去区分：
  - 哪些 item 应进入 `Agenda`
  - 哪些 item 只应进入 `Tasks`
  - 哪些 reminder 只负责 delivery，不应被当成日历主项

### 后端现状

- `server/nanobot/agent/tools/planning.py` 已支持 `create_task / create_event / create_reminder / complete_task / snooze_reminder / list_today`。
- `server/services/app_runtime.py` 已提供 `tasks / events / reminders / notifications / planning overview / timeline / conflicts` 接口，并通过 `/ws/app/v1/events` 推送实时事件。
- `server/services/reminder_scheduler.py` 已具备 `next_trigger_at` 计算、到点创建 notification、广播 `reminder.triggered` 的核心骨架。
- `server/channels/device_channel.py` 已具备设备 TTS 发送能力，`source == "voice"` 时会走语音回放。
- 当前最大后端缺口有三点：
  - `on_reminder_triggered()` 只会广播事件，不会直接驱动设备播报。
  - 资源 schema 还没有 `planning_surface / owner_kind / delivery_mode` 这类产品级元数据。
  - AI 虽然有 planning tool，但没有被明确约束“何时建 event、何时建 task+delivery reminder、何时读取当天/明天计划”。

## 关键设计决策

### 1. 引入最小产品元数据，不重做存储层

- 在现有 `Task / Event / Reminder / Notification` 基础上扩展可选 planning 元数据：
  - `planning_surface`: `agenda | tasks | hidden`
  - `owner_kind`: `user | assistant`
  - `delivery_mode`: `none | device_voice | device_voice_and_notification`
- 旧数据保持兼容：
  - 未带 `planning_surface` 的历史 `Event` 视为 `agenda`
  - 未带 `planning_surface` 的历史 `Task` 视为 `tasks`
  - 未带 `planning_surface` 的历史 `Reminder` 继续按现有逻辑显示，但新建 AI delivery reminder 一律写 `hidden`

### 2. “提醒我”不直接等价于 `Agenda` 行程

- 对“下午 5 点提醒我要开会”这类话术，推荐写入一个 bundle：
  - `Task`: 标题如“开会提醒跟进”，`planning_surface = tasks`，`owner_kind = assistant`
  - `Reminder`: 真实到点触发器，`planning_surface = hidden`，`delivery_mode = device_voice_and_notification`
- 这样可以同时满足：
  - `Tasks` 页面看得到 AI 要做的事
  - `Agenda` 页面不被纯提醒型 delivery item 污染成行程
  - 到点仍能依靠现有 reminder scheduler 驱动设备播报

### 3. AI 日程问答必须复用同一套 day projection

- 不允许 `Chat` 自己凭会话上下文猜“今天/明天有什么要做”。
- `PlanningTool` 的日查询能力要和 `PlanningProjectionService` 使用同一套过滤口径：
  - 同时识别 `agenda` 行程、`tasks` 跟进任务、激活提醒
  - 支持显式日期，不只支持“today”
- 推荐保留 `list_today` 动作以兼容现有工具，但把 `date` 参数的文档、测试和 prompt 语义补齐。

### 4. 设备播报属于后端调度责任，不依赖前端在线

- 17:00 的提醒必须由后端调度直接触发 `DeviceChannel`。
- 前端是否开着 `Agenda` 或 `Tasks` 页面，不应影响到点播报。
- 当设备离线时，最低保底行为是：
  - reminder / notification 状态仍更新
  - `Tasks` 页面仍可见未完成 AI task
  - 不把失败伪装成已播报成功

## 风险与门禁

- 时间口径必须统一保存 ISO datetime，前后端都按本地时区解释“今天/明天”。
- `Task + hidden Reminder` bundle 需要幂等保护，至少依赖 `bundle_id + source_message_id` 防止重复建单。
- 设备播报不能依赖前端在线；如果后端只广播 websocket 事件而不直接调 `DeviceChannel`，这条需求就不算完成。
- 不允许为了达成功能而破坏现有 `Agenda` / `Tasks` 的视觉结构；本轮只加语义过滤和标记，不换 UI 风格。
- 不引入原生日历、系统通知、外部日历同步；本轮先闭环现有本地 planning 体系。

## 建议执行顺序

1. Worker A 先冻结 schema + projection 过滤语义。
2. Worker B 并行补 due reminder -> device speech。
3. Worker C 在 A 的契约之上补 agent tool / prompt。
4. Worker D 最后接前端模型与页面过滤，避免在字段未定前反复返工。
5. 全部 subagent 回来后，主线程先做文件级复核，再向用户汇报，等确认后再跑测试与 git。

## 验收标准

- [ ] 用户说“把明天 3 点的牙医预约加进行程”，对应 `Agenda` 当天格子和日程列表实时出现该 event。
- [ ] 用户说“下午 5 点提醒我要开会”，对应 `Tasks` 页面实时出现一条 AI task，而不是把它主展示成 agenda 行程。
- [ ] 17:00 到点时，设备发出语音提示；如果设备离线，也至少有 reminder / notification 状态可追踪。
- [ ] 用户问“我今天/明天有什么要做”，AI 会读取当前 planning 数据并给出分组回答，而不是凭上下文猜。
- [ ] 整个链路保持现有前端 UI/UX 风格，不破坏现有 `Agenda`、`Tasks`、`Chat` 页面结构。
