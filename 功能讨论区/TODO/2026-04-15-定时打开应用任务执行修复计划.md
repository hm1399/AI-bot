# 定时电脑动作任务执行修复计划

> 仅覆盖本轮新增问题：修复“你一分钟后帮我打开微信 / 打开文件 / 打开网页”这类未来动作请求被落成纯提醒，导致到点只发设备语音/通知、不真正执行电脑动作的问题；保持现有 Tasks 页面视觉结构，不做额外 UI 改版。

## 目标

- [x] 未来时间的“打开应用 / 文件 / 网页”请求要能以结构化方式持久化为“提醒 + 定时电脑动作”，而不是只保留提醒文案。
- [x] reminder 到点后，除现有通知和设备语音外，还要按策略实际触发 `computer_control.open_app / open_path / open_url`。
- [x] Tasks 页面保留现有卡片样式，仅补充最小可见信息，让用户能看出该 AI task 绑定了什么待执行动作。

## 调研结论

- 当前这类请求在 agent 生成阶段只调用了 `planning.create_task` 和 `planning.create_reminder`，没有任何 `computer_control` 的未来执行模型落库。
- `ReminderScheduler` 到点后只会创建 `notification` 并回调 `AppRuntime.on_reminder_triggered()`。
- `AppRuntime.on_reminder_triggered()` 当前只广播 `reminder/notification` 事件，并按 `delivery_mode` 决定是否走设备语音，不会执行电脑动作。
- 前端 `TaskModel` / `ReminderModel` 只解析现有 planning 元数据，所以任务卡片只能展示 reminder 时间，无法展示将来要执行的 app 动作。

## 推荐方案

- [x] 为 task/reminder 增加一份结构化的“定时电脑动作”元数据，并同时透传到 notification metadata，避免到点后靠标题猜动作。
- [x] 扩展 planning tool，让 agent 在创建“未来打开应用 / 文件 / 网页”类请求时，把 `open_app / open_path / open_url` 和目标一起写进 reminder。
- [x] 在 `AppRuntime.on_reminder_triggered()` 增加最小执行分支：解析动作元数据，调用 `computer_control_service.request_action()`，并把结果写回通知事件链路。
- [x] 前端模型和 Tasks 卡片只补一个动作标签，例如 `Action Open WeChat` / `Action Open path` / `Action Open url`，不改变当前布局结构和交互。
