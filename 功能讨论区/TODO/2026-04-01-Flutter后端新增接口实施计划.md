# Flutter Backend API Contract Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在保持现有 `/api/app/v1` 与 `/ws/app/v1/events` 协议不变的前提下，补齐 Flutter 前端契约里尚未落地的新增接口、错误码和事件广播能力。

**Architecture:** 继续保留 `server/services/app_runtime.py` 作为 app-v1 的统一入口，但不要把新的资源 CRUD 全部继续堆进现有大文件。新增一个轻量的 `server/services/app_api/` 子目录，负责本地 JSON 持久化、字段规范化、配置映射和资源级事件构造；`AppRuntimeService` 只做鉴权、HTTP glue code、调用 service/store、复用现有 `_broadcast_event()`。新增资源先统一落盘到 `server/workspace/runtime/`，并改用 `nanobot.utils.atomic_write.atomic_write_text` 做原子写入。

**Tech Stack:** Python 3, aiohttp, MessageBus, SessionManager, LiteLLMProvider, local JSON persistence, `nanobot.utils.atomic_write`, `unittest.IsolatedAsyncioTestCase`

## 1. 当前代码基线

- 新增接口主入口现在集中在 `server/services/app_runtime.py`，现有能力已经覆盖 bootstrap、sessions、messages、runtime、device、capabilities、events websocket。
- `server/tests/test_app_runtime.py` 已经有 App API 的单元测试基线，适合作为新增接口的主回归入口。
- `server/config.py` 与 `server/config.yaml` 目前只覆盖基础 `nanobot/asr/tts/device/app/weather/whatsapp` 配置，没有面向 Flutter 设置页的读写映射。
- `server/channels/device_channel.py` 当前只暴露设备状态快照和播报链路，没有 App 层的统一命令入口。
- `server/services/app_runtime.py` 现有摘要文件写入是普通 `json.dump()`，新增资源不应该复制这个模式，应统一切到原子写。

## 2. 默认实现假设

- Settings 默认不直接覆写受版本控制的 `server/config.yaml`。建议新增 `server/workspace/runtime/app_settings.json` 保存可变覆盖项，读取接口返回“配置基线 + runtime overlay”的合并结果。
- `llm_api_key` 不回显。建议单独保存到 `server/workspace/runtime/app_secrets.json` 或等价私有文件，只返回 `llm_api_key_configured: true|false`。
- Notifications 的 P0 先做“读取、标记已读、全部已读、删除单条、清空全部、广播变更”；自动根据 task due / reminder fire 生成通知作为下一阶段，不塞进这轮接口补齐。
- Reminders 的 P0 先做提醒定义的 CRUD 与事件广播；真正的定时触发如果接入 `nanobot.cron.service.CronService`，再单独补依赖、调度和运行态联动。
- Device Commands 统一落到现有 `DeviceChannel`，不新增第二套设备协议。
