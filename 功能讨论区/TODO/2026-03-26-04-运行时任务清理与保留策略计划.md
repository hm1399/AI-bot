# AI-Bot 运行时任务清理与保留策略计划

> 创建日期：2026-03-26  
> 目标问题：当前运行时任务只增不减，长时间运行后 `_tasks` 和 `_task_order` 会持续膨胀  
> 本文性质：实施计划，不包含大量代码

---

## 1. 问题定义

当前后端已经有“当前任务”和“任务队列”概念，但任务结束后只是改状态，没有形成明确的回收机制。

这会带来几个长期问题：

- 后端运行越久，内存里的历史任务越多。
- `current_task` 和 `task_queue` 虽然还能算出来，但扫描成本会持续变大。
- 后面如果再叠加 Todo、日历、更多通道，这个 runtime 容器会越来越重。
- 真正的问题不是现在能不能跑，而是“跑几天后会不会开始变慢、变乱”。

---

## 2. 目标

建立一套清晰的任务生命周期与保留策略，让运行时任务：

- 在运行中可观察
- 在结束后短期可回看
- 在过期后可清理

目标不是保留全部历史，而是保留**当前调试需要的、近期可追踪的、不会把进程拖胖的**那部分任务。

---

## 3. 范围

本轮覆盖：

- App Runtime 中的任务注册表
- `current_task`
- `task_queue`
- 已完成 / 失败 / 取消任务的保留与清理

本轮先不覆盖：

- 完整任务历史查询 API
- 长期审计系统
- 外部数据库持久化

---

## 4. 设计原则

1. 运行中任务绝不能被误清理。
2. 当前前台展示只需要：
   - `current_task`
   - `task_queue`
   - 少量近期已结束任务
3. 清理策略要可配置，但默认值必须保守可用。
4. 任务清理不能影响会话消息历史。
5. 任务清理逻辑要和事件流语义一致，不能把前端状态搞断。

---

## 5. 实施计划

### Phase 1：明确任务生命周期

建议先把任务状态分成几类：

- `queued`
- `running`
- `completed`
- `failed`
- `cancelled`

然后明确每类状态对应的处理方式：

- `queued` / `running`：
  - 必须常驻内存
  - 参与 `runtime.state`
  - 参与事件流
- `completed` / `failed` / `cancelled`：
  - 可以短期保留
  - 不再参与队列展示
  - 到期后允许清理

### Phase 2：定义保留策略

建议第一版不要太复杂，先定 3 个维度：

- 最多保留最近 N 条结束任务
- 最长保留最近 T 小时结束任务
- 内存达到上限后优先清理最旧的结束任务

建议默认先按“数量上限 + 时间上限”双保险。

### Phase 3：引入统一清理入口

不要在多个位置零散 `pop`。

建议做一个统一任务清理入口，负责：

- 任务结束后尝试清理一次
- 周期性后台清理一次
- 服务启动时做一次轻量修整

这样后面如果要改保留策略，只需要改一个地方。

### Phase 4：补一个“轻量历史视图”思路

虽然这轮不做完整历史 API，但建议提前留好结构。

可以先把结束任务区分成两层：

- 运行态主结构：
  - 只保留活跃任务 + 少量最近结束任务
- 可选历史记录：
  - 后续如有需要，再落到独立历史文件或独立接口

第一版先不做复杂历史，只要别把所有任务永远堆在主运行态里。

### Phase 5：补观测字段

为了后面排查问题，建议补一些内部统计：

- 当前活跃任务数
- 当前已结束但仍保留的任务数
- 最近一次清理时间
- 最近一次清理掉多少任务

这些信息不一定马上对 Flutter 暴露，但对服务端排查很有用。

### Phase 6：补测试

至少补以下测试：

- 活跃任务不会被误删
- 已完成任务达到上限后会清掉最旧的
- 时间过期任务会被清理
- 清理后 `current_task` 与 `task_queue` 仍正确
- 清理不会影响消息历史查询

---

## 6. 推荐实现顺序

1. 先定义生命周期和保留规则
2. 再做统一清理入口
3. 再做结束任务清理触发点
4. 再补统计信息
5. 最后补测试

---

## 7. 风险点

- 如果清理写得太激进，可能把前端还在看的任务状态提前清掉。
- 如果只保留数量，不保留时间，长会话场景仍可能拖很久。
- 如果把历史和运行态混在一起，后面再拆会更麻烦。
- 如果没有统计信息，出现“任务怎么突然没了”会很难定位。

---

## 8. 验收标准

- 服务连续运行较长时间后，任务内存不会无限增长。
- `runtime.state` 仍能稳定返回当前任务和队列。
- 结束任务会按规则清理，不影响活跃任务。
- 长时间压测或多轮消息后，性能没有明显恶化。
- 相关清理逻辑有自动化测试覆盖。

---

## 9. 完成后的直接收益

这项做完后，你后面再做：

- Flutter 首页全局状态
- 多会话并发
- 长时间本地驻留

都会更稳，不会因为“任务越积越多”把整个本地中枢拖慢。
