# AI-Bot 五个稳定性计划 Subagents 执行编排

> 创建日期：2026-03-26
> 目标：基于 5 份稳定性计划与当前代码状态，给出可执行的 `gpt-5.4 / xhigh` 多 subagents 拆分、依赖顺序与分工边界
> 结论先行：不能按“5 个计划 = 5 个实现 subagents 同时开工”处理，必须按共享热区分波次推进

---

## 1. 当前代码面的冲突结论

- `server/services/app_runtime.py` 是最高冲突热区。
- `server/nanobot/bus/queue.py`、`server/nanobot/bus/events.py`、`server/bootstrap.py`、`server/config.py` 是消息总线与配置热区。
- `server/nanobot/providers/base.py`、`server/nanobot/providers/litellm_provider.py` 是模型超时计划最独立的一组文件。
- `server/nanobot/session/manager.py` 与共享原子写工具层，是原子写计划最独立的一组文件。
- 当前 worktree 已有大量未提交改动，后续实现阶段不能让多个 subagents 同时写同一批热区文件。

## 2. 五个计划的主改面

### 计划 01：模型调用超时与卡死保护

- 主改面：Provider 超时契约、`LiteLLMProvider`、`AgentLoop` 失败收口、`AppRuntime` 失败事件语义、配置与测试。
- 核心风险：现在 provider 异常会被压成普通文本，容易被当成 `completed` 而不是 `failed/timeout`。

### 计划 02：App 消息幂等与重试安全

- 主改面：`AppRuntimeService.handle_post_message()`、accepted envelope 复用、幂等索引持久化、事件去重、测试。
- 核心风险：当前 `client_message_id` 只是透传，没有查重、冲突拒绝和重启恢复。

### 计划 03：消息总线队列上限与背压

- 主改面：`MessageBus` 容量与控制消息优先级、observer 时机、App busy 错误、device 输入拒绝、outbound owner、测试。
- 核心风险：现在 observer 先于成功入队执行，直接加 `maxsize` 会制造假 `queued` 状态。

### 计划 04：运行时任务清理与保留策略

- 主改面：`AppRuntimeService` 内存任务表 `_tasks/_task_order`、统一 prune 入口、保留配置、统计、测试。
- 核心风险：只删 `_tasks` 不压缩 `_task_order`，状态查询和扫描成本不会真正下降。

### 计划 05：会话与摘要原子写

- 主改面：共享原子写工具、`SessionManager.save()`、`AppRuntime` 的 Todo/Calendar 摘要写入、恢复日志、测试。
- 核心风险：当前会话和摘要文件都是直接覆盖写，异常退出时存在写坏风险。

## 3. 推荐总顺序

1. 先完成计划 05 的“共享原子写工具 + SessionManager”部分。
2. 并行完成计划 01 的“Provider 超时契约 + LiteLLMProvider + 配置”部分。
3. 先冻结计划 03 的总线 contract，再做其它所有依赖 `publish_inbound()` 语义的工作。
4. 再完成计划 05 的 `AppRuntime` 摘要原子写收口。
5. 再完成计划 01 的 `AgentLoop + AppRuntime` 超时收口。
6. 再完成计划 03 的 App/device/outbound 背压传播。
7. 再完成计划 02 的 App 幂等与 accepted 结果复用。
8. 最后完成计划 04 的任务清理与保留策略。

## 4. 分波次编排

### Wave 1：唯一可安全并行的两组

- Agent W1-A
  模型：`gpt-5.4`
  推理：`xhigh`
  负责：计划 05A
  范围：抽共享原子写工具，接入 `server/nanobot/session/manager.py`，补工具层与 SessionManager 测试。
  主要文件：`server/nanobot/session/manager.py`、新增共享原子写工具文件、相关测试。
  不要碰：`server/services/app_runtime.py`、`server/nanobot/bus/queue.py`、`server/config.py`。
  完成标准：会话保存不再直接覆盖正式文件，失败时保旧文件，测试覆盖到位，单独提交一次 commit。

- Agent W1-B
  模型：`gpt-5.4`
  推理：`xhigh`
  负责：计划 01A
  范围：定义 provider timeout/error contract，接入 `LiteLLMProvider`、配置读取与校验、provider/config 测试。
  主要文件：`server/nanobot/providers/base.py`、`server/nanobot/providers/litellm_provider.py`、`server/config.py`、`server/config.yaml`、`server/bootstrap.py`、相关测试。
  不要碰：`server/services/app_runtime.py`、`server/nanobot/bus/queue.py`、`server/nanobot/session/manager.py`。
  完成标准：单次模型请求超时被结构化识别，不再压成普通 assistant 文本，测试覆盖到位，单独提交一次 commit。

### Wave 2：先冻结总线 contract

- Agent W2-A
  模型：`gpt-5.4`
  推理：`xhigh`
  负责：计划 03A
  范围：建立 `MessageBus` 上限、普通/控制消息分类、observer 与成功入队顺序、基础指标与配置注入。
  主要文件：`server/nanobot/bus/queue.py`、`server/nanobot/bus/events.py`、`server/config.py`、`server/config.yaml`、`server/bootstrap.py`、bus 测试。
  不要碰：`server/services/app_runtime.py` 的业务幂等/清理逻辑。
  完成标准：bus 有明确接受/拒绝 contract，控制消息保留路径定死，observer 不再制造 phantom task，单独提交一次 commit。

### Wave 3：`app_runtime` 热区串行推进

- Agent W3-A
  模型：`gpt-5.4`
  推理：`xhigh`
  负责：计划 05B
  范围：把 Todo/Calendar 摘要写入切到共享原子写工具，补恢复日志与相关测试。
  主要文件：`server/services/app_runtime.py`、共享原子写工具、相关测试。
  前置依赖：W1-A 完成。

- Agent W3-B
  模型：`gpt-5.4`
  推理：`xhigh`
  负责：计划 01B
  范围：`AgentLoop + AppRuntime` 超时收口，保证 `timeout -> failed`，避免 `completed/failed` 双发。
  主要文件：`server/nanobot/agent/loop.py`、`server/services/app_runtime.py`、相关测试。
  前置依赖：W1-B 完成，最好在 W2-A 之后执行。

- Agent W3-C
  模型：`gpt-5.4`
  推理：`xhigh`
  负责：计划 03B
  范围：把背压从总线 contract 传到 App/device/outbound，统一 busy 错误、控制消息保底与 outbound owner。
  主要文件：`server/services/app_runtime.py`、`server/channels/device_channel.py`、`server/services/outbound_router.py`、`server/nanobot/agent/loop.py`、相关测试。
  前置依赖：W2-A 完成。

- Agent W3-D
  模型：`gpt-5.4`
  推理：`xhigh`
  负责：计划 02
  范围：App 幂等键、accepted envelope 复用、冲突拒绝、索引持久化、事件去重与测试。
  主要文件：`server/services/app_runtime.py`、`server/nanobot/bus/queue.py`、`server/nanobot/agent/loop.py`、`server/nanobot/session/manager.py` 或共享原子写工具、相关测试。
  前置依赖：W1-A、W2-A、W3-B、W3-C 完成。

- Agent W3-E
  模型：`gpt-5.4`
  推理：`xhigh`
  负责：计划 04
  范围：任务 ended 状态、统一 prune 入口、保留策略、统计、测试。
  主要文件：`server/services/app_runtime.py`、`server/config.py`、`server/config.yaml`、`server/bootstrap.py`、相关测试。
  前置依赖：W3-B、W3-C、W3-D 完成。

## 5. 明确禁止并行的组合

- 计划 02 与计划 04 不能并行实现。
- 计划 02 与计划 03 不能并行实现。
- 计划 01 的 `AppRuntime` 收口部分与计划 04 不能并行实现。
- 计划 05 的 `AppRuntime` 摘要原子写部分，与任何正在修改 `server/services/app_runtime.py` 的任务都不能并行实现。
- 任何同时修改 `server/config.py`、`server/config.yaml`、`server/bootstrap.py` 的任务，必须串行。

## 6. 每个 implementation subagent 的统一要求

- 模型固定为 `gpt-5.4`。
- 推理强度固定为 `xhigh`。
- 每个 subagent 只负责自己的写入边界，不得顺手改其它计划内容。
- 每完成一个子任务必须运行对应测试，并单独 `git commit`。
- 如发现当前未提交改动与自己任务直接冲突，必须停下并回报，不得自行覆盖。

## 7. 建议的启动方式

- 由于当前主工作区已经是 dirty worktree，真正启动 implementation subagents 前，优先使用独立 worktree 或严格的分波次串行写入。
- 如果直接在当前工作区开多个实现 subagents，最容易冲突的文件是：
- `server/services/app_runtime.py`
- `server/nanobot/bus/queue.py`
- `server/config.py`
- `server/config.yaml`
- `server/bootstrap.py`

## 8. 本次拆分后的直接结论

- 真正可以先并行开的，只有 `计划 05A` 和 `计划 01A`。
- `计划 03` 必须先冻结总线 contract，否则后面的幂等、任务队列、busy 错误都会返工。
- `计划 02` 和 `计划 04` 都是 `app_runtime` 重度任务，必须排在后面串行收口。
- 如果下一步要开始真正派 implementation subagents，建议从 `W1-A` 和 `W1-B` 开始。
