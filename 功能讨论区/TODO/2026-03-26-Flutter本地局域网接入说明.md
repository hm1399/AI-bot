# AI-Bot Flutter 本地局域网接入说明

> 创建日期：2026-03-26  
> 适用范围：Flutter App 接入本地局域网版 AI-Bot 后端  
> 对应后端状态：截至 2026-03-26，Phase C 前三段已落地  
> 当前目标：把“接口怎么调、事件怎么接、断线后怎么恢复”写清楚

---

## 1. 先说结论

Flutter 端不要把这个后端当成“纯聊天接口”来接，而要当成两个通道一起接：

- **REST**：拿快照、历史、控制动作
- **WebSocket**：拿实时变化

推荐理解方式很简单：

- `GET /api/app/v1/bootstrap`：启动时先拿一张“全局快照”
- `GET /api/app/v1/sessions/.../messages`：进入会话页时再拿聊天历史
- `POST /api/app/v1/sessions/.../messages`：发消息
- `ws/app/v1/events`：看“正在思考 / 回复完成 / 设备状态变化 / Todo 摘要变化”

也就是说：

> Flutter 端不要自己猜任务状态，不要靠轮询硬刷页面，而是“启动先拿快照，之后靠事件流补实时变化”。

---

## 2. Flutter 端最少要保存什么

建议 Flutter 本地至少保存这几项：

- `server_base_url`
  - 例如：`http://192.168.31.20:8765`
  - 如果 App 就运行在电脑本机，可用 `http://127.0.0.1:8765`
- `app_auth_token`
  - 对应后端的 `APP_AUTH_TOKEN`
- `last_event_id`
  - 每次收到 WebSocket 事件后都更新
  - 用于断线重连后续接事件
- `last_open_session_id`
  - 方便重启后回到上次会话

建议不要单独保存太多“推导状态”，尽量保存：

- 最后一次 `bootstrap` 的快照
- 当前会话的消息列表缓存
- `last_event_id`

其余状态尽量由接口和事件重新驱动。

---

## 3. 推荐的 Flutter 侧职责拆分

为了后面不乱，建议 Flutter 端至少拆成这几层：

### 3.1 配置层

- 保存服务器地址
- 保存 token
- 保存 `last_event_id`

### 3.2 HTTP Client

职责：

- 统一加 `Authorization: Bearer <APP_AUTH_TOKEN>`
- 统一解析 `ok / data / error`
- 统一做超时、错误码处理

### 3.3 Realtime Client

职责：

- 连接 `ws/app/v1/events`
- 收到每个事件后更新 `last_event_id`
- 断线后自动重连
- 重连时把 `last_event_id` 带回去

### 3.4 本地状态层

建议至少维护 4 组状态：

- `sessions`
- `sessionMessages[sessionId]`
- `runtimeState`
- `deviceState`

Todo / 日历摘要可以先挂在 `runtimeState` 里，不用单独做复杂仓库。

---

## 4. 启动时的标准流程

推荐 Flutter App 每次启动都走这条固定流程：

1. 读取本地保存的 `server_base_url`
2. 读取 `app_auth_token`
3. 调用 `GET /api/app/v1/bootstrap`
4. 用返回结果渲染首页
5. 建立 `WebSocket /ws/app/v1/events`
6. 连接成功后先接收 `system.hello`
7. 后续所有实时变化都从 WebSocket 更新

如果你把启动流程做成别的样子，后面会越来越难维护。

## 5. 服务地址怎么拼

假设本地服务地址是：

- `http://192.168.31.20:8765`

那么：

- HTTP 基础地址：`http://192.168.31.20:8765`
- WebSocket 地址：`ws://192.168.31.20:8765/ws/app/v1/events`

如果是 HTTPS / WSS 以后再说，当前本地局域网版先按 `http/ws` 走。

## 6. 鉴权怎么传

当前后端默认支持：

- Header：`Authorization: Bearer <APP_AUTH_TOKEN>`

建议 Flutter 端统一都走这个 Header。

如果某些 WebSocket 客户端环境不方便加 Header，再考虑 Query 兜底，但默认不要这么做。

## 7. 启动快照接口怎么用

### 接口

- `GET /api/app/v1/bootstrap`

### 作用

启动时一次性拿到：

- 服务版本
- 当前能力开关
- 当前运行态
- 默认会话列表
- 实时事件地址
- 事件重连参数说明

### Flutter 端怎么用

收到 `bootstrap` 后：

- 用 `data.sessions` 渲染首页会话列表
- 用 `data.runtime` 渲染全局状态卡片
- 用 `data.capabilities` 控制某些页面是否显示
- 用 `data.event_stream.path` 和 `resume` 信息准备 WebSocket

### 重点

`bootstrap` 不是只在首次安装时调用。

下面这些场景都要重新调：

- App 冷启动
- WebSocket 重连失败，且服务端明确要求“重新拉快照”
- 你怀疑本地状态和服务端不同步时

## 8. 会话列表怎么用

### 接口

- `GET /api/app/v1/sessions`
- `POST /api/app/v1/sessions`
- `GET /api/app/v1/sessions/{session_id}`

### 当前语义

这里只返回 `app:*` 的会话，不返回设备和 WhatsApp 的聊天记录。

### 推荐用法

首页：

- 启动先用 `bootstrap.sessions`
- 用户下拉刷新时再调 `GET /sessions`

新建对话：

- 调 `POST /sessions`
- 后端会返回新会话对象
- Flutter 直接插入本地会话列表即可

单会话详情：

- 一般不需要频繁调 `GET /sessions/{session_id}`
- 主要用于你只想补一条会话元信息时

### 默认会话

后端会自动保证存在：

- `app:main`

所以 Flutter 端可以默认把它当作“主对话”。

---

## 9. 消息历史怎么用

### 接口

- `GET /api/app/v1/sessions/{session_id}/messages`

### 当前支持参数

- `limit`
- `before`
- `after`

### 最重要的规则

#### 9.1 首次进入会话页

直接调：

```text
GET /api/app/v1/sessions/app:main/messages?limit=50
```

这会返回**最新的 50 条**。

#### 9.2 向上翻历史

如果当前列表最早一条消息是 `msg_100`，就调：

```text
GET /api/app/v1/sessions/app:main/messages?limit=50&before=msg_100
```

意思是：

- 取 `msg_100` 之前更早的消息

#### 9.3 向下补新消息

如果你只想拿某条消息之后的新数据，就调：

```text
GET /api/app/v1/sessions/app:main/messages?limit=50&after=msg_150
```

意思是：

- 取 `msg_150` 之后的更新

### 返回结构怎么理解

返回的重点是：

- `items`
- `page_info`

其中 `page_info` 里最有用的是：

- `has_more_before`
- `has_more_after`
- `next_before`
- `next_after`

Flutter 端可以这样用：

- 上拉加载更多历史时，用 `next_before`
- 需要补更晚的数据时，用 `next_after`

### 去重规则

消息主键用：

- `message_id`

不管是 REST 拉下来的，还是事件流收到的，都按 `message_id` 合并，不要用“文本内容”做去重。

---

## 10. 发消息怎么用

### 接口

- `POST /api/app/v1/sessions/{session_id}/messages`

### 请求体

至少传：

```json
{
  "content": "今天还有什么安排？",
  "client_message_id": "flutter_local_001"
}
```

### 返回后怎么处理

后端会立刻返回：

- `accepted_message`
- `task_id`
- `queued`

Flutter 收到后要立刻做两件事：

1. 先把 `accepted_message` 插入本地消息列表
2. 记住这个 `task_id`

### 重点：不要等 WebSocket 完成后再把用户消息显示出来

正确做法是：

- `POST` 成功后马上显示用户消息
- 后续 AI 处理中间态和最终回复再靠事件流补

### 重点：会收到重复的“用户消息创建事件”

因为当前后端在消息入队时也会广播：

- `session.message.created`

所以 Flutter 端如果已经根据 `POST` 返回插入过这条用户消息，后面再收到同一个 `message_id` 的 `session.message.created`，应该：

- 合并
- 忽略重复

不要再插第二次。

## 11. 全局运行态怎么用

### 接口

- `GET /api/app/v1/runtime/state`

### 它解决什么问题

这个接口不是聊天历史，而是“整个本地中枢现在正在干什么”。

主要看这些字段：

- `current_task`
- `task_queue`
- `device`
- `todo_summary`
- `calendar_summary`

### Flutter 端推荐用法

首页顶部、状态栏、设备页、Todo 卡片、日历卡片都应该主要依赖这里。

不要从聊天消息里反推这些状态。

---

## 12. 停止任务怎么用

### 接口

- `POST /api/app/v1/runtime/stop`

### 两种用法

#### 12.1 明确停止某个任务

```json
{
  "task_id": "task_xxx"
}
```

#### 12.2 不传 task_id

后端会尝试停止当前前台任务。

### Flutter 端建议

如果页面里已经拿到了明确的 `task_id`，就尽量传。

这样更稳，不容易误停别的任务。

---

## 13. Todo / 日历摘要现在怎么接

### 当前定位

这两块现在还是**占位摘要层**，不是完整 Todo / 日历系统。

也就是说：

- 可以展示摘要
- 可以更新摘要
- 可以收到摘要变化事件
- 但还没有完整的任务列表 / 日历 CRUD

### 接口

- `GET /api/app/v1/runtime/todo-summary`
- `POST /api/app/v1/runtime/todo-summary`
- `GET /api/app/v1/runtime/calendar-summary`
- `POST /api/app/v1/runtime/calendar-summary`

### Flutter 端当前建议

普通正式页面先以**只读展示**为主：

- 首页卡片
- 状态面板
- “今天还有什么事”之类概览

这两个 `POST` 目前更适合：

- 调试页
- 本地管理页
- 后面真正 Todo / 日历模块接上前的占位更新

### 字段含义

#### Todo 摘要

- `enabled`
- `pending_count`
- `overdue_count`
- `next_due_at`

#### 日历摘要

- `enabled`
- `today_count`
- `next_event_at`
- `next_event_title`

## 14. 设备页怎么接

### 接口

- `GET /api/app/v1/device`
- `POST /api/app/v1/device/speak`

### 推荐用法

设备状态页：

- 初次进入先调 `GET /device`
- 之后主要靠事件流更新

设备播报：

- 调 `POST /device/speak`
- 如果返回 `DEVICE_OFFLINE`，就提示设备未连接

### 重点

设备当前状态变化，应该主要依赖这些事件：

- `device.connection.changed`
- `device.state.changed`
- `device.status.updated`

不要每隔几秒轮询一次 `GET /device`。

## 15. 能力开关怎么用

### 接口

- `GET /api/app/v1/capabilities`

### 当前建议

Flutter 端可以用它做功能开关：

- `device_control`
- `voice_pipeline`
- `whatsapp_bridge`
- `todo_summary`
- `calendar_summary`
- `app_events`
- `event_replay`
- `app_auth_enabled`

例如：

- 如果 `todo_summary=false`，首页就不要显示 Todo 卡片
- 如果 `device_control=false`，就隐藏“设备播报”按钮

## 16. WebSocket 事件流到底怎么接

### 连接地址

- `ws://<host>:8765/ws/app/v1/events`

### 推荐连接方式

带上：

- `Authorization: Bearer <APP_AUTH_TOKEN>`

如果是断线重连，再带：

- `last_event_id`

示意：

```text
ws://192.168.31.20:8765/ws/app/v1/events?last_event_id=evt_xxx
```

### 连接成功后第一条一定要看 `system.hello`

不要连上就忽略第一条。

因为这条事件里有：

- 当前协议版本
- 服务端时间
- `resume` 结果

---

## 17. 断线重连怎么做

### Flutter 端必须做的事

每收到一条事件，都保存：

- `event_id`

这个值就是下次重连要带回去的 `last_event_id`。

### 重连流程

1. WebSocket 断开
2. 本地保留最后一个 `event_id`
3. 重连时把它作为 `last_event_id`
4. 收到 `system.hello`
5. 看 `hello.payload.resume`

### `resume` 怎么判断

#### 情况 A：可以直接续上

如果：

- `accepted=true`

说明：

- 服务端认得你带回来的 `last_event_id`
- 断线期间漏掉的事件会直接重放

这时 Flutter 端：

- 继续处理重放事件
- 不需要立即重新拉 `bootstrap`

#### 情况 B：服务端要求重拉

如果：

- `should_refetch_bootstrap=true`

说明：

- 服务端缓存里已经找不到那个事件了
- 或者漏掉的事件太多，超过回放上限

这时 Flutter 端必须：

1. 重新调 `GET /api/app/v1/bootstrap`
2. 如果当前正停留在某个聊天页，再补一次该会话的 `GET /messages`

不要继续假装本地状态是对的。

### 当前后端会给出的重连失败原因

- `last_event_id_not_found`
- `replay_limit_exceeded`

## 18. 事件类型到底该怎么消费

下面只写**当前已经实现**的事件，不写草案里还没实现的。

### 18.1 `system.hello`

用途：

- WebSocket 建立后第一条握手消息
- 告诉客户端这次重连是否成功续上

Flutter 端处理：

- 更新本地 `connectionReady=true`
- 读取 `resume`
- 决定要不要重新拉 `bootstrap`

### 18.2 `session.message.created`

用途：

- 当前主要表示“用户消息已被后端接收”

Flutter 端处理：

- 如果本地还没有这条 `message_id`，插入
- 如果本地已经有这条 `message_id`，就合并，不要重复显示

### 18.3 `session.message.progress`

用途：

- 表示 AI 正在处理中

当前常见 `kind`：

- `thinking`
- `tool_hint`

Flutter 端处理建议：

- 把它当“临时进度气泡”或“输入中状态”
- 不一定要永久写进消息历史
- 可以按 `task_id` 或 `message_id` 覆盖更新

### 18.4 `session.message.completed`

用途：

- AI 最终回复完成

Flutter 端处理：

- 插入或更新 assistant 消息
- 清掉对应 `task_id` 的临时进度状态

### 18.5 `session.message.failed`

用途：

- 当前任务失败或被取消

Flutter 端处理：

- 结束 loading
- 显示失败提示
- 如果有临时进度气泡，改成失败状态或移除

### 18.6 `runtime.task.current_changed`

用途：

- 当前前台任务变了

Flutter 端处理：

- 更新首页状态条
- 更新“正在执行中”卡片

### 18.7 `runtime.task.queue_changed`

用途：

- 排队任务变了

Flutter 端处理：

- 更新任务队列 badge
- 更新首页“等待中”列表

### 18.8 `device.connection.changed`

用途：

- 设备连上 / 断开

Flutter 端处理：

- 更新设备在线状态

### 18.9 `device.state.changed`

用途：

- 设备状态机变化

当前状态值：

- `IDLE`
- `LISTENING`
- `PROCESSING`
- `SPEAKING`
- `ERROR`

Flutter 端处理：

- 更新设备页主状态
- 必要时切换页面提示文案

### 18.10 `device.status.updated`

用途：

- 电量、WiFi、充电状态变化

Flutter 端处理：

- 更新设备详情卡

### 18.11 `todo.summary.changed`

用途：

- Todo 摘要变化

Flutter 端处理：

- 更新首页 Todo 卡片
- 更新全局概览

### 18.12 `calendar.summary.changed`

用途：

- 日历摘要变化

Flutter 端处理：

- 更新首页日历卡片
- 更新“下一场事件”提示

## 19. 一个最实际的聊天页处理流程

建议 Flutter 聊天页就按这个顺序来：

1. 进入聊天页
2. 调 `GET /sessions/{id}/messages?limit=50`
3. 渲染历史消息
4. 用户发送消息
5. 调 `POST /sessions/{id}/messages`
6. 用 `accepted_message` 立刻插入本地消息
7. 收到 `session.message.progress`
8. 显示“正在思考”
9. 收到 `session.message.completed`
10. 插入最终回复
11. 收到 `runtime.task.current_changed`
12. 更新顶部全局状态

重点是：

- 用户消息靠 `POST` 返回先落本地
- AI 回复靠事件流完成
- 全局任务状态靠 runtime 事件，不靠聊天文本推断

## 20. 一个最实际的首页处理流程

首页建议只关心 3 类数据：

- 会话列表
- 全局运行态
- 设备 / Todo / 日历摘要

推荐流程：

1. `GET /bootstrap`
2. 渲染首页
3. 建 WebSocket
4. 后续通过这些事件增量更新：
   - `runtime.task.current_changed`
   - `runtime.task.queue_changed`
   - `device.*`
   - `todo.summary.changed`
   - `calendar.summary.changed`

这样首页就不需要高频轮询。

## 21. 当前 Flutter 端最该注意的 6 个坑

### 21.1 不要把事件消息重复插入

去重主键：

- 消息用 `message_id`
- 事件游标用 `event_id`

### 21.2 不要把 progress 当最终聊天记录

`session.message.progress` 更像临时态，不是最终答复。

### 21.3 不要断线后直接无脑继续

一定先看 `system.hello.resume`。

### 21.4 不要靠轮询顶替事件流

当前后端就是按“REST 拿快照 + WebSocket 拿实时”设计的。

### 21.5 不要把 Todo / 日历摘要当完整模块

现在只是摘要层，不是完整 CRUD。

### 21.6 不要把设备、App、WhatsApp 当成同一个会话

它们共享运行态，但聊天记录是分开的。

## 22. 当前最推荐的 Flutter 接法

一句话版本：

> 启动调 `bootstrap`，聊天页按需拉 `messages`，发消息走 `POST`，所有实时变化统一走 `ws/app/v1/events`，每条事件都保存 `last_event_id`。

如果严格按这个方式接，当前这版本地局域网后端已经够 Flutter 第一版页面联调了。

## 24. HTTP 返回包统一长什么样

当前 App API 的 HTTP 返回都有统一外壳。

成功时是：

```json
{
  "ok": true,
  "data": {},
  "request_id": "req_xxx",
  "ts": "2026-03-26T12:00:00+08:00"
}
```

失败时是：

```json
{
  "ok": false,
  "error": {
    "code": "UNAUTHORIZED",
    "message": "unauthorized"
  },
  "request_id": "req_xxx",
  "ts": "2026-03-26T12:00:00+08:00"
}
```

Flutter 端建议统一这样处理：

1. 先看 HTTP status
2. 再看 `ok`
3. `ok=true` 时只读 `data`
4. `ok=false` 时统一转成你自己的 `ApiException`

不要每个页面自己零散判断。

## 25. WebSocket 每条事件外层结构

WebSocket 事件外层字段是稳定的，当前最重要的是这几个：

- `event_id`
- `event_type`
- `scope`
- `occurred_at`
- `session_id`
- `task_id`
- `payload`

可以把它理解成：

- `event_id`：事件游标，断线恢复要靠它
- `event_type`：事件分发主键
- `scope`：这是全局事件还是某个会话事件
- `session_id`：如果和某个聊天会话有关，这里会带出来
- `task_id`：如果和某个任务有关，这里会带出来
- `payload`：真正业务内容

建议 Flutter 端的事件处理顺序固定成：

1. 先解析整条事件
2. 先保存 `event_id`
3. 再按 `event_type` 分发
4. 最后更新页面状态

不要反过来先改 UI，最后才存 `event_id`。

## 26. Flutter 端至少要识别的错误

当前接入时最常见的错误基本就这些：

- `UNAUTHORIZED`
  - token 不对，或者没带 token
- `INVALID_ARGUMENT`
  - 参数不对，或者 JSON body 非法
- `SESSION_NOT_FOUND`
  - 会话不存在
- `TASK_NOT_FOUND`
  - 要停止的任务不存在
- `TASK_NOT_CANCELLABLE`
  - 这个任务当前不能停
- `DEVICE_OFFLINE`
  - 设备没连上，不能播报

Flutter 端建议至少分成 3 类处理：

- 可直接提示用户的业务错误
- 需要重新登录或重新配对的鉴权错误
- 需要自动重试或重拉快照的同步错误

其中最值得单独处理的是：

- `UNAUTHORIZED`
  - 直接引导用户检查服务器地址和 token
- `DEVICE_OFFLINE`
  - 不要弹成“系统异常”，直接提示设备未连接

## 27. 当前已经有的，和还没有的

为了避免 Flutter 端先做错方向，这里把边界说死一点。

### 当前已经有的

- App 会话列表
- App 会话消息分页
- 发消息入队
- 实时事件流
- 事件断线续传
- 全局运行态
- 任务停止
- 设备快照与设备播报
- Todo / 日历摘要占位接口

### 当前还没有的

- 完整 Todo 列表 CRUD
- 完整日历事件 CRUD
- 面向 Flutter 的远程公网接入方案
- 多用户体系
- 复杂权限角色系统
- 更细粒度的消息增量同步协议

所以 Flutter 第一版最稳的目标不是“全功能生活助手”，而是：

> 一个在本地局域网内稳定可用的 AI 控制中枢 App。
