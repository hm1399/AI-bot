# AI-Bot 后端优化强化计划

> 创建日期：2026-03-25  
> 适用范围：`server/`、`server/bridge/` 及后续 App/设备接入相关后端能力  
> 本文性质：实施计划，不包含大量代码  
> 当前判断：现有后端已达到“可演示、可联调”的 Demo 水平，但距离“稳定迭代的产品型后端”仍有明显差距

---

## 0. 当前已确认方向（2026-03-25）

以下内容已和当前产品方向确认，可作为后续优化的约束条件：

- [已完成：方向确认] **总体目标不是上云平台**，而是先把后端做成“运行在用户电脑上的稳定本地 AI 中枢”。
- [已完成：方向确认] **部署形态为本地单机**：硬件放在家中，与电脑走局域网连接；Flutter App 当前阶段只做本地局域网接入与控制。
- [已完成：方向确认] **网络范围控制**：当前阶段只做本地局域网能力，不把“App 远程控制家里电脑”作为本轮范围；若后续时间充裕再单独规划远程接入方案。
- [已完成：方向确认] **短期优先级同时覆盖两件事**：一是把现有设备 Demo 做稳，二是为 Flutter App 接入铺路；但执行顺序上先做 `Phase A + Phase B`，再进入 App 能力建设。
- [已完成：方向确认] **客户端定位**：WhatsApp 仍保留为测试通道，但不是长期正式入口；后续正式客户端以 Flutter App 为主。
- [已完成：方向确认] **会话策略**：设备、Flutter App、WhatsApp 各自保持独立会话，不强行并成一个聊天线程。
- [已完成：方向确认] **共享状态策略**：虽然会话独立，但不同通道都应能看到当前执行中的任务、任务队列，以及后续的 Todo / 日历行程等共享状态。
- [已完成：方向确认] **并发目标**：当前只面向单人单设备、单人多端，不以多人多设备、云端多租户为本轮目标。
- [已完成：方向确认] **高风险能力控制方式**：采用可配置权限，而不是默认全部开放或彻底禁用。

## 1. 计划目标

本轮后端优化不再以“先跑通功能”为核心，而改为以以下四个目标为核心：

1. **稳定性增强**：让现有设备语音链路、Agent、WhatsApp 转发在长时间运行下更可靠。
2. **可扩展性增强**：让后端不仅能服务 ESP32，也能平滑支持 Flutter App、管理端、更多消息通道。
3. **安全性增强**：补齐认证、权限、配置隔离、风险命令约束、敏感信息管理。
4. **工程化增强**：补足测试、日志、健康检查、配置校验、部署文档和运维入口。

---

## 2. 当前基线判断

### 已具备的能力

- 已有可运行入口、配置加载、日志、优雅关闭流程。
- 已有设备 WebSocket 通道、状态机、音频上下行、ASR/TTS 串联。
- 已有 AgentLoop、MessageBus、SessionManager、工具调用闭环。
- 已有 WhatsApp Bridge，能完成基础消息收发与转发。
- 已有手工测试入口，可支持单人单设备 Demo 联调。

### 当前主要短板

- HTTP API 过薄，尚不足以支撑 App、管理端或多客户端接入。
- 安全边界偏弱，设备侧 WebSocket 缺少明确认证与权限隔离。
- 并发模型偏保守，核心消息处理仍偏串行，不利于多来源扩展。
- 通道路由仍有 Demo 化实现，存在 hack 和隐性耦合。
- 自动化测试不足，缺少端到端回归保护。
- 配置、依赖、文档存在不一致，后续维护成本会变高。

### 总体定位

- **现在**：课程项目 / Demo 后端  
- **本轮目标**：稳定的本地 AI 设备 / 电脑中枢  
- **下一阶段目标**：可持续迭代的本地多端产品原型后端  
- **暂不追求**：云端部署、多租户平台、完整云原生生产体系

---

## 3. 设计原则

后续所有优化建议遵循以下原则：

1. **先补短板，再做新功能**：先把稳定性、安全性、接口边界做好，再继续加功能。
2. **优先解耦主链路**：设备接入、AI 编排、第三方消息通道、App API 要逐渐拆清边界。
3. **保留现有 Demo 成果**：优化过程中不能破坏现有“设备说话 → AI 回复 → 设备播报/WhatsApp 同步”的主流程。
4. **渐进式演进**：先做低风险重构与补齐测试，再推进结构升级。
5. **接口先行**：后续 App 接入前，先明确 API、事件模型、认证模型和状态模型。

---

## 4. 总体实施阶段

建议分为 5 个阶段推进，而不是一次性大改。

| 阶段 | 名称 | 目标 |
|------|------|------|
| Phase A | 稳定性止血 | 修掉明显的结构性风险，让当前 Demo 更稳 |
| Phase B | 服务边界整理 | 把设备、Agent、通道、接口层次拆清 |
| Phase C | API 与多客户端接入 | 补齐 Flutter/App 所需 API 与会话能力 |
| Phase D | 安全与运维增强 | 增加认证、权限、观测、部署能力 |
| Phase E | 测试与发布闭环 | 建立回归测试、验收流程和迭代基线 |

---

## 5. Phase A：稳定性止血

### A1. 梳理消息主链路

目标：把当前“设备消息 → Agent → 多通道回发”的路径梳理清楚，减少隐藏耦合。

计划项：

- 明确 `DeviceChannel`、`WhatsAppChannel`、`AgentLoop`、`MessageBus` 的职责边界。
- 统一 inbound / outbound 消息字段约定，补全必要 metadata 定义。
- [已完成] 清理“由运行时临时 patch 某个 channel 内部方法”这类 Demo 化路由方式。
- [已完成] 建立“回复路由策略”的单独模块，而不是散落在入口文件中。

验收标准：

- 不再依赖临时 monkey patch 才能正确路由消息。
- 设备回复、WhatsApp 回复、未来 App 回复都能通过统一路由策略派发。

### A2. 强化设备链路容错

目标：让 ESP32 长时间运行时更稳定。

计划项：

- [已完成：第一版] 统一设备连接生命周期：连接、重连、断线、旧连接替换、异常恢复。
- 为音频接收增加更明确的边界检查、异常日志和超时处理。
- [已完成：第一版] 增加“收到录音开始但迟迟没有 `audio_end`”的超时恢复策略。
- [已完成：第一版] 增加“播放中被打断”“识别失败”“TTS 失败”的状态回收保护。
- 为设备状态、天气推送、心跳任务增加更明确的任务状态管理。

验收标准：

- 断线重连后设备能稳定恢复到可交互状态。
- 连续多轮语音交互不会出现状态卡死或任务泄漏。

### A3. 优化 Agent 并发模型

目标：避免所有请求都被全局锁串死。

计划项：

- [已完成：第一版] 评估当前全局处理锁是否改为“按 session 串行、不同 session 并行”。
- 为长耗时工具调用、ASR、TTS、外部通道发送增加超时与取消路径。
- [已完成：第一版] 为 `/stop`、中断语音播放、取消任务建立统一机制。
- 设计背压策略：设备消息、WhatsApp 消息、App 消息同时到来时如何限流。

验收标准：

- 单一 session 内仍保持上下文一致。
- 不同来源/不同 session 的请求不再完全串行。

## 6. Phase B：服务边界整理

### B1. 重构后端分层

目标：把现有“入口文件拼装一切”的形态，演进成更清晰的分层结构。

建议分层：

- **入口层**：启动、关闭、配置装配、依赖注入。
- **接口层**：HTTP API、设备 WebSocket、第三方通道接入。
- **应用层**：消息路由、会话协调、任务调度、状态编排。
- **领域层**：设备状态、会话模型、消息模型、路由策略。
- **基础设施层**：LLM provider、ASR/TTS、存储、外部桥接。

计划项：

- [已完成：第一版] 把路由装配、服务装配、通道装配从 `main.py` 中拆出去。
- 统一服务注册和依赖传递方式，减少隐式共享。
- 为消息模型、设备事件、通道事件建立统一的数据结构定义。

验收标准：

- `main.py` 只负责组装与启动，不再承载主要业务逻辑。
- 新增一个新通道或新接口时，不需要修改太多旧逻辑。

### B2. 统一配置体系

目标：让配置真正可维护、可校验、可扩展。

计划项：

- 统一 `config.yaml`、环境变量、运行时配置之间的优先级。
- [已完成：第一版] 补齐缺失依赖与配置项的启动前校验。
- 为开发、测试、演示环境提供配置分层方案。
- 将敏感配置、实验配置、设备配置、通道配置分类。
- 输出一份标准配置说明文档和 `.env.example`。

验收标准：

- 新环境拉起时，不需要靠猜配置。
- 配置错误能在启动阶段被明确提示，而不是运行时报错。

### B3. 清理依赖与版本基线

目标：减少“本机能跑、别人环境跑不起来”的风险。

计划项：

- [已完成：第一版] 核对 Python 依赖声明与当前主运行链路的实际 import 是否一致。
- [已完成：第一版] 核对 Node bridge 依赖声明与源码 import 是否一致。
- 固定推荐 Python、Node 版本与关键库兼容范围。
- 补充安装说明、首次启动说明、常见失败排查说明。

验收标准：

- 新机器按文档安装后，能稳定启动服务端和 bridge。

## 7. Phase C：API 与多客户端接入

> 当前状态：**已完成第一版基础能力落地**。当前后端已补上本地局域网版 Flutter App 的最小 REST API、共享运行态、WebSocket 事件流与基础鉴权骨架；后续继续补分页、重连同步、Todo / 日历摘要细化。

### C1. 设计 App 可用 API

目标：让后端不再只服务设备和 WhatsApp，而是能服务 Flutter App。

建议新增的 API 能力：

- 会话列表查询
- 单会话消息历史查询
- 发送文本消息
- 查询设备在线状态
- 查询当前服务健康状态
- 查询当前模型、通道、系统能力
- 发起控制动作（如刷新配置、停止当前任务、设备播报）

计划项：

- [已完成：草案] 先输出 API 草案，再实现最小版本。
- [已完成：第一版] 已实现最小 App API：`/api/app/v1/bootstrap`、`/sessions`、`/sessions/{id}/messages`、`/runtime/state`、`/runtime/stop`、`/device`、`/device/speak`、`/capabilities`。
- [已完成：第一版] 已统一返回结构、错误码与局域网共享 token 鉴权骨架。
- [已完成：第二段] 已为消息历史接口补上 `before / after / limit` 游标分页与分页元信息返回。
- [已完成：第三段] 已补上 Todo / 日历摘要占位接口：支持查询当前摘要、更新摘要，并写入共享运行态返回。
- [进行中] 继续细化“设备侧接口”“App 接口”“管理接口”的长期分层，当前已完成设备接口与 App 接口拆分，管理接口后续单独规划。

验收标准：

- Flutter 端不需要依赖设备 WebSocket 协议，也能完成基础聊天和状态查询。

### C2. 建立实时事件通道

目标：让 App 能实时看到处理进度、状态变化、设备事件。

计划项：

- [已完成：草案] 为 App 设计单独的实时通道，可选 WebSocket 或 SSE。
- [已完成：第一版] 已落地 `WebSocket /ws/app/v1/events`。
- [已完成：第一版] 已接入最小事件集：`session.message.created`、`session.message.progress`、`session.message.completed`、`session.message.failed`、`runtime.task.current_changed`、`runtime.task.queue_changed`、`device.connection.changed`、`device.state.changed`、`device.status.updated`。
- [已完成：第二段] 已支持基于 `last_event_id` 的断线重连事件重放；若游标过旧或超出回放上限，服务端会明确要求客户端重新拉取快照。
- [已完成：第三段] 已接入 `todo.summary.changed` 与 `calendar.summary.changed`，并把摘要更新纳入事件回放缓存。
- [已完成：草案] 统一事件命名和事件载荷模型。

验收标准：

- App 可以实时看到“正在识别 / 正在思考 / 正在回复 / 任务完成”等状态。

### C3. 会话模型升级

目标：从“单通道 demo 会话”升级为“多客户端共享会话”。

> 方向修正（已确认）：这里的“共享”不再指“多个通道共用同一个聊天会话”，而是指“多个通道共享任务态与全局信息，但聊天记录各自独立”。

计划项：

- [已完成：第一版] 已按 `channel:chat_id` 落地 session 唯一键，并为 App 默认启用 `app:main` 会话。
- [已完成：草案] 设计“会话域”和“全局任务域”两层模型：前者保存各通道聊天上下文，后者保存当前任务、任务队列、Todo、日历等共享状态。
- [已完成：第一版] 已补齐 App 会话标题、置顶、归档、最近消息时间等最小字段。
- [已完成：第二段] 已为 App 消息写入补齐稳定 `message_id / task_id` 持久化，供分页游标和事件关联复用。
- [已完成：第三段] 已为 Todo / 日历摘要增加本地持久化文件，重启后仍可恢复占位摘要状态。
- [进行中] 后续继续补会话标签、来源标记、关联设备等字段。
- 评估是否继续以 JSONL 为主，还是引入更适合本地查询与状态同步的存储。

验收标准：

- 能明确区分：同一个人不同渠道的独立会话，以及跨通道共享的任务状态与系统状态。

## 8. Phase D：安全与运维增强

### D1. 认证与权限

目标：补上最基础的安全边界。

计划项：

- [已完成：第一版] 为设备 WebSocket 增加接入认证（当前为可选共享 token，适用于本地局域网阶段）。
- 为 App API 增加身份认证与最小权限控制。
- 为管理类接口单独设置权限级别。
- 明确 `exec` 等高风险工具的调用策略、审计策略和白名单/黑名单策略。
- 区分“本地设备可信调用”“局域网 App 调用”“第三方通道调用”三类权限模型，并为未来远程接入预留扩展位。

验收标准：

- 任意局域网设备不能无认证接入核心服务。
- 高风险操作可追踪、可限制、可关闭。

### D2. 观测性增强

目标：出问题时能快速定位，而不是靠猜。

计划项：

- 统一日志字段：请求来源、session、channel、耗时、错误类型。
- 增加健康检查与就绪检查区分。
- 增加关键指标统计：消息处理耗时、ASR 耗时、TTS 耗时、失败率、重连次数。
- 为异常链路增加错误分级与告警出口。

验收标准：

- 出现“设备没回复”“AI 很慢”“WhatsApp 不转发”等问题时，有明确日志与指标能定位。

### D3. 部署与运行管理

目标：让服务稳定常驻，而不是仅靠手工终端启动。

计划项：

- 设计本地常驻运行方式，如 `launchd`、`systemd` 或守护脚本。
- 给 Python 服务与 Node bridge 分别设计启动、停止、重启策略。
- 加入启动前自检：配置、依赖、端口占用、关键目录、模型/网络依赖可达性。
- 明确日志目录、缓存目录、会话目录、备份目录的管理方式。

验收标准：

- 服务异常退出后可自动恢复。
- 重新部署或重启后不会丢失关键会话数据和 bridge 身份数据。

## 9. Phase E：测试与发布闭环

### E1. 建立测试金字塔

目标：把“只能靠手工验证”升级为“有回归保护”。

建议测试层次：

- **单元测试**：消息模型、配置加载、状态机、路由策略、工具包装层。
- **集成测试**：设备通道、AgentLoop、SessionManager、WhatsApp Channel。
- **端到端测试**：模拟设备音频/文本输入，验证完整响应链路。
- **冒烟测试**：启动服务后检查关键接口与关键依赖是否可用。

计划项：

- 为 ASR/TTS/LLM 提供 mock 或 fake 实现，降低测试成本。
- 为 WebSocket 设备通道编写自动化模拟器。
- 为 bridge 和 Python 侧的消息协议建立契约测试。

验收标准：

- 每次改动后，都能快速知道主链路是否被破坏。

### E2. 建立版本验收清单

目标：每一轮迭代都有明确可验收标准。

建议每次发布前至少检查：

- 服务能启动
- 设备能连接
- 文本消息能来回
- 语音消息能来回
- WhatsApp 能收发
- API 文档与配置文档同步更新
- 关键测试通过

验收标准：

- 不再出现“代码看起来改好了，但不知道能不能跑”的情况。

## 11. 建议迭代顺序

建议不要按“模块”推进，而要按“风险先后”推进：

### 第 1 轮：保现有 Demo 不崩（已确认优先）

- 消息路由重构
- 设备链路容错
- 任务取消/超时
- 配置与依赖清理

### 第 2 轮：把结构拆清（已确认优先）

- `main.py` 瘦身
- 服务层与接口层拆分
- 统一消息模型与配置模型

### 第 3 轮：支持 Flutter App（方向已确认，待前两轮完成后启动）

- 设计并实现基础 API
- 增加实时状态事件
- 会话查询与消息发送闭环

### 第 4 轮：增强安全与运维（保留）

- 认证、权限、审计
- 观测、告警、守护
- 启动与部署文档

### 第 5 轮：测试与发布规范化（保留）

- 自动化测试补齐
- 回归清单固化
- 版本发布流程固定

## 13. 风险与注意事项

### 需要重点避免的风险

- 为了加 App 接口而破坏现有设备语音链路
- 一次性大重构，导致短期内无法稳定演示
- 在没有测试保护时贸然调整 Agent 并发模型
- 把安全逻辑、业务逻辑、通道路由逻辑继续混在一起
- 文档不更新，导致后续自己都无法快速恢复环境

### 建议的实施方式

- 每次只做一个主目标，不要同时做路由、认证、存储三类大改。
- 每个阶段结束都做一次真实设备联调。
- 每完成一轮结构调整，优先补最小测试，而不是继续堆功能。

## 14. 最终目标画像

优化完成后的后端，应该具备以下特征：

- 对设备来说：连接稳定、状态明确、语音链路可恢复。
- 对 App 来说：有清晰 API、有实时事件、有可查询会话。
- 对开发来说：结构清晰、配置明确、测试可跑、日志可查。
- 对运维来说：能守护运行、能定位问题、能安全控制。

一句话总结：

> 现阶段后端要从“能演示的 AI 设备后端”升级为“能持续迭代的多端 AI 后端底座”。
