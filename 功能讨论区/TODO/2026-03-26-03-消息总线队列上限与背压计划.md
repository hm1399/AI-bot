# AI-Bot 消息总线队列上限与背压计划

> 创建日期：2026-03-26  
> 目标问题：当前 MessageBus 使用无上限队列，消息高峰时缺少背压和拒绝策略  
> 本文性质：实施计划，不包含大量代码

---

## 1. 问题定义

现在的 inbound / outbound 队列是无上限的。

这在 Demo 阶段问题不明显，但一旦进入正式功能开发，会遇到几个风险：

- Flutter 在弱网下重试，可能短时间堆很多消息。
- 设备连续触发语音输入，可能把待处理消息越堆越多。
- 模型接口卡住时，消息消费速度下降，队列会持续上涨。
- 进程不会立即报错，但内存会越来越高，最后变成“慢慢坏掉”。

这类问题最麻烦，因为它不是立刻挂，而是越跑越不稳定。

---

## 2. 目标

建立一套适合本地单机版的简化背压体系，让后端在忙的时候：

- 知道自己忙
- 能限制排队长度
- 能优先保留关键控制消息
- 能给 App 明确反馈“现在太忙了”

目标不是做复杂中间件，而是让当前本地后端具备最基础的自我保护能力。

---

## 3. 范围

本轮覆盖：

- MessageBus inbound 队列
- MessageBus outbound 队列
- App 请求的高峰保护
- 控制类消息的优先级保护

本轮先不覆盖：

- 分布式消息系统
- 外部 broker
- 多进程消费者

---

## 4. 设计原则

1. 队列必须有上限，不能无限堆积。
2. 控制类消息优先级高于普通聊天消息。
3. App 在队列满时要收到明确错误，而不是一直假装成功。
4. 背压策略第一版要简单，不要一次上太复杂的调度器。
5. 队列水位要能被观测，不然很难调优。

---

## 5. 实施计划

### Phase 1：给队列加容量边界

先把 inbound / outbound 队列都改成可配置容量。

建议配置项至少包含：

- inbound 最大长度
- outbound 最大长度
- 控制消息预留空间

第一版不追求动态调参，先有明确上限。

### Phase 2：区分普通消息和控制消息

建议至少区分两类消息：

- 普通业务消息
  - App 发消息
  - 设备输入消息
  - 通道普通消息
- 高优先级控制消息
  - `/stop`
  - 关键状态切换
  - 必须尽快到达的系统控制动作

这样做的原因很现实：

如果队列已经满了，最不能被堵死的反而是“停止”和“取消”。

### Phase 3：定义满载时的处理策略

当队列达到上限时，不同来源要有不同处理：

- App HTTP 请求：
  - 直接返回忙碌错误
  - 告诉客户端稍后重试
- 设备连续输入：
  - 需要更保守，避免重复堆积
  - 可以拒绝新输入或提示设备当前忙碌
- 控制消息：
  - 尽量仍可进入

第一版建议优先做“拒绝新普通消息”，不要尝试复杂的自动丢弃老消息。

### Phase 4：把忙碌状态传递到接口层

背压不能只停留在消息总线内部。

需要把“队列满 / 服务繁忙”向上传递到：

- App REST API
- 设备侧必要提示
- 日志与健康状态

这样前端和设备端才知道是“服务在忙”，而不是“神秘失败”。

### Phase 5：增加观测指标

建议至少记录：

- 当前 inbound 深度
- 当前 outbound 深度
- 最近一次满载时间
- 最近一段时间被拒绝的请求数

如果这些都没有，后面很难知道容量到底设得对不对。

### Phase 6：补测试

至少补以下测试：

- 队列到上限后，普通消息会被拒绝
- 高优先级控制消息仍可进入
- App 在满载时收到明确错误
- 消费恢复后，系统可以继续正常工作
- 不同 session 并发时，队列行为仍符合预期

---

## 6. 推荐第一版方案

为了别把工程做重，建议第一版只做：

1. 有上限的队列
2. 普通消息 / 控制消息的简单区分
3. App 忙碌错误返回
4. 基本队列统计

先把“无限堆积”这个大坑堵上，再考虑更细的调度策略。

---

## 7. 风险点

- 如果直接简单丢消息，但接口层不知道，会造成严重误判。
- 如果所有消息一视同仁，队列满时连 `/stop` 都进不去。
- 如果容量设得太小，会造成频繁误伤。
- 如果没有统计指标，后面根本不知道问题出在“模型慢”还是“队列爆了”。

---

## 8. 验收标准

- inbound / outbound 队列都有明确上限。
- 服务忙碌时不会无限吃内存。
- 队列满时，App 会收到明确可处理的错误。
- `/stop` 等关键控制动作在高压下仍尽量可用。
- 队列容量和拒绝行为有测试覆盖。

---

## 9. 完成后的直接收益

这项做完后，后端面对下面这些情况会稳很多：

- Flutter 重试风暴
- 设备连续说话
- 模型短时变慢
- 多通道同时来消息

你后面做正式功能时，就不会一上量就撞上“后端怎么突然越来越卡”的问题。
