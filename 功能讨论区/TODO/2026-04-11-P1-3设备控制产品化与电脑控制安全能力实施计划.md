# P1-3 设备控制产品化与电脑控制安全能力 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 完成总 TODO 里 `P1-3` 的真实剩余项：把现有设备控制补成清晰的产品能力，并把当前底层 `filesystem / shell / exec` 收成安全、可验证、可审计的电脑控制能力集合。

**Architecture:** 继续沿用现有 `Flutter App -> app-v1 HTTP/WebSocket -> AppRuntimeService` 主链路，不新增第二套控制面。设备控制部分只补产品语义和 Settings 联动，不重做已存在的 `device_command_result + runtime snapshot` 闭环；电脑控制部分新增结构化 `computer_control` 服务层、事件流和审批状态机，Chat/语音负责发起意图，Control Center 负责展示、确认和审计。

**Tech Stack:** Flutter + Riverpod, Python `aiohttp`, existing AgentLoop/tool registry, macOS built-in `open` / `pbcopy` / `pbpaste` / `screencapture` / `shortcuts`, AppleScript / System Events (only when native or dedicated wrappers are unavailable)

## 调研结论摘要

### 代码现状

- `server/channels/device_channel.py`、`server/services/app_runtime.py`、`app/lib/models/home/runtime_state_model.dart`、`app/lib/screens/control_center/control_center_screen.dart` 已经具备 `device_command_result`、runtime 设备快照、命令状态回读和 Control Center 真读回 UI。
- 当前真正没收口的是“产品语义”，不是“协议完全不存在”：
  - `Settings` 保存后只有 `device_volume` 会尝试立即下发，`led_enabled / led_brightness / led_color` 仍只是配置字段。
  - `Settings` 前端当前只显示 `Settings saved through the backend.`，没有告诉用户哪些值已经生效，哪些只是保存配置。
  - 能力权限边界和失败反馈仍主要是 `globalMessage` 文案，没有统一能力模型。
- 电脑控制仍停留在 agent 底层工具面：
  - `server/nanobot/agent/loop.py` 只注册了 `read_file / write_file / edit_file / list_dir / exec / web / planning / spawn`。
  - `server/nanobot/agent/tools/shell.py` 当前是通用 shell 包装，不是 allowlist 产品能力。
  - `server/workspace/skills/computer-control/SKILL.md` 和 `server/workspace/SOUL.md` 仍把“控制电脑”直接等同于 raw `exec`。
  - Flutter `CapabilitiesModel`、AppRuntime 能力面、Control Center UI 都还没有“电脑控制”这一层。

### 外部资料与系统能力复核

- 2026-04-11 复核 Apple 官方资料后，当前更稳的 v1 路线是“系统内建命令 + 专用包装 + 必要时 AppleScript/UI scripting”，而不是一开始就做自由鼠标键盘模拟。
- Apple 官方确认 Shortcuts 支持命令行运行，因此“固定脚本 / 固定流程”优先可以走 `shortcuts run` 这类稳定入口。
- Apple 平台安全文档明确说明 macOS 对 `Files & Folders`、`Automation (Apple events)`、`Accessibility`、`Screen Recording` 都有用户授权边界，因此权限模型必须成为产品 contract 的一部分，不能只留给 stderr。
- Apple 官方自动化文档明确说明并非所有 App 都有 scripting dictionary；没有原生脚本支持时才应退到 GUI scripting，而且 Accessibility 需要用户逐 app 授权。
- 本机命令面已经可用并适合作为第一批最小底座：
  - `open` 适合 `open_app / open_path / open_url`
  - `pbcopy / pbpaste` 适合 `clipboard_set / clipboard_get`
  - `screencapture` 适合 `screenshot`
  - `shortcuts` 适合 `run_shortcut`

### 结论

- `P1-3` 不应再按“先补一个更强的 exec”来做，而应新增一层 `computer_control` 结构化能力面。
- 设备控制与电脑控制都必须统一成“可执行 + 可回读 + 可失败说明 + 可审计”的状态机。
- 第一批电脑能力必须是 allowlist、结构化返回、默认 JSON、能写验收用例的动作集合，而不是自然语言直接拼 shell。
- `微信发消息` 只能作为“生活技能适配器”落地，不能反向定义整个电脑控制底座；若当前没有稳定原生入口，只能走专用 UI scripting 适配器并保持二次确认。

## 产品决策

### 1. P1-3 拆成两条执行线

- **线 A: 设备控制产品语义收口**
  - 复用现有 `device_command_result` 和 runtime snapshot。
  - 把 `Control Center` 与 `Settings` 对齐到统一产品语义。
  - 明确哪些设置是 `保存并立即尝试下发`，哪些只是 `保存配置`.

- **线 B: 电脑控制安全能力层**
  - 新增 `computer_control` 服务层、审批状态机、审计日志和事件模型。
  - 不把 raw `exec` 暴露给 App 或产品能力。
  - 优先做可验收的结构化动作，不做通用 GUI agent。

### 2. Control Center 继续作为操作中枢

- 不新增主导航。
- `Control Center` 保留“运维控制台”语义，新增 `Computer Actions` 区块：
  - 快捷动作入口
  - 待确认动作
  - 最近执行记录
  - 权限说明与失败提示

### 3. Settings 保持配置页，但必须说清“保存”和“生效”的区别

- 第一批支持 `save + apply` 的字段限定为：
  - `device_volume`
  - `led_enabled`
  - `led_brightness`
  - `led_color`
- 继续只做配置、不承诺即时生效的字段明确标注为：
  - `led_mode`
  - `wake_word`
  - `auto_listen`
- `Settings` 保存结果需要返回每个受影响字段的 `apply_status`，而不是只给一句通用成功文案。

### 4. 电脑控制 v1 的能力边界

- 第一批结构化动作固定为：
  - `open_app`
  - `focus_app_or_window`
  - `open_path`
  - `open_url`
  - `run_shortcut`
  - `run_script`
  - `clipboard_get`
  - `clipboard_set`
  - `active_window`
  - `screenshot`
  - `system_info`
- `run_script` 只允许执行白名单脚本 ID，不接受任意路径。
- `system_info` 只允许固定 profile，如 `frontmost_app / disk / memory / network / battery / processes_summary`。
- 不在 v1 中加入：
  - 任意 shell 命令执行
  - 通用鼠标键盘代理
  - 浏览器点击/填表
  - 任意联系人发送消息

### 5. 风险与确认模型

- `low`:
  - `open_app`
  - `focus_app_or_window`
  - `open_url`
  - `active_window`
  - `system_info`
  - `clipboard_get`
- `medium`:
  - `open_path`
  - `clipboard_set`
  - `screenshot`
  - `run_shortcut`
  - `run_script`
- `high`:
  - 任意外发类动作
  - 带副作用的生活技能动作
  - 任何未来新增的 UI scripting 发送动作
- `high` 默认必须二次确认；`medium` 是否确认由动作定义和配置决定；`low` 可以直接执行。

### 6. WeChat 路线

- `微信发消息` 不直接实现成“大而全 GUI 自动化”。
- 先拆成：
  - `open_app(WeChat)`
  - `focus_app_or_window(WeChat)`
  - `clipboard_set(message_draft)`
  - `wechat_prepare_message(contact_alias, message)` 适配器
  - `wechat_send_prepared_message(action_id)` 高风险确认动作
- 若当前公开资料下没有稳定原生入口，则 `wechat_prepare_message` 走专用 UI scripting 适配器，并受以下限制：
  - 仅白名单联系人
  - 仅实验开关开启时可用
  - 默认只做到“草稿就位 + 等待确认”
  - 若稳定性不足，不把“自动发送”作为第一批必须完成项

## 验收标准

### 设备控制

- 用户在 `Settings` 保存设备相关字段后，能看到每个字段是 `已下发`、`下发失败` 还是 `仅保存配置`。
- `Control Center` 不再把本地草稿冒充为真实设备状态。
- 设备离线、超时和参数错误都有明确失败原因。

### 电脑控制

- App/agent 能以结构化方式发起 `open_app / open_path / run_script / system_info`。
- 每个动作都有：
  - `action_id`
  - `status`
  - `risk_level`
  - `requires_confirmation`
  - `result/error`
  - 可回看的日志
- 不存在“用户界面显示成功，但底层只是 raw exec 已受理”的假闭环。

### 生活技能

- “打开微信给张三发消息：我今晚晚点到。” 至少能走到 `draft_ready + awaiting_confirmation` 或明确失败原因。
- 高风险动作默认不自动发送。
- 失败时能说明卡在哪一步：联系人映射、权限、窗口定位、UI automation 或用户未确认。
