# Chat 页面桌面化重构与会话交互改版计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development（recommended）或 superpowers:executing-plans 按任务逐项实施。步骤使用 checkbox（`- [ ]`）语法跟踪。

**Goal:** 基于 `app/DESIGN.md` 的桌面工具化视觉约束，重构 Chat 页面为单主轴对话体验：像桌面版微信一样聚焦聊天流，但仍保持 AI-bot 的 operator console 语义；同时把 session 选择/新增收进右上角桌面化入口，移除底部弹窗，隐藏原始 task/resource ids 等对用户无意义的技术细节。

**Architecture:** 保留现有 `AppController + AppState + websocket session/message events` 作为唯一状态来源，不改会话、消息、active session、归档逻辑的后端协议。本轮主要改造 `chat_screen.dart` 的页面编排、`chat` widgets 的拆分方式，以及结构化消息的用户可见呈现层，把永久三栏布局收敛为“顶部命令区 + 单轴对话流 + 底部输入区 + 桌面弹出式会话浏览器”。

**Tech Stack:** Flutter desktop、Riverpod、现有 `LinearThemeTokens` / `LinearSpacing` / `LinearRadius` 主题系统、现有 Chat/Session provider 状态。

---

## 范围与约束

- 本轮只规划 Chat 页面 UI/UX 改版，不改后端协议，不改 Home / Agenda / Tasks / Control / Settings 页面。
- 必须以 `app/DESIGN.md` 的深色、紧凑、桌面优先、工具型风格为主，不做手机式底部弹窗，不做彩色社交聊天风。
- 同时服从你的新要求：Chat 页面消息体验要更像微信，主视觉中心就是对话本身，而不是一堆运营卡片。
- 保留现有会话逻辑能力：
  - 新建会话
  - 切换会话
  - rename / pin / archive / unarchive
  - active / archived 视图
  - active session 同步
- 不在主聊天内容里直接展示 `task_id / event_id / reminder_id / resource_ids` 这类用户无意义的底层标识。
- 如需保留调试能力，只能降到二级入口，不得继续占据主消息区。
- 这是桌面 app，Chat 页面不再保留 bottom sheet 作为主要会话交互方式。

## 调研结论摘要

### DESIGN.md 对本次改版的直接约束

- Chat 必须保持 `tool-like`，但信息组织应清楚，不能像现在这样把消息流切碎。
- 界面必须 `desktop-first`，不能让移动端 heuristics 主导桌面交互。
- Chat 允许深色卡片和工具栏，但不应继续依赖“说明卡 + 结构化总览卡 + 右侧常驻语音卡 + 左侧常驻会话栏”的拼盘式布局。
- 视觉上要继续使用：
  - 深色 canvas / panel / surface 分层
  - 细边框
  - 紧凑按钮和输入框
  - 克制的 accent
- `DESIGN.md` 原文里提到“Chat 不要变成 consumer messenger”，但这是通用守则；本轮以用户的新要求为准：允许借用微信式单轴对话体验，但配色、密度和控件风格仍维持 AI-bot 的桌面工具感。

### 当前 Chat 页面存在的问题

- 桌面端当前是三栏布局：
  - 左侧 `ChatSessionPanel`
  - 中间 `_ConversationView + MessageInput`
  - 右侧 `VoiceHandoffCard`
- 非桌面端仍保留 `_showSessionSheet()` 底部弹窗，这和桌面 App 目标冲突。
- 中间内容又叠了多层：
  - 会话头说明卡
  - 最新结构化结果总览卡
  - 消息列表
  - 输入框
  导致主消息流不再是页面主角。
- `MessageBubble` 目前仍把结构化 planning 内容直接作为大卡片塞进对话，并且会显示 `Resource IDs`，这对普通用户无意义，破坏聊天体验。
- `VoiceHandoffCard` 常驻右侧会占掉大量横向空间，使消息区像 dashboard 而不是聊天页。
- `ChatSessionPanel` 把 header、refresh、新建、筛选、会话列表、菜单动作全部耦合在一个左侧大面板里，不适合迁到右上角按钮后的桌面弹出层。

### 当前前端状态边界

- 现有会话/消息状态来源已经足够，不需要重构业务状态：
  - `AppState.connection.currentSessionId`
  - `AppState.sessions`
  - `AppState.messagesBySession`
  - `AppController.selectSession/createSession/renameSession/setSessionPinned/setSessionArchived/sendMessage`
- 说明本轮可以聚焦展示层，不必动后端和 websocket 事件模型。

## 方案对比

### 方案 A：保留三栏，只整理卡片顺序

- 做法：左侧 session 栏保留，右侧 voice 卡保留，只压缩头部和结构化结果卡。
- 优点：改动最小，风险低。
- 缺点：根问题没解决，Chat 依旧不像聊天页；用户最在意的“消息成为主角”和“会话入口收进右上角”都无法满足。

### 方案 B：单主轴聊天页 + 右上角桌面会话浏览器 + 轻量状态条

- 做法：把常驻左侧 session 栏和移动端 bottom sheet 全部移除；顶部右上角放 `New` 与 `Sessions`，点开后使用桌面化 dialog / popover 浏览和管理会话。页面主区只保留会话头、状态条、消息流、输入框，把 voice 状态收成顶部或空状态中的轻量信息。
- 优点：最符合你的要求，也最接近 `DESIGN.md` 的桌面工具化 + 单任务聚焦。
- 缺点：需要重新编排 ChatScreen 和拆分会话浏览器组件。
- 推荐度：最高。

### 方案 C：命令面板式会话切换器

- 做法：右上角只留一个 command palette 式 session chooser，新增/切换/归档都在命令面板完成。
- 优点：更高级，桌面感更强。
- 缺点：复杂度明显更高，对现有会话管理的 discoverability 不友好；第一轮容易做过头。

### 推荐方案

- 采用 **方案 B**。
- 理由：它最贴近用户目标，也不需要推翻现有状态逻辑；可以通过新增一个桌面会话浏览器组件来完成，而不需要重写 provider 或 backend。

## 目标交互设计

### 页面骨架

- 顶部：会话标题、简短状态元信息、右上角命令按钮组。
- 中部：单主轴消息流，占据页面绝大多数高度与宽度。
- 底部：紧凑输入区。
- 不再常驻：
  - 左侧会话面板
  - 右侧语音大卡
  - 结构化结果总览大卡

### 右上角按钮组

- `New`
  - 直接创建新会话，继续使用现有重命名/自动标题逻辑。
- `Sessions`
  - 打开桌面化 session browser。
- `More` 或会话菜单
  - 放 `Copy ID`、`Rename`、`Pin/Unpin`、`Archive/Restore`。

### Session Browser 交互

- 使用桌面化 `Dialog` 或 anchored `Popover`，不再使用 bottom sheet。
- 内部包含：
  - Active / Archived 切换
  - 会话列表
  - 列表项菜单
  - 新建入口
- 仍调用现有 controller 方法，不自建本地状态副本。

### 消息区交互

- 消息流必须成为页面核心视觉。
- 用户消息和助手消息都保留左右分侧，但整体更克制：
  - 更贴近聊天时间线
  - 更少 dashboard 感
  - 更少大块说明卡
- 来源标签保留，但压低视觉优先级。

### 结构化结果展示策略

- 不再在聊天页顶部单独放 “Latest Structured Result” 总览卡。
- 结构化 planning 信息回归到对应 assistant 消息内，作为折叠/精简的二级内容。
- 对用户直接展示：
  - 类型，如 task / reminder / event
  - 时间摘要
  - 冲突数
  - 是否需要确认
- 不直接展示：
  - `resource_ids`
  - 原始 bundle id
  - 底层 task/reminder/event 主键
- 如果后续确实需要技术细节，只能放到开发者入口或复制动作里，不能占据主聊天卡片。

### Voice 状态展示策略

- 不再常驻右侧大卡。
- 改为以下二选一之一：
  - 顶部状态条中的紧凑 voice status pills
  - 空聊天态中的 voice handoff hint 区
- 原则：语音状态要可见，但不能压缩主消息区。

## 验收口径

- 打开 Chat 后，视觉中心应当明确是消息流，而不是左/右两侧面板。
- 顶部右上角可以完成：
  - 新建会话
  - 打开会话列表
  - 进入会话管理动作
- 页面不再出现底部弹窗式会话管理。
- assistant 返回 task/reminder/event 后，用户能看到“做了什么”的摘要，但看不到原始 `task_xxx / event_xxx / resource_ids`。
- 语音状态仍可见，但不再占据右侧大栏。
- 页面整体风格应更贴近 `DESIGN.md` 的桌面工具感，同时具备你要的微信式单主轴对话体验。

## 风险与注意项

- `DESIGN.md` 里“Chat 不应变成 consumer messenger”与本轮“像微信一样的聊天体验”存在张力，实施时必须把“单轴对话体验”和“工具型深色风格”同时成立，不能做成手机社交聊天 app。
- `DESIGN.md` 中“当前五个主区”已落后于现有含 `Agenda` 的实际壳层；本轮只改 Chat，不回退其他导航结构。
- 当前 `AppController` 已较重，本轮应尽量限制在展示层改造，不要顺手扩大 provider 重构范围。
- 如果会话浏览器选择 popover 而不是 dialog，要额外注意滚动列表、分段筛选和菜单动作的焦点管理；第一轮更建议 dialog。
