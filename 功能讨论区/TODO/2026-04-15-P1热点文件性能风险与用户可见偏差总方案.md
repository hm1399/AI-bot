# P1 热点文件 / 性能风险 / 用户可见偏差总方案

> **For agentic workers:** 后续如按本方案执行，推荐使用 `superpowers:subagent-driven-development`，并统一使用 GPT-5.4 / xhigh。主线程负责热点文件收口与最终汇报；所有步骤使用 checkbox (`- [ ]`) 追踪。

**Goal:** 用一套不破坏现有 UI/UX、不过度推翻现有架构的方式，系统性修复 `waitlist.md` 当前 P1 的 14 个问题：后端 / 前端热点文件、默认 JSON 热路径、消息与广播背压、提醒调度模型，以及几类已经影响用户理解或配置正确性的行为偏差。

## 1. 调研结论摘要

### 1.1 后端热点与传输链路

- `server/services/app_runtime.py` 当前约 3673 行，既负责：
  - 40+ 个 HTTP / WebSocket 路由；
  - `MessageBus` / device / desktop voice / reminder / computer action 的 observer 回调；
  - runtime snapshot、planning summary、event history / replay；
  - settings / sessions / planning / device / computer control 的 API 聚合。
- `AppRuntimeService._broadcast_event()` 仍逐个 `await ws.send_json(event)`；任一慢客户端会直接拖慢整轮广播。
- `server/services/outbound_router.py` 仍是单 consumer 串行路由所有 outbound；`device / desktop_voice / whatsapp / app` 共用同一消费节拍。
- `server/nanobot/bus/queue.py` 仍是无界 `asyncio.Queue()`，且 `publish_inbound()` / `publish_outbound()` 先 `await _notify()` 再 `put()`，慢 observer 会直接阻塞入队。

### 1.2 存储与调度热路径

- `server/config.py` 仍把 `session_storage_mode` 和 `planning_storage_mode` 默认设成 `json`；`server/config.yaml` 目前也没有主动覆盖这两个默认值。
- `server/services/app_api/resource_service.py` 已有 `json / dual / sqlite` 三态和 `SQLitePlanningStore`，但默认运行口径仍以 `JsonCollectionStore` 为主。
- `server/nanobot/session/manager.py` 已支持 `json / dual / sqlite`，但默认会话仍落在 JSONL 全量重写路径。
- `server/services/reminder_scheduler.py` 仍靠固定 `poll_interval_s=15.0` 周期轮询全部 reminder；随着 reminder 数量增长，会持续空转，而且到点精度受轮询周期限制。

### 1.3 前端状态热点与用户可见偏差

- `app/lib/providers/app_providers.dart` 当前约 3578 行，同时承载：
  - 顶层 provider 定义；
  - `AppController` 70+ 个方法；
  - 事件流消费大 switch；
  - `DevicePairingController`；
  - 多个 selector/provider 派生。
- 该文件的 `_handleEvent()` 仍未处理 `desktop_voice.state.changed / transcript / response / error`，前端只消费 runtime 聚合结果，缺少实时语音反馈。
- `app/lib/screens/control_center/control_center_screen.dart` 仍在 `build()` 中根据 runtime token 覆写 `_volume`、`_brightness` 与 `_colorController.text`；只要 runtime 刷新更频繁，就会抢写用户草稿。

### 1.4 设备 / 电脑控制契约偏差

- `server/config.py` 允许 `computer_control.allowed_scripts` 顶层使用字符串数组通过校验，但 `server/services/computer_control/policies.py` 的 list 解析不会生成可执行 `command`，最终运行时仍会被静默忽略。
- `firmware/arduino/demo/face_display.cpp` 的 `faceSetStatusBar()`、`faceSetBattery()`、`faceSetWeather()` 仍为空函数；而 `demo.ino` 与 `device_channel.py` 还在继续推送这些字段，协议能力与屏幕真实能力未收口。
- `server/channels/device_channel.py` 的 `_send_display_update()` 仍默认截断到 `DISPLAY_MAX_CHARS`；固件虽然已支持字幕滚动，但 `display_update` 路径拿到的仍可能是被上游截断后的文本。
- `firmware/arduino/demo/demo.ino` 发送 `device_status` 时仍写死 `battery = -1`、`charging = false`；当前前端首页虽然已对 `battery < 0` 做 placeholder 文案，但协议层仍没有明确“真实遥测未接线”的能力位。

## 2. 关键设计决策

### 2.1 对外接口保持稳定，内部委派拆分

- 后端继续保留 `AppRuntimeService` 作为对外聚合入口，但把：
  - WebSocket client fanout / replay；
  - runtime snapshot projection；
  - planning summary refresh；
  - settings / sessions / planning / device 子路由逻辑
  下沉到新 helper / service。
- 前端继续保留 `appControllerProvider` 作为主要入口，不在这一轮强推全量 provider API 改名；拆的是内部实现和文件边界，不是页面调用方式。

### 2.2 Storage cutover 采用“两阶段”

- **阶段 1：默认改成 `dual`，先 shadow、先观测。**
  - 原因：仓库已经有 `SQLitePlanningStore` 和 `SQLiteSessionBackend`，但直接把默认改成 `sqlite` 风险偏高。
  - `dual` 能先保留 JSON / JSONL 回退面，同时暴露 mismatch / import 状态。
- **阶段 2：在 dual 稳定后再把默认读路径切到 `sqlite`。**
  - 建议门槛：影子 diff 为 0，相关 API / 事件流验收通过，再把 repo 默认值升到 `sqlite`。

### 2.3 `allowed_scripts` 收紧契约，不再继续兼容“字符串数组”

- 推荐决策：把 `computer_control.allowed_scripts` 顶层契约收紧成对象映射。
- 原因：
  - `run_script` 本质上需要确定 command/cwd；
  - 顶层字符串数组没有安全、稳定的可执行语义；
  - 继续“看起来合法、实际无效”只会制造更多误判。
- 兼容策略：校验期直接 fail fast，并输出迁移示例，不再做静默容错。

### 2.4 设备显示协议按“回复字幕 vs 状态提示”收口

- 推荐决策：
  - `text_reply` 继续作为完整回复字幕的唯一正式通道；
  - `display_update` 明确降为短状态 hint；
  - 长文本滚动能力只绑定在真正承载字幕的通道上，不再指望 hint 通道承担完整文本展示。
- 这比在现有 `display_update` 上继续打补丁更稳，也更符合固件当前真实实现。

### 2.5 Reminder 调度改成“next-fire + reconcile”

- 推荐决策：
  - 主调度由“固定 15s 轮询全部 reminder”改成“计算下一次触发时间并 sleep 到该时刻”；
  - 保留低频 reconcile（例如每 5 分钟）兜底，处理时区 / 导入 / 外部补写等异常情况。
- 这样既能消掉固定空转，也能保留简单的本地单进程稳定性。

## 8. 风险与开放问题

- `battery / charging` 的真实硬件采样是否能在本轮直接接通，取决于板级接线是否已定；若未定，建议先落 capability-gated unavailable，而不是硬做伪实时。
- `status_bar_update` 是否要恢复成真实可见 UI，不适合在没有正式 face 设计和固件实现前仓促打开；本方案默认先收口，不默认恢复。
- storage 从 `dual` 升到 `sqlite` 的具体切换时点，建议以 shadow diff 和手工联调为准，不建议在没有观察期的情况下直接升默认。

## 9. 结论

- 这批 P1 不应被当成 14 个互不相关的小修，而应被当成 5 类同源问题来修：
  - 契约和真实能力没收口；
  - transport 没有边界；
  - 默认热路径还在 JSON / JSONL；
  - reminder 调度模型还停留在 demo 阶段；
  - 前后端各有一个高冲突热点文件。
- 推荐执行顺序是：
  - 先修用户可见偏差和契约歧义；
  - 再修 queue / WS / outbound 的系统性瓶颈；
  - 再切 storage 与 scheduler；
  - 最后再拆热点文件。
- 这样落地最稳，也最符合“优雅收口而不是推倒重来”的目标。
