# 2026-04-11 AI 电脑端能力调研

## 目标

- 调研当前项目里，AI 当前到底能直接操纵电脑什么。

- 给后续要补的 CLI / 脚本能力做优先级建议。

## 结论速览

当前项目里的 AI，已经具备一套“通用文件 + Shell + 网页读取”的基础电脑操纵能力，但还没有形成“稳定、可控、可组合”的专用电脑操作工具层。

更准确地说：

1. 当前真正能直接碰电脑环境的核心能力，是 `read_file / write_file / edit_file / list_dir / exec / web_fetch / web_search(依赖 key) / spawn`。
2. 其中最关键的是文件系统工具和 `exec`。很多“AI 帮用户操作电脑”的效果，本质上都是靠通用文件读写和 Shell 命令间接完成。
3. `planning` 已经很有用，但它主要是在操作项目自己的任务 / 日程 / 提醒资源，不属于桌面自动化或操作系统控制。
4. 当前还没有窗口控制、应用启动切换、剪贴板、浏览器自动化、屏幕理解、鼠标键盘模拟、进程诊断这类专用能力。
5. 后续最值得补的，不是继续放大原始 `exec` 权限，而是补一层窄接口、结构化、可审计、默认 JSON 输出的 CLI / 脚本。

## 当前真正的电脑操纵能力

### 1. 文件系统操作

当前 `AgentLoop` 默认注册了：

- `read_file`
- `write_file`
- `edit_file`
- `list_dir`

这意味着 AI 现在已经可以直接做这些事：

- 读取本地文件内容
- 新建或覆盖文件
- 按文本片段替换文件内容
- 浏览目录结构

这类能力适合：

- 改文档
- 改代码
- 生成配置文件
- 巡检工作区内容

当前边界要注意：

- 相对路径默认解析到 `server/workspace`
- bootstrap 默认没有开启 `restrict_to_workspace`
- 也就是说，从当前代码口径看，绝对路径访问并没有被收紧到项目仓库

所以这套能力已经是“真实电脑操作”，但它还是通用入口，不是面向具体任务包装好的工具。

### 2. Shell 命令执行

当前默认注册了 `exec`。

这意味着 AI 现在可以通过 Shell 间接完成大量电脑任务，例如：

- 调用项目脚本
- 查询 git / rg / find / ls / cat 之类命令
- 执行构建、安装、导出、转换类脚本
- 串联已有命令行工具完成更复杂操作

当前实现特点：

- 有超时控制
- 有少量危险命令正则拦截
- 默认工作目录是 `server/workspace`
- 但默认不是严格 allowlist
- 返回值主要还是原始文本，不是稳定结构化结果

这意味着当前 AI 的“电脑操纵能力”很大一部分来自 `exec`，但这也是当前最不稳定、最难控、最难复用的一层。

### 3. 网页读取和条件式搜索

当前默认注册了：

- `web_fetch`
- `web_search`

当前能做的事：

- 抓取网页正文
- 提取 HTML / JSON 内容
- 在配置了 `BRAVE_API_KEY` 时做网页搜索

这类能力扩展了 AI 的信息获取范围，但它更接近“联网读取”而不是“操纵本机桌面”。

要点：

- `web_fetch` 已可直接使用
- `web_search` 取决于运行环境是否配置搜索 key

### 4. 子代理分派

当前默认注册了 `spawn`。

这不是桌面控制工具，但它对“让 AI 在电脑上并行处理任务”很重要，因为它允许：

- 把复杂任务拆成后台子任务
- 让不同子代理分别处理文件、网页、命令执行等工作

所以它更像“执行编排能力”，不是直接的 OS 控制能力。

## 不应混入“电脑操纵”口径的现有能力

### 1. `planning` 是应用内资源操作，不是桌面自动化

`planning` 现在已经可以：

- 创建任务
- 创建事件
- 创建提醒
- 完成任务
- 延后提醒
- 列出当天事项

这套能力会写入项目自己的 runtime 资源，并同步到 `Tasks / Agenda / Control / Home`。它很适合做产品闭环，但它控制的是项目内的结构化数据，不是操作系统对象。

### 2. 多会话聊天和任务队列是执行入口，不是电脑能力本身

聊天、排队、状态回推、structured summary 这些能力说明项目已经有一套较完整的 agent 运行时，但它们属于交互层和编排层，不是“AI 在电脑上能点什么、开什么、改什么”的能力本身。

### 3. 当前语音链路也不应被表述成“AI 已经能广义操纵电脑”

现有语音路径仍然是 `device-first / desktop mic bridge`，更准确的表述是“设备触发，电脑代采桌面麦克风，AI 处理后再回传结果”。它是输入输出链路，不是桌面控制工具。

## 当前缺失的电脑操纵能力

这部分是后面最值得补的地方。

### 1. 应用与窗口控制

当前没有专门工具来做：

- 打开某个 App
- 聚焦或切换窗口
- 打开某个文件或目录
- 调系统菜单
- 最小化、关闭、切换前台窗口

所以现在 AI 还不能稳定地说“帮我打开 X 软件并切到前台”。

### 2. 剪贴板、通知、系统交互

当前没有专门工具来做：

- 读取剪贴板
- 写入剪贴板
- 发送系统通知
- 调起系统分享或打开 URL

这些能力对桌面助理很关键，但目前不在 Agent 工具面里。

### 3. 浏览器自动化

当前没有专门工具来做：

- 打开浏览器页面
- 点击按钮
- 填表单
- 登录站点
- 抓页面截图
- 提取动态页面内容

所以现在的网页能力停留在“读网页”，还不是“操作网页”。

### 4. 屏幕理解和 GUI 自动化

当前没有专门工具来做：

- 桌面截图
- OCR
- 识别当前活动窗口内容
- 模拟鼠标
- 模拟键盘快捷键

所以它还不具备真正意义上的 GUI agent 能力。

### 5. 结构化系统诊断

当前没有专门工具来稳定输出：

- 进程列表
- 端口占用
- 服务健康状态
- CPU / 内存 / 磁盘使用率
- 最近日志摘要

这些事现在理论上可以通过 `exec` 临时拼出来，但没有稳定接口，也没有统一输出格式。

## 当前代码里有基础设施，但不应算作已落地电脑能力

### 1. Cron

`CronTool` 已存在，但 bootstrap 当前没有把 `cron_service` 接进默认运行时。所以不能对外说“AI 已经能定时自动执行电脑任务”。

### 2. MCP

MCP 包装层和懒连接逻辑已存在，但默认 bootstrap 也没有把 `mcp_servers` 注入当前运行时。所以不能对外说“当前 build 已经开放外部工具生态接入”。

### 3. 唤醒词 / Auto Listen

设置项里有 `wake_word` 和 `auto_listen`，但当前 runtime 状态明确仍是 `implemented = false`、`active = false`，原因字段也是 `configured_only_not_runtime_enabled`。这部分不能被算进“AI 已经能持续常驻操纵电脑”。

## 对当前能力的准确口径

如果只从“AI 操纵电脑”这个角度来概括，当前更准确的说法是：

> 现在 AI 已经有“通用文件 + Shell + 网页读取”这套基础电脑操纵能力，但还缺“稳定、可控、可组合”的专用 CLI 工具层。

换句话说：

> 现在能做很多事，但主要是靠通用入口硬拼出来；真正缺的不是想象力，而是专门为 AI 包装过的工具面。

## 后续优先增强的 CLI / 脚本方向

下面这部分是最适合后续落地的重点。

### 1. 第一优先级: 诊断类工具

建议先补一批只读、安全、结构化输出的命令，例如：

- `ai ps`
- `ai ports`
- `ai health`
- `ai logs tail`
- `ai disk usage`
- `ai env doctor`

目标不是替代系统命令，而是把常见诊断动作包装成稳定 JSON 输出，例如：

- 当前服务是否在线
- 端口被谁占用
- 某个进程是否存在
- 最近 200 行日志里有哪些 error / warning

这是最容易提升 AI 电脑操作稳定性的第一批工具。

### 2. 第二优先级: 项目操作类工具

建议把最常见的仓库动作，从原始 `exec` 提升为专用命令，例如：

- `ai rg`
- `ai find`
- `ai git status`
- `ai git diff`
- `ai run task`
- `ai file stat`
- `ai file diff`

目标是让 AI 少写自由 Shell，多走窄接口。

这样做的价值是：

- 参数更稳定
- 输出更稳定
- 更容易做权限限制
- 更容易给前端展示
- 更容易做审计日志

### 3. 第三优先级: 结构化数据工具

建议补：

- `ai json get / set / query`
- `ai yaml get / set`
- `ai csv preview / query`
- `ai sqlite schema / query / update`

这类工具非常适合 AI，因为它们天然需要结构化输入输出，也比让模型手写 `jq`、`sqlite3`、`awk` 稳定得多。

### 4. 第四优先级: 轻量桌面控制工具

建议从低风险动作开始，而不是一上来就放鼠标键盘模拟：

- `ai open app`
- `ai open path`
- `ai open url`
- `ai clipboard get`
- `ai clipboard set`
- `ai notify`
- `ai active-window`
- `ai screenshot`

这批工具一旦做好，AI 才会开始像真正的桌面助理。

### 5. 第五优先级: 浏览器自动化包装层

如果后续要做更强的“帮我在电脑上办事”，浏览器自动化几乎是必补项。建议不要直接暴露一堆底层动作，而是先包装成高层脚本：

- `ai browser open`
- `ai browser extract`
- `ai browser click`
- `ai browser fill`
- `ai browser screenshot`
- `ai browser download`

后面再逐步增加：

- 登录态管理
- 元素等待与重试
- 上传下载目录控制
- 页面结构化抽取

### 6. 第六优先级: 安全与审计层

后续每新增一类 CLI / 脚本，建议同时补这几条约束：

- 默认 JSON 输出
- 支持 `--dry-run`
- 支持明确的路径 / 项目 / URL 白名单
- 支持超时
- 支持操作日志
- 高风险动作需要显式确认

原则上，应优先新增“受限的小工具”，而不是继续放大“自由大命令”。

## 设计建议

后续如果要为 AI 增强电脑操纵能力，建议坚持这几条：

- 优先做窄接口工具，不要优先扩展原始 `exec`
- 优先做机器可读输出，不要只返回一大段文本
- 优先做高频动作包装，不要追求一次性万能
- 优先做低风险自动化，再逐步上升到 GUI 控制
- 优先做可审计和可回放，不要做“执行了但说不清怎么执行的黑盒动作”
