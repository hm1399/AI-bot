# 2026-04-11 AI 电脑端能力调研

## 结论速览

当前项目里的 AI，不只是“聊天回复”。按现有代码，AI 已经具备三类电脑端能力：

1. 作为桌面工作台里的对话助手，处理多会话聊天，并把进度、排队状态、结果实时同步到 App。
2. 作为带工具的执行代理，在服务端所在电脑上读写文件、执行 Shell、抓取网页，并把结果回流到聊天上下文。
3. 作为轻量规划助手，创建任务 / 日程 / 提醒、识别冲突、完成任务、延后提醒，并把结果同步到 `Tasks / Agenda / Control / Home`。

## 已落地能力

### 1. 多会话 AI 聊天

- 用户可以在 `Chat` 页创建、切换、重命名、置顶、归档会话。
- 用户发送消息后，后端会先入队，再通过实时事件把 `queued / thinking / tool_hint / completed` 状态推回前端。
- 首页能看到当前运行任务和排队任务，说明聊天不是同步阻塞式回复，而是带运行时队列的 agent 执行流。

### 2. AI 在电脑上的直接工具能力

当前 `AgentLoop` 默认注册了以下工具：

- 文件系统：`read_file`、`write_file`、`edit_file`、`list_dir`
- Shell：`exec`
- 网页：`web_fetch`
- Web 搜索：`web_search`
- 规划：`planning`
- 子代理：`spawn`

这些能力意味着，当前 AI 理论上已经能在服务端所在电脑上：

- 读取和修改本地文件
- 列目录、整理工作区内容
- 执行终端命令
- 抓取网页正文
- 在满足条件时做网页搜索
- 把复杂任务拆给后台子代理

### 3. 轻量规划助手

当前 AI 的一等规划工具不是“泛泛建议”，而是已经有结构化写入能力。现阶段它能直接：

- 创建任务
- 创建事件
- 创建提醒
- 完成任务
- 延后提醒
- 列出某一天的任务 / 事件 / 提醒

这套能力会自动写入运行时资源库，并同步更新：

- `Tasks` 页的任务工作台
- `Agenda` 页的月历和日程视图
- `Control Center` 的提醒与通知面板
- `Home` 页的 todo / calendar summary

### 4. 冲突识别与结构化结果回显

- AI 在创建事件时会检查时间冲突。
- 规划工具返回的 `bundle_id / resource_ids / normalized_times / conflicts / confirmation_needed` 会被保存在消息 metadata。
- 前端聊天气泡会把这些规划结果渲染成结构化 summary 卡片，而不只是纯文本回复。

这意味着当前项目已经具备“AI 一边规划，一边把结构化结果回显给用户确认”的基础闭环。

### 5. 设备协同语音链路

当前语音链路是“设备发起，电脑采音，AI 处理，设备反馈”：

- 用户在设备上长按开始说话
- 电脑侧 `desktop_voice_service` 负责桌面麦克风采音
- 后端 ASR 转文字后，把转写结果送入 AgentLoop
- AI 回复再经桌面语音桥或设备通道回传
- 设备侧会显示“识别中 / 回复中”，并优先用设备喇叭播放回复

另外还有一个现成的硬件触发能力：

- 设备摇一摇会自动向 AI 发送“讲一个有趣的笑话或者冷知识”

## 条件成立时可用的能力

### 1. Web 搜索

- `web_search` 工具已经注册。
- 但它依赖 `BRAVE_API_KEY` 或等价配置；如果没有 key，调用会直接返回配置错误。
- 所以它属于“代码已接入，但取决于运行环境是否配好搜索 key”。

### 2. 服务端电脑上的广义文件 / Shell 操作

- 当前启动路径没有把工具限制在 workspace 内。
- 相对路径默认落在 `server/workspace`，但绝对路径和 Shell 命令并没有在 bootstrap 层收紧到项目目录。
- 因此从代码口径看，AI 的电脑操作范围比“只改项目仓库”更大。

这部分能力已经存在，但当前没有前端确认弹窗或权限审批 UI，主要靠工具自身的基础防护和模型行为约束。

## 当前不应算作“已落地正式能力”的部分

### 1. 唤醒词 / Auto Listen

- 设置页里可以保存 `wake_word` 和 `auto_listen`。
- 但后端 runtime 明确把这两项标记为 `configured only`，`implemented = false`，`active = false`。

结论：

- 现在还不能对外说“已经支持常驻唤醒词”或“已经支持免按键持续监听”。

### 2. App 直接录音

- 现在的 Flutter App 已经不再自己录音。
- Chat 页和 provider 文案都明确写成“device-first / desktop mic bridge”。

结论：

- 当前语音入口不是“点 App 里的麦克风直接说”，而是“设备按住说，电脑负责代采桌面麦克风”。

### 3. 设备控制不是 AI 的一等工具

- `Control Center` 里有 `wake / sleep / mute / toggle_led / set_volume / set_led_color / set_led_brightness` 等手动控制。
- 但 AgentLoop 当前没有专门注册设备控制工具。

结论：

- 这些是“用户手动控制能力”，不是当前 build 里已经整理好的“AI 可直接调用的正式工具面”。

### 4. Cron / MCP

- 代码库里有 `CronTool` 和 MCP 包装层。
- 但当前 `server/bootstrap.py` 没把 `cron_service`、`mcp_servers` 接进运行时。

结论：

- 这两块属于“有基础设施，但当前构建默认不可用”，不能算作当前项目里已经开放给用户的 AI 能力。

## 给产品口径的建议说法

如果要用一句话概括当前项目里 AI 在电脑端能做什么，比较准确的说法是：

> 现在的 AI 已经能在桌面工作台里持续对话，并在服务端电脑上执行文件 / Shell / 网页读取等工具操作，同时还能帮用户创建和联动任务、日程、提醒，并通过设备完成按住说话的语音闭环。

如果要补一句限制条件，建议写成：

> 但当前正式稳定的语音入口仍是“设备按住说话”，唤醒词、自动常听、Cron、MCP 和设备控制型 AI 工具还不能算已完整开放。
