# Agenda脏手动行程导致误冲突修复计划

> 仅覆盖本轮新增问题：用户每次通过语音新增行程后，Agenda / Chat 会持续出现一个不相干的标题 `12` 行程，并反复提示“有冲突”，但用户实际并未创建该行程，也不存在预期内冲突。要求保持现有 Chat / Agenda UI 样式，不擅自改视觉，只修手动创建校验、后端入库防线与本地已有脏数据对当前体验的影响。

## 目标

- [x] 阻止前端手动 Agenda 创建继续写入不可解析或明显异常的 event 时间数据。
- [x] 阻止后端 planning bundle / event 入口接收非法或倒置的 event 时间窗口，避免脏数据再次落库。
- [x] 让当前环境里已存在的脏 `agenda_manual` 事件不再继续制造“标题 12 / 误冲突”的用户可见问题。

## 调研结论

- [x] `server/workspace/runtime/events.json` 当前已有两条 `created_via: "agenda_manual"` 的脏 event：
  - `event_0e175f95`：`title = "333"`，`start_at = "1"`，`end_at = "2"`，属于明显非法时间值。
  - `event_28c53ef5`：`title = "12"`，`start_at = "2026-04-16T20:52:00.000"`，`end_at = "2026-04-30T20:52:00.000"`，跨越 14 天，正在参与后端冲突判定。
- [x] 会话日志已证明近期语音新增考试行程时，后端冲突提示引用的正是旧脏事件 `event_28c53ef5`，不是本次语音链路即时新建出的 `12` 行程。
- [x] 前端手动编辑器 [app/lib/widgets/planning/planning_editor_dialog.dart](/Users/mandy/Documents/GitHub/AI-bot/app/lib/widgets/planning/planning_editor_dialog.dart) 当前只校验标题非空、event 的 `start_at / end_at` 非空，不校验时间是否可解析，也不校验 `end_at > start_at`。
- [x] 后端资源层 [server/services/app_api/resource_service.py](/Users/mandy/Documents/GitHub/AI-bot/server/services/app_api/resource_service.py) 对 event 当前也只要求 `title / start_at / end_at` 是字符串，不校验 ISO 时间合法性与时间窗口顺序。
- [x] 后端冲突构建 [server/services/planning/planning_projection_service.py](/Users/mandy/Documents/GitHub/AI-bot/server/services/planning/planning_projection_service.py) 会把所有可解析的 event 纳入 overlap 判定，因此旧脏长跨度事件会持续污染后续语音行程的冲突结果。

## 修复方案

- [x] 前端手动创建防线：
  - 为 Planning editor 的 event 保存补充本地校验，要求 `start_at / end_at` 可解析，且 `end_at` 严格晚于 `start_at`。
  - 保持现有 UI 结构，只补错误提示文案，不改版面。
- [x] 后端入库防线：
  - 为 `AppResourceService.create_event / update_event` 的标准化逻辑补充事件时间校验。
  - 让 planning bundle 走到 `create_event` 时自然继承同一套校验，避免 app 手动创建和其他入口口径不一致。
- [x] 当前脏数据止血：
  - 清理当前工作区中已确认异常的两条 `agenda_manual` 旧 event：`event_0e175f95 (333)` 与 `event_28c53ef5 (12)`，同步移除 JSON 与 SQLite shadow 中的记录。
  - 只处理当前已确认异常的目标记录，不扩散到其他正常行程。
- [x] 回归测试：
  - 先补后端资源层测试，覆盖非法 event 时间被拒绝、倒置时间窗口被拒绝。
  - 先补前端 Planning editor 校验测试，覆盖非法时间与倒置时间窗口的保存拦截。
  - 按当前用户约束，本轮先写测试，不执行，待你确认后再跑。
