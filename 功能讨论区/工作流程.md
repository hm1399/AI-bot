# 硬件上电后的联调流程：WhatsApp 替代 App，设备触发电脑麦克风代采

> 更新时间：2026-04-06
>
> 这份文档按你现在要的联调口径写：
> `硬件上电 -> 连局域网 -> 扫 WhatsApp 二维码 -> 按设备触摸键 -> 电脑开始录音 -> 后端处理 -> 回到硬件喇叭播报`
>
> 重要前提：本文默认你手上的硬件已经刷好支持这条链路的固件版本，也就是设备触摸事件会触发桌面麦克风桥接，回复结果会回到硬件喇叭播放。
> 如果你当前板子里还是旧版 `demo.ino`，那它可能仍是“设备本地麦克风直采”或“仅文字反馈”，不是本文这条流程。

---

## 一、你现在要跑的就是这一条链路

整条链路拆开看，就是下面这 7 步：

1. 硬件上电，自动连接你已经配置好的 WiFi
2. 电脑端先起 `WhatsApp bridge`
3. 终端弹出 WhatsApp 二维码，你扫码登录
4. 电脑端再起后端服务和桌面麦克风桥接
5. 硬件连上服务端后，按住设备触摸键
6. 电脑麦克风自动开始录音，你对着电脑说话
7. 松开触摸键后，后端做 ASR、Agent、TTS，再把回复发回硬件喇叭播报，同时可同步到 WhatsApp

你现在不需要跑 Flutter App，WhatsApp 就是这轮辅助验证通道。

---

## 二、启动前只确认 4 件事

### 1. 电脑和硬件在同一个局域网

- 你的电脑接你已经给硬件配置好的那个 WiFi
- 硬件上电后也会自动连这个 WiFi

### 2. 服务端依赖已经装好

默认你已经完成过一次本地环境安装。如果没装过，再补：

```bash
cd /Users/mandy/Documents/GitHub/AI-bot/server
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

如果 `sounddevice` 在 macOS 装不上，先装：

```bash
brew install portaudio
```

### 3. `server/.env` 里至少有这些配置

```env
OPENROUTER_API_KEY=你的key
APP_AUTH_TOKEN=app-local-12345678
DEVICE_AUTH_TOKEN=
OPENWEATHERMAP_API_KEY=
```

说明：

- `APP_AUTH_TOKEN` 给后端和桌面麦克风桥接用
- `DEVICE_AUTH_TOKEN` 这里先留空，避免设备连不上

### 4. 如果这轮只想测硬件和 WhatsApp，就别跑 App

- 不需要开 Flutter
- 不需要连前端
- 主验证对象就是：
  - 硬件能不能触发录音
  - 电脑麦克风会不会自动开始采音
  - 后端会不会回结果
  - 硬件喇叭会不会播报
  - WhatsApp 会不会同步收到内容

## 三、正确启动顺序

这一步最重要。顺序乱了，最容易把问题混在一起。

### 第 1 步：先启动 WhatsApp bridge

开终端 1：

```bash
cd /Users/mandy/Documents/GitHub/AI-bot/server/bridge
npm install
npm run build
npm start
```

正常情况下，终端会打印一个 WhatsApp 二维码。

### 第 2 步：用手机扫 WhatsApp 二维码

打开手机 WhatsApp，按这个路径扫：

- `WhatsApp`
- `设置`
- `已关联的设备`
- `关联设备`
- 扫终端里的二维码

扫完以后，这个终端不要关。

你这轮用 WhatsApp 替代 App 测试，所以这个二维码登录是主流程的一部分，不是可选项。

### 第 3 步：启动后端服务

开终端 2：

```bash
cd /Users/mandy/Documents/GitHub/AI-bot/server
source .venv/bin/activate
python3 main.py
```

可选健康检查：

```bash
curl http://127.0.0.1:8765/api/health
```

如果后端没起来，后面的硬件按键、电脑录音、喇叭播报都不会发生。

### 第 4 步：启动桌面麦克风桥接

开终端 3：

```bash
cd /Users/mandy/Documents/GitHub/AI-bot/server
source .venv/bin/activate
python3 tools/desktop_voice_client.py --token "$APP_AUTH_TOKEN"
```

这个工具正常启动后，会一直挂着等待服务端控制。  
你这轮的目标不是在终端里手打 `start`，而是让硬件触摸动作来自动触发它开始/结束录音。

第一次在 macOS 上运行时，如果系统弹出麦克风权限窗口，要点允许。

### 第 5 步：最后再给硬件上电

前面 3 个终端都起来以后，再给硬件上电。

你现在假设硬件已经配好，所以这里只看运行结果：

- 硬件自动连上 WiFi
- 硬件自动连上服务端
- 屏幕进入待机或在线状态

到这里，准备工作就完成了。

## 四、你真正要演示的操作流程

下面就是你要的那条主链路。

### 1. 按住硬件触摸键

你按住设备触摸键以后，设备会告诉后端：

- 现在开始语音交互
- 这次录音不是设备自己采
- 要让电脑端的桌面麦克风桥接开始录音

### 2. 电脑端自动开始录音

只要链路正常，终端 3 里的 `desktop_voice_client.py` 会收到服务端控制消息，然后自动开始采音。

也就是说：

- 你不用在电脑上手动打 `start`
- 设备按键本身就是开始录音的触发器

### 3. 你对着电脑麦克风说话

这一步不是对着硬件麦克风说，而是对着电脑麦克风说。

简单说：

- 手在硬件上按住
- 嘴对着电脑麦克风说

### 4. 松开硬件触摸键

松开以后，设备会告诉后端这次录音结束。

然后后端开始做：

1. ASR 语音转文字
2. Agent 生成回复
3. TTS 合成语音
4. 把结果回传给硬件

### 5. 硬件播报回复

如果这版硬件链路已经对齐好，你会看到或听到这几个结果：

- 硬件屏幕状态变化
- 硬件喇叭播报回复
- WhatsApp 同步收到本轮回复内容

这就是你这轮要验证的完整闭环。

## 七、最常见的 5 个问题

### 1. 扫不了 WhatsApp 二维码

先查：

- `node --version` 是否至少 `20`
- 你启动的是不是 `server/bridge`
- 终端二维码有没有完整显示出来

### 2. 硬件按了触摸键，但电脑没开始录音

先查：

- 终端 3 的 `desktop_voice_client.py` 有没有启动
- 这个桥接有没有成功连接到 `/ws/desktop-voice`
- 当前硬件固件是不是真的支持“触摸事件触发桌面麦克风代采”

### 3. 电脑录了音，但后端没回结果

先查：

- `OPENROUTER_API_KEY` 是否有效
- 后端日志里有没有 ASR 或模型调用错误
- 录音是不是太短，或者麦克风权限没放开

### 4. WhatsApp 有消息，但硬件喇叭不播报

这通常说明：

- 文本链路通了
- 但硬件音频回放链路没通

这种情况优先查硬件喇叭、功放和设备端音频播放部分，不要先怀疑 WhatsApp。

### 5. 硬件没连上服务端

先查：

- 硬件和电脑是不是同一个 WiFi
- 电脑防火墙有没有拦 Python
- 后端是不是已经正常启动
