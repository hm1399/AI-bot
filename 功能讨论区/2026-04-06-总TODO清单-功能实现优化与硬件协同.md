## 当前进度快照

### 前端当前进度

已完成：

- Flutter 工程主骨架已成立，已具备 `Connect / Home / Chat / Tasks / Control Center / Settings / Demo Mode` 页面。
- 已接入 `bootstrap / runtime / sessions / settings / tasks / events / notifications / reminders / device / app events websocket` 主协议。
- 已有本地连接保存、事件流重连、`last_event_id` 恢复、Demo Mode、本地 provider 状态管理。

当前缺口：

- `app/lib/services/chat/voice_capture_service.dart` 仍是 stub，电脑麦克风和手机麦克风都没有真实采集实现。
- `voiceAvailableProvider` 依据后端 capability 判定可用，但真实语音采集未落地，语音按钮与真实能力不一致。
- 多会话虽然 provider 已有 `selectSession()`，但 `ChatScreen` 没有完整的会话切换/新建入口。
- `Tasks / Events / Notifications / Reminders` 目前主要还是读展示，页面层 CRUD 不完整。
- `DeviceService.sendCommand()` 已有，控制中心页面还没做成真正可操作的设备控制面板。
- 连接页明确保留了 `mDNS / zeroconf` 为后续事项。
- 自动化验证很弱，当前只有一个启动级 `widget_test`。
- 本机当前 Flutter/Dart 版本不满足 `app/pubspec.yaml` 中 `sdk: ^3.11.1`，前端测试与分析无法在当前环境完成。

### 后端当前进度

已完成：

- `server/services/app_runtime.py` 已提供完整 `app-v1` HTTP + WebSocket 接口主干。
- Sessions、Messages、Settings、Tasks、Events、Notifications、Reminders、Runtime、Device、Capabilities 已接入。
- App 鉴权、事件流 replay、聊天入队与回流、设备命令入口、设备状态快照已具备。
- `server/services/app_api/resource_service.py` 已有本地 JSON 持久化 CRUD。
- `server/tests` 当前可跑通，既有调研结论为 `37 passed`。

当前缺口：

- 当前没有“电脑麦克风实时采集”的正式接入链路。
- `server/tools/test_client.py` 只能发送音频文件或模拟事件，不是实时电脑麦克风采集工具。
- `reminders` 目前是 CRUD 数据，不是实际调度执行系统。
- `wake_word`、`auto_listen` 目前主要停留在设置字段，没有真正检测或状态机闭环。
- 人格系统、场景模式、外部日历同步、手机推送通知都还没有产品级实现。
- `server/services/app_runtime.py` 继续膨胀，后续维护成本会越来越高。

### 硬件 / 固件当前进度

已完成：

- `firmware/arduino/demo/demo.ino` 仍保留 WiFi、`/ws/device`、屏幕显示、时间状态栏、天气状态栏、触摸录音的主结构。
- `server/channels/device_channel.py` 已具备设备状态机、触摸事件、摇一摇事件、音频接收、ASR、TTS 下发、设备命令下发。
- 硬件仍可作为设备在线状态、显示面板、物理交互入口的协同端。

当前缺口：

- 设备端 demo 对服务端返回的二进制音频仍是忽略处理，TTS 播放链路没有闭环。
- demo 固件没有完整 shake 上报和互动玩法闭环。
- 当前最关键的新现实约束是：`硬件麦克风已不可作为当前主路线输入源`。
- 因为输入源切换到了电脑麦克风，硬件侧角色需要重新定义，不能再默认“录音入口一定在 ESP32”。

## 总体目标重排

现阶段建议把总体目标拆成两条线并行推进：

### 主交付线

- 用 `电脑麦克风 + 当前后端 + 现有设备协同` 先完成真实语音闭环。
- 先做到“能稳定说话、能识别、能返回、能执行、能看到/听到结果”。

### 恢复与增强线

- 修复硬件麦克风。
- 补回设备播放、摇一摇、物理交互和更完整的桌面摆件能力。
