  - AppRuntime 写回 session message / runtime task 时只保留了其中一部分。
  - 因此前端目前还拿不到足够稳定的来源标签来做产品化展示。
- “电脑控制”当前仍是底层 `filesystem / shell / exec` 工具集合，不是面向用户的产品能力集合。
- 人格系统、场景模式、外部日历同步、手机推送通知都还没有产品级实现。
- reminder 触发后的跨渠道策略仍未完成，当前真正打通的只有 App 内 notification + websocket 事件流。

  - demo 固件只有 `long_press / long_release`。
  - 后端也只有旧的 `single / double / long_press / long_release` 语义，没有 `triple` 和允许 / 拒绝 / 打断状态机。

- 当前最关键的新现实约束是：`硬件麦克风已不可作为当前主路线输入源`。
- 当前应把设备定义为前台交互入口和状态反馈承载端，实际采音由电脑麦克风代采，不能再默认“设备自己录音”。
- 当前 demo 固件还没有拆成明确的运行模式，例如：
  - `设备按住说话（电脑代采）`
  - `设备本地麦克风直采`
  - `仅显示 / 控制`
- 3D 外壳整机拼装、装配后稳定性检查和最终产品包装还没有完成，当前硬件仍处于“能联调、未完成最终形态”的阶段。

## 建议执行顺序

### 第一阶段

- 先完成 `P0-5` 的真实剩余闭环，核心是把 `P1-3` 里的设备控制闭环，以及 `P2-2` 里的 `device_command` 消费 / `device_status` 主动上报一起收口，让 `Control Center / Settings / demo 固件 / runtime` 从 `accepted` 走到 `已执行 + 可回读 + 可失败提示`。
- 同阶段把 `P2-5` 里“设置是否真实生效”“设备 / ASR / TTS 统一状态提示”“planning 数据来源层级”这些真状态提示补齐，避免 UI 继续给出“看起来可用、实际未闭环”的错觉。
- 同步补 `P2-3` 中与设备命令、desktop_mic、source metadata、planning bundle 直接相关的验证，先把当前主路线的正确性门禁补起来。
- 保持 `main.py` 单进程内置桌面麦克风为默认联调路径，继续把 `desktop_mic` 当成当前主前提，不再按旧 `device_mic` 主路线安排任务。

### 第二阶段

- 主交付线：推进 `P1-6 物理交互产品化` 与 `P2-2` 中 `shake / tap / 交互映射` 相关固件恢复，把 `摇一摇 / 拍一拍 / touch 语义` 做成不干扰当前语音主线的差异化体验。
- 展示线：同步完成 `P2-6` 的 3D 外壳整机拼装、装配稳定性检查和最终产品包装，让当前版本尽快形成可展示整机。
- 平台线：并行推进 `P2-4` 的性能观测、`JSON / JSONL -> SQLite` 热路径迁移、`MessageBus` 背压、App 事件独立发送队列、outbound lane 拆分与 reminder next-fire 调度，先把后续扩张前的单机热点清掉。

### 第三阶段

- 在主闭环和平台承载更稳之后，再推进 `P1-3` 里偏“电脑控制产品化”的部分，以及 `P1-4` 的通知中心 / 跨渠道策略；这两组都依赖前面的权限边界、source metadata 稳定性和通道隔离能力。
- 之后再补 `P1-5` 场景模式与人格系统骨架、`P2-5` 剩余工程体验项，以及 `P2-1` 硬件麦克风恢复；如果未来恢复 `device_mic`，必须与 `desktop_mic` 并存，而不是回退当前主线。

## 一句话结论

当前项目不是“缺功能列表”，而是已经进入了 `主干已通、下一步要按依赖和展示价值重新排顺序` 的阶段。现阶段最重要的是先把 `设备控制真闭环 + 状态回读` 做实，再把 `物理交互 + 整机展示形态` 做出来，最后才是 `电脑控制 / 跨渠道 / 场景人格 / 硬件麦克风恢复`。
