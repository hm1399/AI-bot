# AI-bot 任务调度与性能优化建议（大白话版）

> 更新时间：2026-04-11  
> 说明：这份文档不是改代码，而是把这次调研结论用大白话整理出来，方便后面决定先做什么、后做什么。

---

## 先说结论

这个项目现在的架构，其实**很适合单机跑、局域网跑、本地设备联调跑**。

它现在最大的问题，不是模型不够快，也不是 `aiohttp` 不够快，而是：

- 很多数据还在用本地 JSON / JSONL 文件存
- 一些地方是“整份读出来、整份再写回去”
- 有些消息流还是“单条路排队走到底”
- 提醒调度还是“每隔一会儿把全部提醒扫一遍”
- WebSocket 广播还是“一个客户端慢，别的客户端也会一起被拖慢”

所以如果要优化，**最划算的方向不是一上来上大集群**，而是先把这几个热点做顺。

---

## 现在这套架构，哪里已经做得不错

先说优点，不然会误以为要全推翻。

### 1. 不同会话其实已经能并行

后端已经有“按 session 加锁”的设计。  
意思是：

- 同一个会话里的消息会按顺序处理，不会乱
- 不同会话之间可以并行，不是全局只能一条一条跑

这点其实挺重要，说明主干思路没有问题。

### 2. ASR / TTS 已经尽量避开主事件循环

语音识别和语音合成都已经用了 `asyncio.to_thread()`。

这说明：

- 重 CPU 或阻塞型的语音工作，没有直接卡死主循环
- 现在“卡”的地方，更多是在存储、广播、队列，而不是已经写好的 ASR/TTS 调用方式

### 3. 当前架构对本地开发很友好

现在主流程就是：

- 一个 Python 主进程
- 一个 AgentLoop
- 一个 AppRuntimeService
- 一个 outbound router
- 外加设备、桌面麦克风、可选 WhatsApp bridge

这个模式对现在项目阶段是合适的。  
也就是说：**不是不能扩，而是现在还没到必须拆成一堆服务的地步。**

## 现在最明显的几个性能热点

下面这些，是我觉得最值得优先处理的地方。

### 1. 会话和运行时数据，还是“整份重写”

现在聊天 session、任务、事件、提醒、通知这些数据，很多都还是本地文件持久化。

问题不是“文件存储一定不行”，而是现在的写法比较吃亏：

- 改一条数据，也可能要把整份文件重写
- 数据一多，写入耗时会越来越明显
- 并发写的时候，后面也更容易出现覆盖风险

大白话讲就是：

> 现在像是每次改 Excel 里一个格子，都把整个表另存一次。

这个在数据少的时候没感觉，数据一多就开始拖。

### 2. 会话列表读取，有点像“每次打开目录都重新翻全书”

聊天会话列表现在不只是简单读取一个索引，而是会去重新扫 session 内容，算：

- 最后一条消息时间
- 消息数
- 摘要

这在会话少的时候没问题。  
但后面如果用户聊天多了，就会变成：

> 为了显示左边的会话列表，后端还要把很多历史消息重新看一遍。

### 3. WebSocket 广播还是串行发

现在事件广播的方式，本质上还是：

- 遍历每个连接中的客户端
- 一个一个 `send_json`
- 谁慢，整轮广播都会被拖住

如果以后出现：

- 某个前端窗口网络状态差
- 某个客户端挂起
- 某个 web 端标签页很慢

那别的正常客户端也可能一起受影响。

### 4. outbound 路由是单通道排队

现在 outbound router 基本是一条总出站通道：

- 设备回复走这里
- WhatsApp 回复走这里
- App 相关消息也会经过这一层判断

这就容易出现一个典型问题：

> 某个慢通道，把整个出站链路后面的消息都堵住了。

比如：

- 设备回放比较慢
- WhatsApp 发送接口抖了一下
- 某种外部渠道超时了

那后面消息也得跟着排队。

### 5. Reminder 现在还是“轮询扫全表”

现在提醒调度器是：

- 每隔 15 秒醒一次
- 把全部 reminder 拿出来
- 一个个检查有没有到点

这个方式实现简单，早期很好用。  
但数据量一大就会浪费：

- 没到点的提醒也要反复扫
- reminder 越多，空转越多
- 时间精度也受轮询间隔影响

### 6. MessageBus 还是无界队列

现在 inbound / outbound queue 没设上限。

这代表什么？

- 正常时候没问题
- 一旦某段时间进消息特别快、消费特别慢
- 队列会一直长，没有明显的“刹车机制”

大白话就是：

> 车流可以无限往高速上放，但后面没做限流，也没做拥堵保护。

## 最值得做的优化方向

我把建议分成三档：**先做、再做、以后再说。**

---

## 第一档：最推荐，收益最大，风险也最可控

这档我建议尽快做。

### 建议 1：把 JSON / JSONL 热路径迁到 SQLite

我最推荐的是：

- 先不上 Redis
- 先不上 Kafka
- 先不上大规模微服务
- 先把热路径数据迁到 **SQLite**

适合迁的内容：

- session 元数据
- 消息记录
- task / event / reminder / notification
- app runtime 里那些需要持久化的资源

为什么先推荐 SQLite：

- 学习和接入成本低
- 单机非常合适
- 比现在的整文件读写更稳
- 后续如果打开 WAL 模式，读写并发也会明显更好
- 还不用先引入额外运维成本

### 建议 2：给队列加上限、分优先级

建议把消息队列分成至少三类：

- 高优先级：`/stop`、设备控制、用户刚发来的消息
- 中优先级：普通 assistant reply、状态同步
- 低优先级：进度提示、历史整理、后台补任务

同时队列要设 `maxsize`，不能无限长。

这样做好处很直接：

- 高优先级消息不会被低价值消息淹掉
- 后端出现积压时，不会无限吃内存
- 一些“可以丢掉”的旧进度消息，有机会被主动降级

### 建议 3：给 WebSocket 客户端一人一条发送队列

不要再用“广播时逐个 await 发送”的模式。

更好的方式是：

- 每个客户端自己有一个发送队列
- 后端广播时只负责把事件塞进各自队列
- 每个客户端单独有 writer task 去发送

这样：

- 慢客户端不会拖死全体
- 还能针对不同事件做丢弃策略

例如：

- `progress` 这类更新很频繁的事件，可以只保留最新一条
- `message.completed` 这种关键事件则必须保送

### 建议 4：outbound 路由拆成几条 lane

建议至少拆成：

- `app_realtime`
- `device_voice`
- `external_channels`

也就是不要所有出站消息都挤在一条单车道上。

这样做以后：

- 设备语音回放慢，不会拖慢 app 事件
- 外部渠道抖动，不会拖慢主交互链路

### 建议 5：提醒调度改成“只盯最近要触发的那个”

比起每 15 秒全量扫描，更推荐：

- 内存里维护一个按 `next_trigger_at` 排序的小根堆 / 优先队列
- 调度器只睡到“最近那个提醒该触发的时间点”
- 新 reminder 加进来时，再重排

这样：

- reminder 少的时候也更省
- reminder 多的时候也不会全表空转
- 到点更准

### 建议 6：做一个“记忆管理页面”

除了后端存储和调度优化，我还建议以后补一个**用户可见的记忆管理页面**。

原因很简单：

- 不是所有记忆都应该一直留着
- 有些会话过期了、没用了、甚至会误导后续回答
- 用户应该能自己决定“哪些信息保留，哪些信息删掉”

这个页面可以先做成比较实用的版本，不用一开始就很复杂。

建议先让用户能管理这些内容：

- 过时的聊天会话记录
- 不再有用的设备交互记录
- 已经过期的任务、事件、提醒
- 长期记忆里的旧信息或错误信息

大白话讲，就是给用户一个地方，能看见系统到底记住了什么，并且能自己动手清理。

### 这个页面建议怎么做

我建议默认分成几块：

- 会话记录管理
- 长期记忆管理
- 提醒 / 任务 / 事件清理
- 删除记录与恢复区

其中最重要的是“会话记录管理”。

用户最好可以：

- 按时间筛选旧会话
- 按来源筛选，比如 `App / device / WhatsApp`
- 先预览摘要，再决定删不删
- 批量归档
- 批量删除
- 一键清理很久以前的会话

### 删除时不要一上来就真删

这里我不建议一开始就做成“点一下就永久删除”。

更稳的方式是分三层：

1. 先归档
2. 再软删除
3. 最后才永久删除

这样用户后悔的时候还能救回来，也比较安全。

比如：

- 普通旧会话：先归档
- 明确不要的会话：移到回收站
- 只有用户二次确认后：才永久清除

### 长期记忆也应该能管理

这个功能不要只盯聊天记录。

因为现在系统还有 `MEMORY.md / HISTORY.md` 这类长期记忆。  
如果里面有过时内容、错误信息、用户不想保留的内容，也应该能处理。

建议这里至少支持：

- 看摘要
- 按条删除
- 标记“这条过时了”
- 手动编辑纠正

不然以后系统可能一直拿旧记忆来影响新回答。

### 这个页面为什么和 SQLite 建议是配套的

如果后面真的做这个页面，那 SQLite 这条建议就更有价值。

因为做“记忆管理页面”会天然需要：

- 搜索
- 筛选
- 分页
- 批量更新
- 软删除
- 恢复
- 真删除

这些操作用现在的 JSON / JSONL 文件也不是完全不能做，  
但会越做越别扭。

如果底层换成 SQLite，这种页面会好做很多，也更稳。

### 建议 7：把会话列表要用的信息做成索引字段

左边聊天会话列表常用的信息：

- 标题
- 最后消息时间
- 最后摘要
- 消息数
- pinned / archived

这些不应该每次都去扫整段会话历史现算。  
应该在每次写入消息时顺手更新。

这样会话列表读取就会轻很多。

## 第二档：中期可以考虑，适合你开始明显扩量时

这档不是现在立刻必须，但后面很可能要走到。

### 建议 8：把语音链路拆成独立 worker

现在 ASR/TTS 已经用线程规避阻塞了，这很好。  
但如果以后：

- 设备同时多起来
- 桌面麦克风也常开
- 语音任务越来越多

那可以把下面几块拆出去：

- ASR worker
- TTS worker
- 主 API / Agent worker

这样主进程压力会更可控。

### 建议 9：把 scheduler 单独拆出去

现在 reminder 调度和 app runtime 在一起。  
未来如果 reminder / todo / calendar 再变复杂，可以单独拆一个 scheduler 进程。

好处：

- 责任更清晰
- 重启 scheduler 不会影响主 API
- 以后做更复杂的周期任务也更方便

### 建议 10：如果真有多进程/多实例需求，再上 Redis Streams

如果以后真变成：

- 一个 API 进程
- 一个 Agent worker
- 一个语音 worker
- 一个 scheduler worker

那本地内存队列就不够了。  
这时可以考虑 Redis Streams。

它适合拿来做：

- worker group 消费
- 消息确认
- pending message 重领
- 解耦多进程之间的事件流

但我要强调一下：

> 这是“后面真有需要时”的方案，不是现在第一步方案。

---

## 第三档：先别急着上，不然复杂度容易爆

下面这些东西不是不能用，而是**现在直接上，八成会比收益更麻烦**。

### 1. 不建议现在就全面微服务化

如果现在直接拆成：

- API 服务
- Agent 服务
- Scheduler 服务
- Voice 服务
- Event 服务
- Storage 服务

那你马上会多出一堆问题：

- 服务之间怎么通信
- 部署怎么配
- 本地开发怎么跑
- 日志怎么串
- 出错后怎么查

现在项目还没大到必须为这些复杂度买单。

### 2. 不建议现在就 Kafka / 大消息平台

这个项目现在更像“本地 AI 助手 + app + 设备联动”，不是日志平台，也不是超大吞吐事件系统。  
直接上 Kafka，太重了。

### 3. 不建议只靠“多开几个 worker”硬顶

如果底层问题还是：

- 文件整份重写
- 广播串行
- 队列无界
- 提醒全量轮询

那只靠加 worker，很多问题只是被推迟，不是真解决。

---

## 我建议的落地顺序

如果让我来排优先级，我会这么排：

### 第 1 步：先加观测

先补指标，不然你后面改了也不知道有没有真变快。

建议先记录：

- inbound queue 长度
- outbound queue 长度
- 消息排队等待时间
- session 保存耗时
- session 列表读取耗时
- reminder 调度延迟
- websocket 发送耗时
- TTS 首包时间

### 第 2 步：把热路径存储迁到 SQLite

这是性价比最高的一步。

先迁：

- session
- messages
- tasks
- reminders
- notifications

### 第 3 步：把 WebSocket 广播改成每客户端独立队列

这一步用户体感会很明显，尤其是前端越来越多的时候。

### 第 4 步：拆 outbound lane

让设备、App、外部渠道互不拖累。

### 第 5 步：reminder 改成 next-fire 调度

把空转轮询换掉。

### 第 6 步：根据真实压力，再决定要不要 Redis / 多进程

不要提前上复杂度。

---

## 如果只想先做最少版本，我建议做什么

如果现在时间不多，只能先做一个“最小优化包”，那我建议只做下面 4 件：

1. `session/runtime` 热路径迁 SQLite  
2. WebSocket 广播改成每客户端独立发送队列  
3. outbound router 拆 lane  
4. reminder 从全量轮询改为 next-fire 触发

做完这四件，整个项目的调度和性能底盘就会稳很多。

---

## 这套建议适不适合你现在的项目阶段

我的判断是：

- **适合现在就做的：** SQLite、队列分级、有界队列、WS 独立发送队列、提醒调度优化
- **适合后面再做的：** worker 拆分、Redis Streams、进程间事件总线
- **暂时不建议的：** 重型微服务化、Kafka、过早分布式

一句话总结就是：

> 先把“单机架构做扎实”，再考虑“多进程拆分”，最后才是“更重的分布式”。

---

## 最后一句大白话总结

这个项目现在不是“架构方向错了”，而是“底盘还停留在早期版本”。

最值得做的不是推翻重来，而是：

- 把本地文件热路径升级成更稳的本地数据库
- 把单通道排队改成分流
- 把广播改成每客户端独立发送
- 把 reminder 从“扫全表”改成“盯最近一个”

这样能用最小代价，把你现在这套 AI-bot 主链路撑得稳很多。
